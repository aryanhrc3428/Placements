## Section 1: Review of Hardware Atomic Primitives

To construct dependable high-level synchronization APIs, computer architectures must provide underlying hardware-level operations that run to completion indivisibly. The following four foundational atomic primitives form the baseline for system synchronization:

### 1.1 Test-and-Set (`test&set`)
* **Hardware Logic:** Reads a value from a memory address, sets the value at that address to `1`, and returns the original value.
* **Conceptual Pseudocode:**
  ```c
  int test_and_set(int *address) {
      int result = *address; // Read original value
      *address = 1;          // Set value to 1 atomically
      return result;         // Return original value
  }
  ```

### 1.2 Swap (`swap`)
* **Hardware Logic:** Atomically exchanges the contents of a CPU register with a value in memory.
* **Conceptual Pseudocode:**
  ```c
  int swap(int *address, int register_value) {
      int temp = *address;       // Read current value from memory
      *address = register_value; // Write register value to memory
      return temp;               // Return memory value back to register
  }
  ```

### 1.3 Compare-and-Swap (`compare&swap`)
* **Hardware Logic:** Compares the contents of a memory location to an expected register value (`reg1`). If they are equal, the memory location is updated with a new value (`reg2`) and returns success. Otherwise, memory remains unchanged and returns failure.
* **Conceptual Pseudocode:**
  ```c
  int compare_and_swap(int *address, int reg1, int reg2) {
      if (reg1 == *address) {
          *address = reg2; // Put new value into memory
          return SUCCESS;  // Return success status
      } else {
          return FAILURE;  // Memory remains unchanged
      }
  }
  ```

### 1.4 Load-Linked & Store-Conditional (`ll` & `sc`)
* **Hardware Logic:** Used together to achieve lock-free synchronization across modern RISC architectures. `load-linked` (`ll`) reads a memory location and registers its address in a hardware tracking table. `store-conditional` (`sc`) only writes a new value to that address if no other thread has modified it since the `ll` instruction. If a modification is detected, the store fails.
* **Conceptual Assembly Loop:**
  ```assembly
  loop:
      ll   r1, M[address]   # Load value and establish hardware link
      movi r2, 1             # Prepare tentative new value
      sc   r2, M[address]   # Store conditionally; sets r2 to 1 on success, 0 on failure
      beqz r2, loop         # If the store conditional failed (r2 == 0), retry the loop
  ```

### 1.5 Implementing Locks with Atomics & Spin Locks
Using atomic instructions, we can implement locks. A naive approach is a spin lock:
```c
int thelock = 0;
void acquire(int *thelock) {
    while (test_and_set(thelock)); // Busy-wait until lock is free
}
void release(int *thelock) {
    *thelock = 0;
}
```
* **Busy-Waiting:** The thread consumes CPU cycles while waiting, taking cycles away from the thread holding the lock.
* **Priority Inversion:** If a busy-waiting thread has a higher priority than the thread holding the lock, the system makes no progress. This issue occurred in the original Martian rover.

**Multiprocessor Spin Locks (`test&test&set`):**
To reduce cache invalidation traffic on multiprocessors, threads can read the lock value without attempting atomic writes until it appears free:
```c
void acquire(int *thelock) {
    do {
        while (*thelock); // Wait until might be free (read-only, stays in cache)
    } while (test_and_set(thelock)); // Atomic grab of lock
}
```

### 1.6 Better Locks: Linux futex (Fast Userspace Mutex)
To avoid busy-waiting entirely, we need to put waiting threads to sleep. The Linux `futex` provides an interface to conditionally sleep in the kernel.
```c
#include <linux/futex.h>
int futex(int *uaddr, int futex_op, int val, const struct timespec *timeout);
```
* `FUTEX_WAIT`: If `val == *uaddr`, sleep until awoken (atomic check inside the kernel).
* `FUTEX_WAKE`: Wake up at most `val` waiting threads.
* **Key Insight:** `futex` allows lock implementations (using atomic swap/compare-and-swap with UNLOCKED, LOCKED, and CONTESTED states) that require *zero overhead (no syscalls)* in the uncontested case, only trapping into the kernel when threads actually need to sleep or wake others.

### 1.7 Lock-Free Data Structures using Compare&Swap
`compare&swap` is foundational for lock-free data structures. For example, an atomic add to a lock-free linked list queue:
```c
void addToQueue(Object *object) {
    int r1;
    do {
        r1 = *root;              // Get pointer to current head
        object->next = r1;       // Save link in new object
    } while (!compare_and_swap(&root, r1, object)); // Repeat until no conflict
}
```

## Section 2: The Semaphore Abstraction

A **Semaphore** is a high-level synchronization variable initialized to a non-negative integer value, offering a clean alternative to the complexity of raw load and store operations. 

### 2.1 Core Mechanics and Primitives
Invented by Edsger Dijkstra, a semaphore exposes an internal integer state through two restricted, atomic system operations:

#### 1. The $P()$ Operation (`down()`, `wait()`)
* **Behavior:** Attempts to decrement the internal semaphore value. If the value is strictly greater than zero, it is decremented, and the thread continues executing immediately. If the value is zero, the calling thread is suspended and placed on the semaphore's internal wait queue until a wakeup signal arrives.
* **Dijkstra Origin:** Named after the Dutch word *Proberen* (meaning "to test").

#### 2. The $V()$ Operation (`up()`, `signal()`)
* **Behavior:** Increments the internal semaphore value. If any threads are currently suspended on the wait queue, the kernel removes one thread and places it onto the scheduler's ready queue to resume execution.
* **Dijkstra Origin:** Named after the Dutch word *Verhogen* (meaning "to increment").

### 2.2 Critical Invariants and Structural Safety
* **Atomic Enforcement:** The execution of both $P()$ and $V()$ must be fully serialized. If two threads call $P()$ on a semaphore with a value of `1` simultaneously, the internal kernel mechanics ensure that only one thread decrements it to `0` and proceeds, while the second thread blocks.
* **Non-Negative Value Invariant:** A semaphore's value can never drop below zero. This behavior provides a reliable foundation for enforcing scheduling constraints across multiple processes.

### 2.3 Categorization and Two Main Uses
Semaphores are generally utilized for two distinct purposes:
1. **Mutual Exclusion (Binary Semaphore):** Initialized with an integer value of exactly `1`. It behaves identically to a standard mutual exclusion lock (Mutex), guaranteeing that at most one thread can enter a protected critical section at any given time.
2. **Scheduling Constraints (Counting Semaphore / Sync Semaphore):** Initialized to `0` or an arbitrary positive integer value $N$. This allows threads to wait for a specific condition or signal from another thread.
   * *Example:* Thread 1 waits for Thread 2 (`ThreadJoin` waiting for `ThreadFinish`). Initialize semaphore to `0`. `ThreadJoin` calls $P()$, and `ThreadFinish` calls $V()$ to signal completion.
   * It can also represent a pool of $N$ identical resources, allowing up to $N$ threads concurrent access.

## Section 3: Solving the Bounded Buffer (Producer-Consumer) Problem

The Bounded Buffer problem is a classic synchronization challenge where a set of `Producer` threads populate slots in a fixed-size queue, and a set of `Consumer` threads remove items from it.



### 3.1 Coordination Constraints
To prevent data corruption, a correct solution must enforce three synchronization rules:
1. **Mutual Exclusion:** Only one thread can modify the buffer's internal pointers or data arrays at any given time.
2. **Buffer Overflow Prevention:** A producer thread must block and yield the CPU if it tries to add an item when all buffer slots are full.
3. **Buffer Underflow Prevention:** A consumer thread must block and yield the CPU if it tries to remove an item when the buffer is completely empty.

### 3.2 Semaphore Synchronization Scheme
We can enforce these rules cleanly by using three separate semaphores, each dedicated to a specific constraint:
* `semaphore mutex = 1;` — A binary semaphore providing mutual exclusion over buffer modifications.
* `semaphore empty = N;` — A counting semaphore initialized to the total buffer capacity $N$, tracking the number of vacant slots available for producers.
* `semaphore full = 0;`  — A counting semaphore initialized to `0`, tracking the number of populated items available for consumers.

### 3.3 Implementation Code in C
```c
#include <stdio.h>
#include <pthread.h>
#include <semaphore.h>

#define N 8 // Fixed buffer capacity

int buffer[N];
int in = 0;  // Write pointer for producers
int out = 0; // Read pointer for consumers

sem_t mutex; // Enforces mutual exclusion
sem_t empty; // Tracks vacant slots (initialized to N)
sem_t full;  // Tracks populated items (initialized to 0)

void *producer(void *arg) {
    int item;
    while (1) {
        item = produce_new_item();

        // Check if there is an empty slot in the buffer; blocks if empty == 0
        sem_wait(&empty); 
        
        // Acquire the binary mutex to enter the critical section safely
        sem_wait(&mutex); 
        
        /* --- START CRITICAL SECTION --- */
        buffer[in] = item;
        in = (in + 1) % N; // Circular buffer increment
        /* ---- END CRITICAL SECTION ---- */
        
        // Release the binary mutex
        sem_post(&mutex); 
        
        // Increment the full counter to notify waiting consumers
        sem_post(&full); 
    }
}

void *consumer(void *arg) {
    int item;
    while (1) {
        // Check if the buffer contains any items; blocks if full == 0
        sem_wait(&full); 
        
        // Acquire the binary mutex to enter the critical section safely
        sem_wait(&mutex); 
        
        /* --- START CRITICAL SECTION --- */
        item = buffer[out];
        out = (out + 1) % N; // Circular buffer increment
        /* ---- END CRITICAL SECTION ---- */
        
        // Release the binary mutex
        sem_post(&mutex); 
        
        // Increment the empty counter to notify waiting producers
        sem_post(&empty); 

        consume_item(item);
    }
}
```

## Section 4: The Monitor Abstraction

While semaphores are powerful, managing multiple variables spread across complex application paths can easily introduce bugs. To make synchronization more manageable, software systems use an object-oriented paradigm called a **Monitor**.

### 4.1 Definition and Core Components
A monitor is a high-level programming language construct that encapsulates shared data structures, a mutual exclusion lock, and one or more **Condition Variables** to orchestrate thread interactions.



1. **The Monitor Mutex Lock:** A thread must acquire the monitor's lock before it can execute any internal method or access encapsulated variables. The monitor guarantees that at most one thread can be active inside its boundaries at any time.
2. **Condition Variables (CV):** Explicit synchronization queues inside the monitor that allow threads to safely sleep and release the monitor lock when they cannot make forward progress.

### 4.2 Condition Variable Operations
A condition variable ignores standard integer increments and instead exposes three atomic operations:

* `wait(&cond, &lock)`: Called when a thread cannot proceed. It performs three actions atomically:
  1. Puts the calling thread to sleep on the condition variable's wait queue.
  2. Releases the monitor's lock so other threads can enter.
  3. On wakeup, it re-acquires the monitor lock before returning to the application code.
* `signal(&cond)`: Wakes up a single thread waiting on that condition variable's queue. If no threads are waiting, the signal is a no-op and is discarded.
* `broadcast(&cond)`: Wakes up all threads currently sleeping on that condition variable's queue.

## Section 5: Hoare vs. Mesa Monitor Semantics

When a thread executes a `signal()` operation on a condition variable, a choice must be made: which thread runs next, the signaler or the waiter? This choice is defined by two different monitor semantics.

### 5.1 Mesa Semantics (Signal-and-Continue)
* **Execution Flow:** When a thread issues a `signal()`, the awakened thread is moved from the condition variable's queue to the scheduler's ready queue. The signaling thread retains the monitor lock and continues running until it exits the monitor or hits its own wait condition.
* **The Guarding `while` Loop Imperative:** Because the signaling thread keeps running, it could modify the shared state before the awakened thread gets a chance to execute. As a result, the awakened thread cannot assume the condition is still true when it resumes. It **must re-evaluate the condition inside a `while` loop** before proceeding:
  ```c
  while (resource_is_unavailable) {
      cond_wait(&condition, &lock); // Re-checks the condition upon re-acquiring the lock
  }
  ```
* **Production Context:** Mesa semantics are used by modern production systems, including POSIX threads (`pthreads`), Linux, Java, and C#.

### 5.2 Hoare Semantics (Signal-and-Urgent-Wait)
* **Execution Flow:** When a thread issues a `signal()`, the monitor lock is transferred immediately and atomically to the awakened thread. The signaling thread is suspended and placed on an internal "urgent queue," allowing the awakened thread to run immediately without interference.
* **The Guarding `if` Statement:** Because the lock is transferred atomically, no other threads can modify the state in between. The awakened thread is guaranteed that the condition is true when it wakes up, allowing it to use a simple `if` statement:
  ```c
  if (resource_is_unavailable) {
      cond_wait(&condition, &lock); // Can proceed safely immediately upon wakeup
  }
  ```
* **Production Context:** Hoare semantics are conceptually elegant but introduce significant scheduling overhead, making them rare in production operating system kernels.

## Section 6: Deep Dive: The Readers/Writers Problem

The Readers/Writers problem models access to a shared resource, like a database, where multiple threads need access concurrently.

### 6.1 System Optimization Constraints
To maintain data consistency while maximizing performance, the solution must enforce two rules:
1. **Reader Concurrency:** Multiple reader threads can safely read the database at the same time, as they do not modify the data.
2. **Writer Mutual Exclusion:** Only a single writer thread can access the database at a time. While a writer is active, all other readers and writers must be blocked from entering.

### 6.2 Monitor Tracking Variables
To implement this cleanly under Mesa semantics, the monitor tracks the system's execution state using four internal integers and two condition variables:
* `AR` (Active Readers): The number of threads currently reading inside the database.
* `WR` (Waiting Readers): The number of reader threads blocked outside, waiting to enter.
* `AW` (Active Writers): The number of writer threads inside the database (can only be 0 or 1).
* `WW` (Waiting Writers): The number of writer threads blocked outside, waiting to enter.
* `cond_t okToRead;` — Condition variable used to suspend reader threads.
* `cond_t okToWrite;` — Condition variable used to suspend writer threads.

### 6.3 Complete Implementation Code (Mesa Semantics)

The code blocks below implement the Readers/Writers coordination loops using standard pthreads and Mesa monitor tracking state variables:

```c
#include <pthread.h>
#include <stdio.h>

// Monitor State Registers
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t okToRead = PTHREAD_COND_INITIALIZER;
pthread_cond_t okToWrite = PTHREAD_COND_INITIALIZER;

int AR = 0; // Active Readers
int WR = 0; // Waiting Readers
int AW = 0; // Active Writers
int WW = 0; // Waiting Writers

// --- READER OPERATIONS ---

void Reader_CheckIn() {
    pthread_mutex_lock(&lock);
    
    // Under Mesa semantics, we must check conditions inside a while loop.
    // A reader must block if a writer is active (AW > 0) or if a writer is waiting (WW > 0).
    // Prioritizing waiting writers (WW > 0) prevents writer starvation.
    while (AW > 0 || WW > 0) {
        WR++; // Increment waiting readers counter
        pthread_cond_wait(&okToRead, &lock); // Sleep and release lock
        WR--; // Decrement waiting readers counter on wakeup
    }
    
    AR++; // Reader is now active!
    pthread_mutex_unlock(&lock);
}

void Reader_CheckOut() {
    pthread_mutex_lock(&lock);
    AR--; // Reader has finished and left
    
    // If this was the last active reader, check if any writers are waiting
    if (AR == 0 && WW > 0) {
        pthread_cond_signal(&okToWrite); // Wake up a single waiting writer
    }
    
    pthread_mutex_unlock(&lock);
}

// --- WRITER OPERATIONS ---

void Writer_CheckIn() {
    pthread_mutex_lock(&lock);
    
    // A writer must block if any readers are active (AR > 0) or another writer is active (AW > 0)
    while ((AW + AR) > 0) { 
        WW++; // Increment waiting writers counter
        pthread_cond_wait(&okToWrite, &lock); // Sleep on the writer condition variable
        WW--; // Decrement waiting writers counter on wakeup
    }
    
    AW++; // Writer is now active!
    pthread_mutex_unlock(&lock);
}

void Writer_CheckOut() {
    pthread_mutex_lock(&lock);
    AW--; // Writer has finished and left
    
    // Give priority to waiting writers to prevent starvation
    if (WW > 0) {
        pthread_cond_signal(&okToWrite); // Wake up a single waiting writer
    } 
    // If no writers are waiting, wake up all waiting readers concurrently
    else if (WR > 0) { 
        pthread_cond_broadcast(&okToRead); // Wake all readers
    }
    
    pthread_mutex_unlock(&lock);
}

// --- EXECUTION IMAGE WRAPPER ---

void *Reader_Task(void *arg) {
    Reader_CheckIn();
    AccessDatabase(ReadOnly); // Safe concurrent reading
    Reader_CheckOut();
    return NULL;
}

void *Writer_Task(void *arg) {
    Writer_CheckIn();
    AccessDatabase(ReadWrite); // Safe exclusive writing
    Writer_CheckOut();
    return NULL;
}
```

### Question 1: Constructing Semaphores from Monitors
Can we build a fully functional Dijkstra Counting Semaphore using only a Monitor (a single Mutex Lock and one Condition Variable)? 

If yes, write out the complete, valid C implementation for the initialization, $P()$, and $V()$ functions, ensuring compliance with Mesa semantics. If no, provide a technical proof detailing why it is mathematically impossible.

#### Solution:
Yes, it is entirely possible to implement a counting semaphore using a monitor framework. We track the semaphore's non-negative state using an internal integer variable, wrapped inside a `while` loop to comply with Mesa semantics.

```c
#include <pthread.h>

typedef struct {
    pthread_mutex_t lock;      // Monitor Lock
    pthread_cond_t cond;        // Condition Variable
    int value;                  // Internal Semaphore State
} monitor_semaphore_t;

// Initialization Routine
void sem_monitor_init(monitor_semaphore_t *s, int initial_value) {
    pthread_mutex_init(&s->lock, NULL);
    pthread_cond_init(&s->cond, NULL);
    s->value = initial_value; // Must be non-negative
}

// P() Operation (Wait / Down)
void sem_monitor_P(monitor_semaphore_t *s) {
    pthread_mutex_lock(&s->lock);
    
    // Mesa Semantics Requirement: we must use a while loop, 
    // as the condition might change before a thread wakes up and re-acquires the lock.
    while (s->value == 0) {
        pthread_cond_wait(&s->cond, &s->lock); // Sleep until a V() signal arrives
    }
    
    s->value--; // Decrement the internal counter atomically
    pthread_mutex_unlock(&s->lock);
}

// V() Operation (Signal / Up)
void sem_monitor_V(monitor_semaphore_t *s) {
    pthread_mutex_lock(&s->lock);
    
    s->value++; // Increment the internal counter
    pthread_cond_signal(&s->cond); // Wake up a single waiting thread
    
    pthread_mutex_unlock(&s->lock);
}
```

### Question 2: Starvation Hazards in Readers/Writers Design
Analyze the `Writer_CheckOut()` routine provided in Section 6.3:

```c
if (WW > 0) {
    pthread_cond_signal(&okToWrite); // Prioritize waiting writers
} else if (WR > 0) {
    pthread_cond_broadcast(&okToRead); // Wake waiting readers
}
```

1. What strategy does this coordination logic use to prioritize threads?
2. What are the starvation risks for reader threads under high write traffic? Explain how you would modify the design to guarantee fairness across both readers and writers.

#### Solution:
1. **Prioritization Strategy:** This coordination scheme directly **prioritizes writers over readers**. When a writer exits the database, it checks if any writers are waiting (`WW > 0`) before checking for waiting readers. If a writer is found, it signals that writer immediately, ignoring any readers that may have been waiting on the queue.
2. **Starvation Risks:** If the system faces high write traffic where new writer threads continue to arrive and increment `WW`, reader threads will be starved indefinitely. They will remain trapped on the `okToRead` condition variable queue, unable to enter.
3. **The Fix (Fairness Modification):** To guarantee fairness and prevent reader starvation, we can track arrival order using a FIFO timestamp queue or use an integer counter to limit how many consecutive writers can pass through before waiting readers must be served. Alternatively, we can use a state bit flag that blocks arriving writers if reader threads have been waiting on the queue for more than a set threshold.
