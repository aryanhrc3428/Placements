## Section 1: Thread Scheduling & Process Lifecycle Architecture

An operating system kernel manages execution contexts by continuously mapping virtual threads to physical processing cores. This orchestration relies on a core kernel component called the scheduler.

### 1.1 Context Recall: The Execution Environment
A thread represents an isolated execution stream tracking its own Program Counter (PC), stack pointer (SP), and machine register states. When a running thread is suspended (e.g., when a thread executes a voluntary `yield()` call inside a processing loop), its current execution state is saved to its **Thread Control Block (TCB)**. The kernel scheduler can then place that TCB onto different execution tracking queues:
* **Ready Queue:** A collection of TCBs that are fully operational and waiting to be allocated a CPU core.
* **Wait/Blocked Queue:** A collection of TCBs that are suspended and cannot run until a specific event or condition is resolved (such as I/O data returning or a lock being released).


### 1.2 The Complete Process/Thread Lifecycle State Machine
As an instruction stream executes, its operational status changes, transitioning through five distinct architectural states:

```
      admitted              scheduler dispatch
 [New] ───────> [Ready] ───────────────────────────> [Running] ───────> [Terminated]
                  ▲                                      │      exit
                  │          time slice expired          │
                  │       (or hardware interrupt)        │
                  └──────────────────────────────────────┤
                  │                                      │
                  │        I/O or event completion       ▼  I/O or event wait
                  └──────────────────────────────── [Waiting]
```
*Visual layout of lifecycle state transitions.*

1. **New:** The process or thread is currently being initialized by the kernel (allocating memory blocks, instantiating TCB/PCB records); it is not yet eligible to run.
2. **Ready:** The thread is fully operational and waiting in the ready queue for the scheduler to allocate it a CPU core.
3. **Running:** The thread has been allocated a CPU core; its instructions are executing directly on the physical hardware.
4. **Waiting:** The thread has issued a blocking request (e.g., an I/O read or a synchronization lock request) and has yielded the CPU. It is placed on a wait queue until the event completes.
5. **Terminated:** The thread has finished executing its code. It enters a dead or "zombie" state until its parent process reclaims its exit status, at which point its kernel resources are freed.

### 1.3 Per-Device Queue Multiplexing
The kernel does not maintain a single, global queue for blocked threads. Instead, to avoid lock contention and optimize scheduling, it distributes blocks into **separate queues for each physical device, interrupt vector, or condition variable**.


* **Queue Layout:** Hardware drivers (such as USB Controller Unit 0, Storage Disk Unit 0, Disk Unit 2, or Ethernet Network Adapter 0) each maintain a private queue tracking the Process Control Blocks (PCBs) and TCBs waiting on that specific device.
* **Policy Isolation:** Because these queues are isolated from each other, the kernel can enforce different scheduling policies (such as First-In-First-Out, Priority Scheduling, or Round-Robin) on different queues to match the performance needs of each device.

## Section 2: Concurrency Paradigms & Low-Level Nondeterminism

When applications share data across multiple threads, scheduling choices can introduce complex, non-deterministic bugs.

### 2.1 Server Concurrency Abstractions Comparison
Consider a banking server processing transactions across an automated teller network. Systems engineers can organize this application using three different architectural paradigms:

1. **Iterative Single-Threaded Model:** Processes requests sequentially. While simple, this model forces the entire server to block whenever a request waits for disk I/O, preventing the system from overlapping computation and hardware latencies.
2. **Event-Driven Architecture:** Uses a single thread that loops over non-blocking events via a central coordinator call like `WaitForNextEvent()`. This model overlaps computation and I/O without the overhead of threads, but it significantly increases code complexity. Developers must split their application logic into hundreds of distinct, non-blocking callback fragments (`StartOnRequest`, `ContinueRequest`, `FinishRequest`), and a single accidental blocking I/O step can stall the entire server.
3. **Multi-Threaded Shared Address Model:** Allocates a dedicated virtual thread to handle each client request. This allows code to be written cleanly and sequentially from start to finish, as the operating system automatically handles blocking and context switches behind the scenes. However, because these threads share the same process address space, any shared state can easily be corrupted if access is not properly synchronized.

```
Thread 1 (Transaction A)               Thread 2 (Transaction B)
┌──────────────────────────────────┐   ┌──────────────────────────────────┐
│ load  r1, acct->balance          │   │                                  │
│ add   r1, amount1                │   │                                  │
│               [ --- Context Switch Interrupts Execution --- ]           │
│                                  │   │ load  r1, acct->balance          │
│                                  │   │ add   r1, amount2                │
│                                  │   │ store r1, acct->balance          │
│               [ --- Context Switch Restores Thread 1 --- ]              │
│ store r1, acct->balance          │   │                                  │
└──────────────────────────────────┘   └──────────────────────────────────┘
Result: Transaction B's balance update is completely overwritten and lost!
```

### 2.2 Memory Interleaving at the Hardware Layer
When multiple threads read and write to the same memory locations concurrently, the final state of the program depends entirely on the exact order in which the scheduler interleaves their instructions.

#### Non-Atomic Assignment Interference
*Initial State: Shared variable $y = 12$.*
* **Thread A:** `x = 1; x = y + 1;`
* **Thread B:** `y = 2; y = y * 2;`

Depending on where context switches occur, the variable $x$ can non-deterministically result in one of three values:
* **Scenario 1 ($x = 13$):** Thread A runs to absolute completion before Thread B begins execution ($x = 12 + 1 = 13$).
* **Scenario 2 ($x = 5$):** Thread B runs to completion first ($y$ becomes 4), and then Thread A executes ($x = 4 + 1 = 5$).
* **Scenario 3 ($x = 3$):** Thread B executes its first assignment ($y = 2$), then Thread A preempts it and runs completely ($x = 2 + 1 = 3$), and finally Thread B finishes ($y = 2 \times 2 = 4$).

#### Explaining Word Corruption Math
*Initial State: Shared variable $x = 0$.*
* **Thread A:** `x = 1;` (Binary: `0001`)
* **Thread B:** `x = 2;` (Binary: `0010`)

On standard modern architectures, word-sized loads and stores are guaranteed to be atomic by the hardware, meaning $x$ will cleanly result in either `1` or `2`. 

However, on older architectures with multi-word writes (such as double-precision floating-point values or bulk array copies), a write operation can be split across multiple bus cycles. If a context switch occurs mid-write, the bits from both threads can interleave. For example, if Thread A writes the lower bits (`...0001`) and Thread B overwrites the upper bits (`...0010`), the combined result could be `0011`, producing an impossible final value of `3`.

### 2.3 Case Study: Non-Atomic Global Counters
Consider a program where two threads compete to modify a single shared counter variable `i`:
* **Thread A:** Loops while `i < 10`, executing `i = i + 1;`.
* **Thread B:** Loops while `i > -10`, executing `i = i - 1;`.

#### Register Interference Trace
At the assembly level, incrementing and decrementing memory variables are non-atomic operations that require loading data into registers, modifying the value, and writing it back:

```
        Thread A Context (r1)                 Shared Memory (i)               Thread B Context (r1)
                                                ┌───────────┐
                                                │   i = 0   │
1. load  r1, M[i]   (r1 = 0)                    ├───────────┤
2. add   r1, r1, 1  (r1 = 1)                    │   i = 0   │
                                                ├───────────┤                 3. load  r1, M[i]   (r1 = 0)
                                                │   i = 0   │                 4. sub   r1, r1, 1  (r1 = -1)
5. store M[i], r1                               ├───────────┤
                                                │   i = 1   │
                                                ├───────────┤                 6. store M[i], r1
                                                │   i = -1  │
                                                └───────────┘
```
*Lost update trace on a multiprocessor or uniprocessor with hyperthreading.*

In this scenario, Thread A increments the counter in its local register, but before it can write the update back to memory, Thread B reads the original value (`0`). When Thread A writes its updated value (`1`), Thread B immediately overwrites it with `-1`. 

This data race can cause updates to be lost indefinitely on both uniprocessors and multiprocessors, making the final outcome of the loop completely non-deterministic.

### 2.4 Real-World Impact: The Therac-25 Disasters
Failing to synchronize shared variables can have catastrophic real-world consequences, as seen in the infamous **Therac-25 radiation therapy machine accidents**.


The Therac-25 was a computer-controlled radiation therapy system where an electron accelerator was managed entirely by software. The system suffered from a series of race conditions on shared configuration variables due to poor software design. 

If an operator updated prescription data rapidly on the control console, the high-speed data entry triggered a race condition. The software failed to detect that the physical turntable had moved into an unshielded position, delivering raw, massive radiation overdoses that directly caused the deaths of several patients.

## Section 3: Synchronization Definitions & High-Level Lock Interfaces

To prevent data races and ensure deterministic behavior, cooperating threads must use synchronization primitives.

### 3.1 Core Definitions
* **Synchronization:** The use of atomic operations to coordinate the execution of multiple threads to ensure correct behavior when accessing shared data.
* **Mutual Exclusion:** A mechanism that guarantees only one thread can perform a specific operation or access a shared resource at any given time.
* **Critical Section:** A block of code that accesses a shared resource and must not be entered by more than one thread concurrently. Mutual exclusion is the mechanism used to protect a critical section.

### 3.2 The Lock Abstraction
A **Lock** (or Mutex) is a software object used to enforce mutual exclusion. It exposes a clean, simple interface with two fundamental atomic operations:
* `acquire(&mylock)`: Attempts to acquire the lock. If the lock is free, the calling thread marks it as busy and continues executing. If the lock is busy, the calling thread blocks and waits until the lock becomes available.
* `release(&mylock)`: Marks the lock as free. This operation should only be called by the thread currently holding the lock, allowing waiting threads to proceed.

```c
// Fixing the Bank Transaction Race Condition using Locks
void Deposit(int acctId, double amount) {
    acquire(&mylock); // Block other threads from entering the critical section
    
    /* --- START CRITICAL SECTION --- */
    acct = GetAccount(acctId);
    acct->balance += amount; // Protected allocation update
    /* ---- END CRITICAL SECTION ---- */
    
    release(&mylock); // Allow the next waiting thread to enter
    StoreAccount(acct);
}
```
For this synchronization model to work correctly, **all cooperating functions (such as Withdraw or Transfer) must share and use the exact same lock variable** when accessing the underlying data structure.

### 3.3 The Synchronization API Layers
To build higher-level synchronization primitives, operating systems stack abstractions, moving from raw hardware instructions to user-friendly APIs:

> **[Diagram]: Synchronization Abstraction Layers**
> ```text
> ┌────────────────────────────────────────────────────────┐
> │   Programs / Shared Programs                           │
> ├────────────────────────────────────────────────────────┤
> │   Higher-Level API                                     │
> │   (Locks, Semaphores, Monitors, Send/Receive)          │
> ├────────────────────────────────────────────────────────┤
> │   Hardware Primitives                                  │
> │   (Load/Store, Disable Ints, Test&Set, Compare&Swap)   │
> └────────────────────────────────────────────────────────┘
> ```
> *We implement various higher-level synchronization primitives using atomic operations, since it is painful to build everything directly using just load/store atomic primitives.*

## Section 4: The Classical "Too Much Milk" Coordination Dilemma

To understand how synchronization primitives are built, we look at a classic coordination problem: ensuring room-mates do not accidentally buy too much milk.

```
Person A Timeline                     Shared Fridge State             Person B Timeline
┌─────────────────────────┐           ┌─────────────────┐             ┌─────────────────────────┐
│ Look in fridge: no milk │ ────────> │  Fridge Empty   │             │                         │
│ Leave for the store     │           ├─────────────────┤             │                         │
│                         │           │  Fridge Empty   │ <─────────  │ Look in fridge: no milk │
│ Buy milk at store       │           ├─────────────────┤             │ Leave for the store     │
│ Arrive home; stock milk │ ────────> │  Milk = 1 Pack  │             │                         │
│                         │           ├─────────────────┤             │ Buy milk at store       │
│                         │           │  Milk = 2 Packs │ <─────────  │ Arrive home; stock milk │
└─────────────────────────┘           └─────────────────┘             └─────────────────────────┘
Result: The roommates have failed to coordinate, resulting in too much milk!
```

### 4.1 Defining Correctness Properties
Any valid solution to this coordination problem must guarantee two core correctness properties:
1. **Safety:** Never more than one person buys milk at the same time.
2. **Liveness:** Someone will always buy milk if it is needed.

We will try to solve this problem using only basic, atomic memory reads and writes (loads and stores) as our building blocks. Note that using a physical key lock on the refrigerator door is a poor abstraction; it over-locks the resource and prevents roommates from accessing other items like orange juice.

### 4.2 Solution #1: Note as a Lock (Check-Then-Leave)
In this attempt, a roommate leaves a note to signal they are going to buy milk, and removes it once they return.

```c
// Solution #1 Implementation Code
if (noMilk) {
    if (noNote) {
        leave Note;
        buy milk;
        remove note;
    }
}
```
* **Failure Analysis:** This attempt fails because checking the conditions and leaving the note are separate, non-atomic actions. Thread A can check that there is no milk and no note, but if a context switch occurs right before it executes `leave Note;`, Thread B can run, see that there is no note, and proceed to buy milk as well. This leads to intermittent, unpredictable duplication bugs that are incredibly difficult to debug.

### 4.3 Solution #1.5: Leave Note First (Leave-Then-Check)
To fix Solution #1, we try leaving the note first before checking the fridge status.

```c
// Solution #1.5 Implementation Code
leave Note;
if (noMilk) {
    if (noNote) {
        buy milk;
    }
}
remove Note;
```
* **Failure Analysis:** This version causes complete starvation and livelock. If Thread A leaves a note, it will immediately fail its own internal check because `noNote` is now false, causing it to remove the note and exit without ever buying milk. As a result, no thread will ever buy milk.

### 4.4 Solution #2: Labeled Private Notes
Next, we try using distinct, labeled notes for each thread so they can track each other's status independently.

```c
// Thread A Code       // Thread B Code
leave note A;                               leave note B;
if (noNote B) {                             if (noNote A) {
    if (noMilk) {                               if (noMilk) {
        buy Milk;                                   buy Milk;
    }                                           }
}                                           }
remove note A;                              remove note B;
```
* **Failure Analysis:** This approach introduces a serious **Starvation / Livelock** vulnerability. If Thread A leaves note A and is immediately context-switched out before checking note B, Thread B can run and leave note B. When both threads resume, they will see each other's notes, assume the other roommate is handling the errand, and remove their own notes without anyone ever buying milk.

### 4.5 Solution #3: Rigorous Two-Note Asymmetric Coordination
To build a fully working solution using only atomic loads and stores, we must break the symmetry between the two threads, making their execution paths structurally distinct.

```c
// Thread A Implementation   // Thread B Implementation
leave note A;                                     leave note B;
while (note B) {                                  if (noNote A) {
    /* busy-wait loop: do nothing */                 if (noMilk) {
}                                                     buy milk;
if (noMilk) {                                        }
    buy milk;                                     }
}                                                 remove note B;
remove note A;
```

#### Complete Mathematical Safety Proof Walkthrough
We verify that Solution #3 is perfectly safe by tracing all possible execution timelines across two exhaustive cases:

##### Case 1: Thread A Leaves its Note First
Thread A executes `leave note A` before Thread B evaluates its conditional check `if (noNote A)`.

```
Thread A                                      Thread B
┌────────────────────────────────────────┐    ┌──────────────────────────────────┐
│ leave note A;                          │    │                                  │
│               [ --- Context Switch Intercepts Execution --- ]                  │
│                                        │    │ leave note B;                    │
│                                        │    │ if (noNote A) { --> Evaluates to │
│                                        │    │   FALSE! Skip buying path.       │
│                                        │    │ }                                │
│                                        │    │ remove note B;                   │
│               [ --- Context Switch Restores Thread A --- ]                     │
│ while (note B) {                       │    │                                  │
│   // Exits loop as note B is removed   │    │                                  │
│ }                                      │    │                                  │
│ if (noMilk) { buy milk; }              │    │                                  │
│ remove note A;                         │    │                                  │
└────────────────────────────────────────┘    └──────────────────────────────────┘
Result: Safe. Only Thread A buys milk.
```

##### Case 2: Thread B Reaches its Conditional Check First
Thread B evaluates `if (noNote A)` before Thread A executes its initialization assignment `leave note A`.

```
Thread A                                      Thread B
┌────────────────────────────────────────┐    ┌──────────────────────────────────┐
│                                        │    │ leave note B;                    │
│                                        │    │ if (noNote A) { --> Evaluates to │
│                                        │    │   TRUE! Enter critical section.  │
│               [ --- Context Switch Intercepts Execution --- ]                  │
│ leave note A;                          │    │                                  │
│ while (note B) {                       │    │                                  │
│   // Trapped in busy-wait loop!        │    │                                  │
│ }                                      │    │                                  │
│               [ --- Context Switch Restores Thread B --- ]                     │
│                                        │    │   if (noMilk) { buy milk; }      │
│                                        │    │ remove note B;                   │
│               [ --- Context Switch Restores Thread A --- ]                     │
│ // note B is now clear; loop exits     │    │                                  │
│ if (noMilk) {                          │    │                                  │
│   // Evaluates to FALSE! Milk is full. │    │                                  │
│ }                                      │    │                                  │
│ remove note A;                         │    │                                  │
└────────────────────────────────────────┘    └──────────────────────────────────┘
Result: Safe. Only Thread B buys milk; Thread A busy-waits and then skips.
```

This asymmetric approach can be generalized to support $n$ competing threads using **Leslie Lamport's Bakery Algorithm (1974)**.

#### Downsides of Pure Load/Store Coordination
While Solution #3 is correct, it has significant engineering drawbacks for real-world development:
* **High Complexity:** The logic is highly asymmetric and difficult to reason about, making it prone to errors as systems scale.
* **Lack of Generalization:** The code must be tailored differently for each participating thread, meaning you cannot easily run the same code block across an arbitrary number of worker threads.
* **Resource Waste:** While Thread A waits for Thread B, it traps the CPU core in a continuous, high-speed spin loop (`while (note B)`). This is known as **busy-waiting**, and it wastes significant clock cycles and power.

To avoid these limitations, modern architectures provide specialized hardware instructions to build clean, efficient synchronization primitives like locks.

## Section 5: Uniprocessor Lock Implementations (Interrupt Disabling)

On single-core processors, concurrency is achieved entirely by time-slicing threads via interrupts. We can exploit this behavior to build uniprocessor-safe synchronization primitives.

### 5.1 Naïve Interrupt Disabling
Because the operating system scheduler can only regain control of a single core through internal events or external hardware interrupts, a thread can achieve absolute atomicity by temporarily disabling all maskable hardware interrupts.

```c
// Naïve Uniprocessor Lock Model
void LockAcquire() { disableInterrupts(); } //
void LockRelease() { enableInterrupts();  } //
```

#### Operational Risks
This naïve approach has severe system security and stability flaws:
* **Denial of Service:** If the kernel exposes raw interrupt controls to user mode, a malicious or buggy program could call `LockAcquire()` and enter an infinite loop (`while(TRUE);`). Because the timer interrupt is disabled, the kernel can never regain control, completely freezing the computer.
* **Real-Time I/O Failure:** Disabling interrupts delays all incoming hardware notifications. Critical safety alerts—such as an automated industrial warning packet like `"Reactor about to meltdown. Help?"`—will be lost or delayed, creating severe real-world hazards.

### 5.2 The Guarded-Variable Internal Lock Model
To implement locks safely on a uniprocessor, the kernel restricts interrupt disabling to its own internal functions, keeping the time spent with interrupts disabled incredibly short.

```c
// Guarded Variable Internal Uniprocessor Lock
int value = FREE; // System-wide lock tracking variable

void Acquire() {
    disableInterrupts(); // Enter a short internal "meta"-critical section
    if (value == BUSY) {
        putThreadOnWaitQueue(current_thread); // Suspend execution
        sleep();                              // Yield the CPU
    } else {
        value = BUSY;                         // Grab the lock cleanly
    }
    enableInterrupts();  // Exit the internal "meta"-critical section
}

void Release() {
    disableInterrupts();
    if (anyoneOnWaitQueue()) {
        thread = removeThreadFromWaitQueue();
        placeOnReadyQueue(thread);            // Wake up a waiting thread
    } else {
        value = FREE;                         // Mark the lock as available
    }
    enableInterrupts();
}
```

### 5.3 The Enable Position Problem & Inter-Thread Interrupt Hand-off
When a thread must go to sleep inside `Acquire()`, choosing exactly when to re-enable interrupts introduces a subtle and dangerous race condition.

* **Re-enabling before putting the thread on the queue:** If interrupts are enabled before updating the wait queue, an incoming interrupt can trigger a context switch. A concurrent thread could call `Release()`, find the wait queue empty, and exit. When the original thread resumes, it will proceed to place itself on the queue and go to sleep, missing the wakeup signal entirely and hanging permanently.
* **Re-enabling after queuing but before calling `sleep()`:** If interrupts are enabled after updating the queue but before entering the scheduler, a concurrent `Release()` can pull the thread off the wait queue and place it on the ready queue. However, the original thread still thinks it needs to sleep, so it will continue into the scheduler and block anyway, causing a permanent deadlock.

#### The Solution: The Next-Thread Interrupt Hand-off Protocol
To solve this race condition, the thread must call `sleep()` with interrupts **fully disabled**. The responsibility for re-enabling interrupts is then shifted to the *next* thread scheduled by the kernel.

```
  Thread A Execution Path                          Thread B Execution Path
┌─────────────────────────┐                      ┌─────────────────────────┐
│ disableInterrupts();    │                      │                         │
│ put on wait queue       │                      │                         │
│ sleep() --> (Switches) ─┼───[Context Switch]──>│ (Awakes from sleep)     │
│                         │                      │ enableInterrupts();     │
└─────────────────────────┘                      └─────────────────────────┘
Every thread that wakes up inside the kernel scheduler returns from its own past
switch call and immediately re-enables interrupts.
```

### 5.4 Step-by-Step Simulation Trace
Let's trace two threads, A and B, interacting through this uniprocessor lock implementation:

#### Timeline Trace State Matrix
| Step | Active Thread | Executing Action | `value` State | Wait Queue | Thread A State | Thread B State |
| :---: | :--- | :--- | :---: | :---: | :---: | :---: |
| **1** | Thread A | Calls `Acquire()`; traps inside meta-section | `0` | Empty | **Running** | Ready |
| **2** | Thread A | Sets `value = 1`; exits meta-section and enters critical section | `1` | Empty | **Running** | Ready |
| **3** | — | **Timer Interrupt Fires; Scheduler Context Switch occurs** | `1` | Empty | Ready | **Running** |

| **7** | Thread A | Disables interrupts, pops Thread B off wait queue, and moves it to ready queue | `1` | Empty | **Running** | Ready |

## Section 6: Multiprocessor Lock Implementations (Atomic Instructions & Spinlocks)

On multi-core and multiprocessor hardware, disabling interrupts is not sufficient to enforce synchronization. Disabling interrupts on one core does not stop other cores from executing instructions and accessing shared memory simultaneously. Broadcasting interrupt-disable messages across all cores is far too slow and resource-intensive to be practical.

Instead, multiprocessor systems rely on hardware-level **Atomic Read-Modify-Write (RMW) Instructions**. These instructions read, modify, and write back a memory location in a single, indivisible hardware operation, coordinated across cores via cache coherence protocols.

### 6.1 Hardware Read-Modify-Write Semantic Blocks
Modern CPU architectures provide several RMW primitives:

```c
// 1. Test-And-Set Architecture Semantic Block
int test&set(int *address) {
    int result = *address; // Read current value from memory
    *address = 1;          // Set value to 1 atomically
    return result;         // Return the original value
}

// 2. Atomic Swap Architecture Semantic Block (x86 Idiom)
void swap(int *address, int *register_val) {
    int temp = *address;
    *address = *register_val;
    *register_val = temp; // Swap values between register and memory atomically
}

// 3. Compare-And-Swap Architecture Semantic Block
bool compare&swap(int *address, int expected_val, int new_val) {
    if (*address == expected_val) {
        *address = new_val; // Update value if it matches expectation
        return SUCCESS;
    }
    return FAILURE;
}

// 4. Load-Linked & Store-Conditional Architecture Semantic Block (R4000, alpha)
// Uses a loop to load a value, compute, and conditionally store it back.
void load_linked_store_conditional(int *address) {
    loop:
        ll r1, M[address];         // Load Linked
        movi r2, 1;                // Can do arbitrary computation
        sc r2, M[address];         // Store Conditional
        beqz r2, loop;             // Retry if store conditional failed
}
```

### 6.2 Basic Multiprocessor Spinlock via `test&set`
Using these hardware primitives, we can build a simple multiprocessor lock called a **Spinlock** that operates entirely in user space without requiring kernel context switches.

```c
// Basic test&set Spinlock Implementation
int thelock = 0; // 0 = FREE, 1 = BUSY

void acquire(int *thelock) {
    while (test&set(thelock)); // Spin loop: busy-wait until thelock becomes 0
}

void release(int *thelock) {
    *thelock = 0; // Reset the lock value atomically
}
```

#### Operational Bottlenecks: Cache Line Ping-Ponging
While correct, basic spinlocks have severe performance flaws when highly contended:
* **High CPU Overhead:** Waiting threads consume 100% of their CPU time spinning in the `while` loop, wasting clock cycles that could be used by other threads.
* **Cache Line Ping-Ponging:** Because `test&set` is an atomic write instruction, every single iteration of the spin loop attempts to write to memory. This forces the underlying cache coherence protocol to invalidate that cache line across all other CPU cores. The cache line is forced to constantly bounce back and forth between cores ("ping-ponging"), consuming massive amounts of interconnect network bandwidth and slowing down the entire system.
* **Priority Inversion Vulnerability:** Spinlocks are highly susceptible to priority inversion. If a high-priority thread attempts to acquire a spinlock held by a low-priority thread, it will spin at 100% CPU utilization. On a single-core system, this completely starves the low-priority thread, preventing it from ever running to release the lock, locking up the system indefinitely. This exact bug famously crippled the **Mars Pathfinder Rover mission in 1997**.

### 6.3 The Optimized `test&test&set` Spinlock
To eliminate excessive interconnect network overhead, systems developers use an optimized pattern called a **`test&test&set` lock**.

```c
// Optimized test&test&set Spinlock Implementation
void acquire(int *thelock) {
    do {
        while (*thelock); // 1. Pure Read Spin: Loop locally in cache without issuing writes
    } while (test&set(thelock)); // 2. Atomic Try: Only attempt an actual write when the lock appears free
}
```
* **Performance Profile:** By spinning on a plain memory read (`while (*thelock)`), the cache line remains in a shared, read-only state across all waiting CPU cores. Waiting threads read the value directly from their local L1/L2 caches without generating any network or bus traffic. Only when the lock holder writes `0` during release is the cache line invalidated, prompting the waiting cores to drop out of the inner loop and execute `test&set` to compete for the lock.

### 6.4 Guarded Sleep Lock with test&set
To completely eliminate long-term busy-waiting, we can combine a short spinlock (called a `guard`) with an internal kernel block queue.

```c
// Guarded Sleep Lock Implementation
int guard = 0;   // Short spinlock protecting access to the lock's internal metadata
int thelock = FREE; // Actual lock status variable

void acquire(int *thelock) {
    while (test&set(&guard)); // Spin briefly to acquire the guard
    
    if (*thelock == BUSY) {
        putThreadOnWaitQueue(current_thread); // Queue the thread
        go_to_sleep_and_release_guard(&guard); // Atomically sleep and reset guard = 0
        // On wakeup, the guard is guaranteed to be 0!
    } else {
        *thelock = BUSY; // Acquire the actual lock
        guard = 0;       // Release the guard
    }
}

void release(int *thelock) {
    while (test&set(&guard)); // Spin briefly to acquire the guard
    
    if (anyoneOnWaitQueue()) {
        thread = removeThreadFromWaitQueue();
        placeOnReadyQueue(thread); // Wake up a waiting thread; hold thelock as BUSY
    } else {
        *thelock = FREE;           // Mark the lock as available
    }
    guard = 0;                     // Release the guard
}
```
* **Note on Guard Reset:** The guard variable must be reset *atomically* as part of the sleep call. Resetting the guard *before* calling sleep creates a race condition where a concurrent release can check the queue and miss the wakeup. Resetting it *after* sleep is impossible, as the thread is already asleep and cannot execute instructions.

### 6.5 Architectural Lock Mechanics Comparison Matrix
The table below compares the implementation trade-offs of the different locking mechanisms covered so far:

| Dimension | Raw Interrupt Lock | Guarded Interrupt Lock | Raw Test-And-Set Lock | Guarded Test-And-Set Lock |
| :--- | :--- | :--- | :--- | :--- |
| **Multiprocessor Safe?** | **No** | **No** | **Yes** | **Yes** |
| **Long Busy-Waiting?** | **No** | **No** | **Yes** (Consumes 100% CPU) | **No** (Short spin on guard only) |
| **System Impact** | Severe; blocks all system activity and scheduling. | Minimal; interrupts are disabled only for a few instructions. | Wasteful; causes severe cache invalidation traffic. | Efficient; balances short spin times with kernel sleeping. |
| **Argument Utilization** | Lock argument is completely ignored. | Lock variable tracks state changes. | Operates entirely in user space. | Integrates user space tracking with kernel blocking. |

## Section 7: Modern OS Synchronization (Linux Futex Architecture)

To maximize performance, modern operating systems avoid entering the kernel for locking unless absolutely necessary. This optimization is achieved using the Linux **Futex** (Fast Userspace Mutex) architecture.

### 7.1 The Futex System Call API
```c
#include <linux/futex.h>
int futex(int *uaddr, int futex_op, int val, const struct timespec *timeout); //
```
* `uaddr`: A pointer to a standard 32-bit integer alignment tracking variable located entirely within user-space memory.
* `futex_op`: The requested operational command:
  * `FUTEX_WAIT`: If the value at `uaddr` is equal to `val`, the kernel puts the current thread to sleep until a corresponding `FUTEX_WAKE` call is issued. This check is performed atomically inside the kernel to prevent race conditions.
  * `FUTEX_WAKE`: Wakes up at most `val` threads currently blocked on the `uaddr` memory location.
  * `FUTEX_FD`, `FUTEX_WAKE_OP`, `FUTEX_CMP_REQUEUE`: More advanced operations for complex wait/wake routing.
* `timeout`: An optional pointer to a `timespec` structure that specifies a timeout for the operation.

### 7.2 Evolutionary Futex Implementations

#### Implementation Variant 1: Naïve Futex Integration
This initial attempt uses `test&set` alongside `FUTEX_WAIT` to eliminate long-term busy-waiting.

```c
// Futex Variant 1 Implementation
int mylock = 0; //

void acquire(int *thelock) {
    while (test&set(thelock)) {
        // If the lock is busy, call futex to put the thread to sleep conditionally
        futex(thelock, FUTEX_WAIT, 1, NULL); //
    }
}

void release(int *thelock) {
    *thelock = 0; // Unlock the state variable
    futex(thelock, FUTEX_WAKE, 1, NULL); // Wake up a waiting thread
}
```
* **Performance Profile:** While this eliminates long-term busy-waiting, it introduces a new performance issue: every single call to `release()` must invoke the `futex` system call to wake up potential waiters, even if no other threads are waiting. This creates unnecessary system call overhead for uncontended locks.

#### Implementation Variant 2: The Waiter-Flag Optimization
To avoid unnecessary system calls, we introduce a boolean flag `maybe_waiters` to track if any threads are currently blocked on the lock.

```c
// Futex Variant 2 Implementation
int mylock = 0;
bool maybe_waiters = false; // Flag to track potential waiters

void acquire(int *thelock, bool *maybe) {
    while (test&set(thelock)) {
        *maybe = true; // Mark that a thread is about to go to sleep
        futex(thelock, FUTEX_WAIT, 1, NULL); // Sleep on the lock variable
        *maybe = true; // Ensure subsequent sleepers are not permanently stuck
    }
}

void release(int *thelock, bool *maybe) {
    *thelock = 0; // Unlock the state variable
    if (*maybe) {
        *maybe = false; // Reset the waiter flag
        futex(thelock, FUTEX_WAKE, 1, NULL); // Issue the system call to wake up a thread
    }
}
```
* **Performance Profile:** This implementation successfully eliminates system calls when the lock is uncontended, as `release()` will skip the `futex` call if `maybe_waiters` is false. However, managing concurrent updates to both `thelock` and `maybe_waiters` introduces subtle edge-case race conditions.

#### Implementation Variant 3: The Optimized Three-State Futex Architecture
To resolve the coordination issues of Variant 2, we pack both the lock status and the waiter tracking into a single, unified 32-bit integer that supports three distinct states:
1. `UNLOCKED` (0): The lock is fully available; no threads hold it.
2. `LOCKED` (1): A single thread currently holds the lock; no other threads are waiting.
3. `CONTESTED` (2): A thread holds the lock, and one or more additional threads are currently sleeping in the kernel waiting to acquire it.

```c
// Optimized Three-State Futex Implementation
typedef enum { UNLOCKED, LOCKED, CONTESTED } Lock; //
Lock mylock = UNLOCKED; //

void acquire(Lock *thelock) {
    // Fast Path: Attempt to grab the lock assuming it's completely uncontended
    if (compare&swap(thelock, UNLOCKED, LOCKED)) 
        return; // Success! No system call or overhead incurred

    // Slow Path: The lock is busy; loop and prepare to sleep in the kernel
    while (swap(thelock, CONTESTED) != UNLOCKED) {
        // Sleep using futex unless another thread releases the lock right before we wait
        futex(thelock, FUTEX_WAIT, CONTESTED, NULL); //
    }
}

void release(Lock *thelock) {
    // Atomically reset the lock to UNLOCKED and check its previous state
    if (swap(thelock, UNLOCKED) == CONTESTED) {
        // If the lock was CONTESTED, we must issue a system call to wake up a sleeper
        futex(thelock, FUTEX_WAKE, 1, NULL); //
    }
}
```
This multi-state design is the gold standard for high-performance libraries, as it ensures zero system call overhead in the common, uncontended case while seamlessly falling back to kernel-level sleeping when thread contention occurs.

## Section 8: Lock-Free Concurrency Design Pattern

While locks ensure correctness, they can still introduce bottlenecks due to thread blocking and scheduling overhead. For highly performance-critical structures, developers can use atomic instructions to build **Lock-Free Data Structures** that coordinate threads without ever using explicit locks.


### 8.1 Example: Lock-Free Concurrent Queue Insertion
The following function shows how to use the `compare&swap` instruction to safely append a new node to the head of a shared singly linked list without using any locks:

```c
// Lock-Free Queue Enqueue Implementation
void addToQueue(Node *object) {
    do {
        // 1. Read the current head of the queue into a local register
        Node *current_head = M[root]; //
        
        // 2. Set our new node's next pointer to point to this expected head
        object->next = current_head;  //
        
    // 3. Atomically update the root pointer to our new node IF the head hasn't changed
    // If another thread updated the root in the meantime, compare&swap fails and we retry
    } while (!compare&swap(&root, current_head, object)); //
}
```
By using an optimistic retry loop with `compare&swap`, this pattern guarantees that at least one thread will make forward progress on every iteration, achieving high-throughput concurrency without the risk of deadlocks or priority inversions.

## Section 9: Lecture Summary

* **Atomic Operations:** A fundamental building block—an operation that runs to completion or not at all. Used to construct various synchronization primitives.
* **Hardware Atomicity Primitives:** Architectures provide basic tools like Disabling Interrupts, `test&set`, `swap`, `compare&swap`, and `load-linked`/`store-conditional`.
* **Lock Construction:** 
  * Developers must be careful not to waste or tie up machine resources (e.g., shouldn't disable interrupts for long, shouldn't spin-wait for long).
  * **Key Idea:** Use a separate lock variable and employ hardware mechanisms to protect modifications to that variable.
* **User-Level Locks:** OS primitives (like Linux's `futex`) package up the functionality of sleeping conditionally, allowing user-level locks to be fast when uncontended and safely trap into the kernel when contended.

### Question 1: Uniprocessor Interrupt Enable/Disable Placement Analysis
An engineer proposes an alternative implementation for the internal kernel `Acquire()` lock function on a uniprocessor, placing the `enableInterrupts()` call directly before the `sleep()` command to simplify the code:

```c
void BrokenAcquire() {
    disableInterrupts();
    if (value == BUSY) {
        putThreadOnWaitQueue(current_thread);
        enableInterrupts(); // Alternate placement position
        sleep();
    } else {
        value = BUSY;
        enableInterrupts();
    }
}
```
Provide a detailed interleaving trace demonstrating exactly how this modified function can break the system's correctness guarantees.

#### Solution:
This alternative placement introduces a critical race condition known as a **Lost Wakeup Fault**, which can lead to permanent system deadlocks. 

Consider the following execution timeline involving two threads where Thread A currently holds the lock (`value == BUSY`):
1. **Thread B** calls `BrokenAcquire()`, disables interrupts, discovers that `value == BUSY`, and places its TCB on the lock's wait queue.
2. **Thread B** executes the line `enableInterrupts()`.
3. Right before Thread B can execute the next instruction (`sleep()`), a **hardware timer interrupt fires**, intercepting execution and forcing a context switch to Thread A.
4. **Thread A** completes its work and calls `Release()`.
5. Inside `Release()`, Thread A disables interrupts and checks the wait queue. It finds Thread B on the queue, pops it off, places it on the scheduler's ready queue, and then re-enables interrupts and exits.
6. The scheduler selects **Thread B** to run next.
7. Thread B resumes execution exactly where it left off: it calls **`sleep()`**.
8. **The Breakdown:** Because Thread B enters the sleep state anyway, it completely misses the wakeup signal that was already issued by Thread A. Thread B will sleep indefinitely on the wait queue, and since it still technically holds the lock state, no other thread can ever acquire the lock, deadlocking that subsystem permanently.

### Question 2: Spinlock Cache Contention Analysis
Suppose a four-core multiprocessor system runs four threads that are heavily competing to acquire a single basic spinlock protected by a hardware `test&set` instruction. The lock variable is initially busy (`thelock == 1`).

1. If all four threads are executing a basic spinlock loop (`while (test&set(thelock));`), describe the impact on the shared hardware interconnect bus.
2. Explain how converting the lock to a `test&test&set` architecture (`while (*thelock);`) alters the cache line states and reduces bus overhead.

#### Solution:
1. **Basic Spinlock Impact:**
   The basic `test&set` instruction is treated as a write operation by the processor hardware, regardless of the value it reads. 
   * As a result, every single iteration of the spin loop across all four cores attempts to write to the lock variable's memory location.
   * Under standard cache coherence protocols (like MESI), a write operation requires the core to acquire exclusive ownership of the cache line, forcing it to send invalidation messages to all other cores.
   * This causes the cache line containing `thelock` to constantly bounce back and forth between the four cores' local caches ("cache line ping-ponging"). The shared interconnect bus becomes completely saturated with invalidation and data transfer traffic, severely degrading the performance of all other memory accesses on the system.

2. **Optimized `test&test&set` Alteration:**
   By modifying the lock loop to first check the value using a plain memory read (`while (*thelock);`), we change how the cache lines are managed:
   * A plain read instruction only requires shared, read-only access to the cache line.
   * This allows the cache line containing `thelock` to reside simultaneously in a Shared (S) state across the local L1/L2 caches of all four cores.
   * As long as the lock remains busy, all four cores read the value `1` directly from their own local caches without generating any network or interconnect bus traffic. The bus remains completely free until the lock holder releases the lock, making this design significantly more efficient.
