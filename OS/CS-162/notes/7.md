## Section 1: Architectural Paradigms of Network Servers

The design of a network server illustrates the classic progression of operating system abstractions, transitioning from isolated multi-process protection to lightweight, shared-memory multi-threading.

### 1.1 Multi-Process Server with Strict Protection (Iterative)
A baseline multi-process server can isolate client connections into independent address spaces using the `fork()` system call. 


```c
// Pre-sockets setup code executed by parent process
listen(server_socket, MAX_QUEUE); //

while (1) {
    // Accept a new incoming client connection, allocating a new socket descriptor
    int conn_socket = accept(server_socket, NULL, NULL); //
    
    pid_t pid = fork(); // Duplicate the parent process
    
    if (pid == 0) {
        /* Child Process Context */
        close(server_socket);       // Close the inherited listening socket to preserve boundaries
        serve_client(conn_socket);  // Handle client request within a sandboxed address space
        close(conn_socket);         // Release the client connection socket
        exit(0);                    // Terminate the child process safely
    } else {
        /* Parent Process Context */
        close(conn_socket);         // Release the parent's copy of the client socket
        wait(NULL);                 // Block parent until the child process finishes execution
    }
}
close(server_socket); //
```

#### Operational Bottlenecks
This design follows the standard command shell pattern. Because the parent process invokes a blocking `wait(NULL)` inside the primary loop, it cannot return to the `accept()` system call until the current client finishes and the child process terminates. While the network stack can queue incoming requests up to `MAX_QUEUE`, this server behaves **iteratively** and cannot handle multiple clients concurrently.

### 1.2 Multi-Process Concurrent Server
To convert the iterative server into a concurrent one, the parent process skips the blocking `wait(NULL)` call. This allows the parent to immediately loop back and accept the next client while the child handles the previous one.

```c
// Concurrent Multi-Process Server Loop
while (1) {
    int conn_socket = accept(server_socket, NULL, NULL); //
    pid_t pid = fork(); //
    
    if (pid == 0) {
        /* Child Process Context */
        close(server_socket); //
        serve_client(conn_socket); //
        close(conn_socket); //
        exit(0); //
    } else {
        /* Parent Process Context */
        close(conn_socket); // Parent returns immediately to accept()
        // wait(NULL); Omitted to achieve execution concurrency
    }
}
```

#### Architectural Trade-offs
* **Advantage:** Provides fault containment and protection. If a bug or crash occurs while servicing a client, the failure is isolated to that child's private address space, leaving the primary server process unaffected.
* **Disadvantage:** High system overhead. Allocating a full process requires duplicating page tables, tracking data structures, and memory maps. Furthermore, these independent processes must be careful not to introduce data races when interacting through shared resources like the global file system.

### 1.3 Lightweight Concurrent Server (Multi-Threaded)
To avoid process creation overhead, modern high-performance servers handle each connection in a separate thread within the same process address space using POSIX threads (`pthread_create`).

```c
// Main loop of a Multi-Threaded Concurrent Server
while (1) {
    int conn_socket = accept(server_socket, NULL, NULL); //
    pthread_t thread_id;
    
    // Spawn a lightweight thread sharing the process address space
    pthread_create(&thread_id, NULL, serve_client_wrapper, (void *)(long)conn_socket); //
}
```
* **Performance Profile:** Threads are much more efficient to create and context-switch than full processes because they do not require changing the active address space or reloading memory translation structures.
* **The Cost of Giving Up Protection:** Because all threads share the same address space and global file descriptors, they are highly susceptible to data races. A single memory corruption bug or synchronization error in one thread can corrupt shared structures or crash the entire server process.

### 1.4 The Thread Pool Architecture
The multi-threaded design above features **unbounded thread allocation**, which introduces a major stability risk: if the site faces a sudden surge in traffic, spawning an unlimited number of threads will exhaust system memory and degrade overall throughput. To prevent this, professional systems employ a **Thread Pool** model, bounding the maximum level of multiprogramming with a fixed set of worker threads.

```
Master Thread                    Shared FIFO Queue                Bounded Worker Thread Pool
┌──────────────────┐             ┌───────────────┐               ┌───────────────────────────┐
│ accept()         │ ──(Push)──> │ Connection FD │ ──(Pop)───>   │ Thread 1   Thread 2       │
│ Enqueue(con)     │             │ Connection FD │               │ Thread 3   Thread 4       │
│ wakeUp(queue)    │             └───────────────┘               │     │
└──────────────────┘                                             └───────────────────────────┘
```

#### Master and Worker Thread Algorithms
```c
// Master Thread Loop
master() {
    allocThreads(worker, queue); // Allocate a fixed pool of worker threads
    while(TRUE) {
        con = AcceptCon();       // Block waiting for a new client connection
        Enqueue(queue, con);     // Append connection handle to the shared queue
        wakeUp(queue);           // Signal and wake up a sleeping worker thread
    }
}

// Worker Pool Routine
worker(queue) {
    while(TRUE) {
        con = Dequeue(queue);    // Attempt to pop a connection off the queue
        if (con == null) {
            sleepOn(queue);      // Block and yield the CPU if no work is available
        } else {
            ServiceWebPage(con); // Handle client I/O requests
        }
    }
}
```

## Section 2: Kernel Management of File Descriptors & Aliasing under `fork()`

To manage process communication and sharing safely, the operating system kernel separates user-level file identifiers from the underlying file state.

### 2.1 The Process Control Block (PCB)
The kernel represents each individual process using a central record called the **Process Control Block (PCB)**. This structure tracks the process's execution status (Running, Ready, Blocked), saved register states, Process ID (PID), user ownership, scheduling priorities, memory space boundaries, and an indexed list of open files. The scheduler uses these PCBs to allocate CPU time and hardware resources based on its internal policy decisions.

### 2.2 Lifecycle of an Open File Descriptor
When an application calls `open("foo.txt")`, the kernel verifies permissions, allocates a system-wide **Open File Description** object, and returns its index from the per-process file descriptor table (e.g., index 3).


#### Step 1: Initial Allocation
```
Process Table (User Space) │ Kernel Space Table
┌────────────────────────┐ │ ┌─────────────────────────────────────────────────────┐
│ File Descriptor Table: │ │ │ Open File Description Object:                       │
│ [0] -> stdin           │ │ │ ┌─────────────────────────────────────────────────┐ │
│ [1] -> stdout          │ │ │ │ File Target : foo.txt                           │ │
│ [2] -> stderr          │ │ │ │ Current Pos : 0                                 │ │
│ [3] ───────────────────┼─┼─> │ Reference Cnt: 1│ │
└────────────────────────┘ │ │ └─────────────────────────────────────────────────┘ │
                           │ └─────────────────────────────────────────────────────┘
```

#### Step 2: Offset Tracking During Reads
When the process executes `read(3, buf, 100)`, the kernel reads 100 bytes from disk, copies them into user space, and automatically increments the file position pointer inside the Open File Description object to `100`.
```
Open File Description Object:
┌──────────────────────────────────────┐
│ File Target : foo.txt                │
│ Current Pos : 100│
│ Reference Cnt: 1                     │
└──────────────────────────────────────┘
```

#### Step 3: Descriptor Release
When the process executes `close(3)`, the kernel clears entry 3 from the process's file descriptor table and decrements the Open File Description's reference count. Once the reference count drops to zero, the object is deallocated.

### 2.3 File Descriptor Duplication and Sharing under `fork()`
When a process calls `fork()`, the child process inherits an exact copy of the parent's file descriptor table. However, the underlying entries are not duplicated; instead, they are **aliased**, meaning both the parent and child process tables point to the *same* system-wide Open File Description object.

```
Process 1 (Parent)
┌───────────────────────┐
│ FD [3] ───────────────┐│
└───────────────────────┘│   Kernel Space: Open File Description Object
                         ├──> ┌───────────────────────────────────────────────┐
Process 2 (Child)        │    │ File Target : foo.txt                         │
┌───────────────────────┐│    │ Current Pos : 100│
│ FD [3] ───────────────┘    │ Reference Cnt: 2                              │
└───────────────────────┘    └───────────────────────────────────────────────┘
```

#### Dynamic Behavior of Shared Offsets
Because the parent and child processes share the same Open File Description object, updates to the file pointer are visible to both:
1. If Process 1 executes `read(3, buf, 100)`, it reads data from offset 100 to 200, and the shared file pointer automatically advances to `200`.
2. If Process 2 executes a read call next, `read(3, buf, 100)`, it does *not* read from the beginning of the file; instead, it reads from offset 200 to 300, and advances the shared pointer to `300`.

#### Close Semantics on Aliased Fields
If Process 1 executes `close(3)`, its local entry is cleared and the object's reference count decrements from 2 to 1. The file remains open and accessible within Process 2 via its descriptor 3. The underlying Open File Description object remains alive until every process referencing it has closed its respective descriptor.

### 2.4 Practical Applications of File Descriptor Aliasing
This aliasing behavior is essential for sharing system resources across process boundaries.

#### Shared Terminal Emulators
When a parent shell spawns a child application via `fork()`, the standard I/O streams—`stdin` (0), `stdout` (1), and `stderr` (2)—are copied directly to the child's descriptor table. Because these descriptors are aliased to the same terminal device driver object, both the parent and child can write output using `printf` to the same physical terminal window without clobbering each other's configurations. If one process closes its terminal descriptor, it remains open and fully functional in the other.

## Section 3: Pipe Architecture & Inter-Process Communication

A **Pipe** is a unidirectional inter-process communication channel implemented as an internal queue inside kernel memory.

```c
#include <unistd.h>
int pipe(int pipe_fd[2]); // Allocates two file descriptors
```
* `pipe_fd[0]`: The **Read End** of the channel.
* `pipe_fd[1]`: The **Write End** of the channel.

### 3.1 Single-Process Pipe Lifecycle
While a single process can read and write to its own pipe, this pattern is rarely useful outside of multithreaded coordination.

```c
#include <unistd.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

#define BUFSIZE 256 //

int main(void) {
    char *msg = "Message in a pipe.\n"; //
    char buf[BUFSIZE];                  //
    int pipe_fd[2];                     //

    if (pipe(pipe_fd) == -1) {          // Instantiate the kernel queue
        fprintf(stderr, "Pipe failed.\n"); //
        return EXIT_FAILURE;               //
    }

    // Write bytes into the pipe's write end (pipe_fd[1])
    ssize_t writelen = write(pipe_fd[1], msg, strlen(msg) + 1); //
    printf("Sent: %s [%ld %ld]\n", msg, (long)(strlen(msg) + 1), (long)writelen); //

    // Read bytes out of the pipe's read end (pipe_fd[0])
    ssize_t readlen = read(pipe_fd[0], buf, BUFSIZE); //
    printf("Rcvd: %s [%ld]\n", buf, (long)readlen);   //

    close(pipe_fd[0]); //
    close(pipe_fd[1]); //
    return EXIT_SUCCESS;
}
```

### 3.2 Inter-Process Pipelines (`fork()` Integration)
To use a pipe for communication across process boundaries, developers combine `pipe()` and `fork()`. After the fork, the parent and child close their unused ends of the pipe to create a clean, unidirectional data channel.

```
Parent Process (FD Table)             Kernel Memory Buffer            Child Process (FD Table)
┌───────────────────────┐             ┌──────────────────┐            ┌───────────────────────┐
│ [4] Write End ────────┼──(write)──> │  FIFO Data Queue │ ──(read)──>│ [3] Read End ─────────│
└───────────────────────┘             └──────────────────┘            └───────────────────────┘
* Parent closes FD[3]                          * Child closes FD[4]
```

```c
// Process Pipeline Implementation
int pipe_fd[2];
pipe(pipe_fd); // 1. Create the pipe first so descriptors are inherited

pid_t pid = fork(); // 2. Duplicate the process

if (pid > 0) {
    /* Parent Process Context */
    close(pipe_fd[0]); // Close the unused read end of the pipe
    
    write(pipe_fd[1], msg, strlen(msg) + 1); // Send data to child
    close(pipe_fd[1]); // Close the write end when finished to flush EOF
} else if (pid == 0) {
    /* Child Process Context */
    close(pipe_fd[1]); // Close the unused write end of the pipe
    
    ssize_t readlen = read(pipe_fd[0], buf, BUFSIZE); // Blocks until data arrives
    printf("Child Rcvd: %s [%d]\n", buf, (int)readlen); //
    close(pipe_fd[0]);
    exit(0);
}
```

## Section 4: The Dispatch Loop & Thread Execution Lifecycles

At its core, thread scheduling and execution inside an operating system kernel boils down to a continuous loop called the **Dispatch Loop**.

```c
// Conceptual execution model of the kernel scheduler
Loop {
    RunThread();          // Transfer control of the core to a thread
    ChooseNextThread();   // Execute scheduling policy choices
    SaveStateOfCPU(curTCB);  // Save current thread state to its control block
    LoadStateOfCPU(newTCB);  // Load the next thread's context into CPU registers
}
```
This dispatch loop is an infinite loop that runs for the entire lifetime of the operating system. The kernel never exits this loop unless the machine is shut down or faces a critical hardware failure.

### 4.1 Transferring CPU Control to User Space (`RunThread`)
To execute a thread, the kernel loads its saved context (registers, stack pointers, and environment details) directly into the physical CPU core. It then performs a jump instruction to the thread's saved Program Counter (`PC`). 

At this exact instant, the kernel **yields complete physical control** of the processor core to the user-space code. While user instructions are executing, the operating system kernel is completely inactive on that core. To regain control and run the dispatch loop again, the kernel relies on either **Internal Events** or **External Events**.

### 4.2 Internal Events (Voluntary Relinquishment)
Internal events are triggered directly by the running thread voluntarily yielding its execution time. This occurs through three main paths:
1. **Blocking on I/O operations:** Requesting files or network data implicitly blocks the thread and yields the CPU.
2. **Synchronization waits:** A thread yields control when it blocks on a lock or waits for a signal from another thread.
3. **Explicit yielding:** Calling the `yield()` system call voluntarily relinquishes the remainder of the thread's time slice.

```c
// Example of a voluntary yielding loop
computePI() {
    while(TRUE) {
        ComputeNextDigit();
        yield(); // Voluntarily hand control back to the OS dispatcher
    }
}
```

## Section 5: Thread Synchronization Primitives & Internal Context Switching Mechanics

When a thread calls `yield()`, the system transitions from the user-level context down to the kernel-level stack to perform a switch.


### 5.1 The Two-Stack Execution Model
To prevent user code from compromising system stability, every thread has both a **User-Level Stack** and a **Kernel-Level Stack**. 
* **User-Level Stack:** Tracks standard function execution frames while running in user space (Mode Bit = 1).
* **Kernel-Level Stack:** Used when a thread traps into the kernel (via a system call, exception, or interrupt). The kernel cannot trust the user-space stack pointer, so it switches to this protected kernel stack to handle operations safely.

```c
// Kernel workflow to switch executing threads
run_new_thread() {
    newThread = PickNextRunnableThread(); // Policy Choice
    switch(curThread, newThread);          // Low-level assembly context switch
    ThreadHouseKeeping();                  // Deallocate terminated resources
}
```

### 5.2 Low-Level Assembly Context Switching Mechanics
The assembly-level `switch()` function unloads the register context from the active CPU core, saves it into the current thread's TCB, and loads the saved context of the next thread.

```assembly
# Conceptual implementation of switch(tCur, tNew)
switch:
    # 1. PRESERVE ACTIVE CONTEXT INTO OLD TCB (tCur)
    Store CPU.r0    -> TCB[tCur].regs.r0    # Save general-purpose registers
    Store CPU.r7    -> TCB[tCur].regs.r7    #
    Store CPU.sp    -> TCB[tCur].regs.sp    # Save active Stack Pointer
    Store CPU.retpc -> TCB[tCur].regs.retpc # Save Return Address pointer

    # 2. LOAD NEW CONTEXT FROM TARGET TCB (tNew)
    Load  TCB[tNew].regs.r0    -> CPU.r0    # Restore target register frames
    Load  TCB[tNew].regs.r7    -> CPU.r7    #
    Load  TCB[tNew].regs.sp    -> CPU.sp    # Switch to the new thread's stack pointer
    Load  TCB[tNew].regs.retpc -> CPU.retpc # Load the target return address

    return                                  # Jump to instructions at CPU.retpc
```

### 5.3 Technical Challenges and Risks in Switch Code
Writing context-switching code is exceptionally risky; because it manages low-level CPU registers directly, even a minor mistake can introduce silent, catastrophic bugs. 

For example, if the implementation forgets to save or restore a single register, the application will not crash immediately. Instead, the system will experience intermittent, non-deterministic failures that occur only when a context switch happens to interrupt a thread while it is actively using that specific register. Testing this code is incredibly difficult; because there are an infinite number of possible instruction interleavings, you cannot build an exhaustive test suite to catch every edge case.

#### Cautionary Tale: The Topaz Kernel Failure
To optimize execution speeds, engineers building the classic Topaz operating system kernel optimized their assembly context switch to save a single CPU instruction. This optimization was well-documented and worked perfectly—but it relied on a hard-coded assumption that the total kernel footprint would remain under 1 MB. 

As time passed, new developers added features to the kernel without removing old ones, eventually pushing its size past the 1 MB boundary. Because the original optimization constraint was forgotten, the system began experiencing completely unpredictable failures and data corruption. This serves as a warning to systems programmers: **always design for simplicity over clever optimizations**.

## Section 6: Preemption, External Events, & the Interrupt Architecture

### 6.1 The Need for Preemption
If an operating system relied solely on voluntary internal events, a buggy or malicious application could easily hijack the computer. For example, if a program computing PI falls into an infinite loop without performing any I/O or calling `yield()`, it would hog the processor core indefinitely and lock up the system. 

To keep any single thread from monopolizing resources, the operating system uses **External Events** to forcefully reclaim control.

### 6.2 Preemption via Timer Interrupts
The primary tool for preemption is the hardware **Timer**. The timer acts like a physical alarm clock, periodically sending an interrupt signal to the CPU at a fixed interval (typically every few milliseconds). 

When the timer fires, the CPU hardware intercepts execution, flips the mode bit to enter kernel mode, and branches directly to the kernel's timer interrupt handler. This handler runs periodic housekeeping tasks and invokes `run_new_thread()` to force a scheduling decision, ensuring the system remains responsive.

```
Running Thread Context                 Hardware Timer Tick             Kernel Preemption Path
┌─────────────────────────┐            ┌─────────────────┐             ┌─────────────────────┐
│ Instruction 1           │            │                 │             │ TimerInterrupt()    │
│ Instruction 2 (Interrupt) ──(Trap)──> │ Fire Interrupt  │ ─────────>  │ Housekeeping        │
│ Instruction 3           │            │ │     │ run_new_thread()    │
└─────────────────────────┘            └─────────────────┘             └─────────────────────┘
```

### 6.3 The Hardware Interrupt Controller
Hardware interrupts are managed and routed using a specialized component called the **Interrupt Controller**.


The interrupt controller features several core mechanisms to manage device signals:
* **Interrupt Mask:** A configurable bitmask that allows the operating system to selectively enable or disable individual device interrupt lines.
* **Priority Encoder:** Evaluates all active, enabled hardware signals and selects the one with the highest priority to forward to the CPU.
* **Global Interrupt Disable Flag:** An internal CPU register bit that allows the kernel to temporarily turn off all maskable interrupts during critical code sections.
* **Non-Maskable Interrupt (NMI):** A dedicated, high-priority hardware line reserved for critical errors (like RAM parity failures or hardware faults) that cannot be disabled by the CPU.

### 6.4 Lifecycle of a Hardware Interrupt Context Switch
Unlike a system call, a hardware interrupt is an unexpected, involuntary context switch that handles requests immediately without evaluating what to run next. When an external device (such as a network card) sends an interrupt signal, the CPU handles it through a strict hardware and software lifecycle:

```
[ Active User Instruction Execution Pipeline ]
                      │
            Asynchronous Interrupt Signalled
                      │
                      ▼
[ Hardware Pipeline Flush & Protection Transition ]
• Complete current instruction; flush pending instructions.
• Disable all maskable interrupts globally; switch CPU to Kernel Mode.
• Save the interrupted process's Program Counter (PC).
                      │
                      ▼
[ Interrupt Service Routine (ISR) Dispatch ]
• Look up entry point in vector table; branch to target handler.
• Set mask flags to adjust system priority.
• Re-enable maskable interrupts globally; save registers to kernel stack.
                      │
                      ▼
[ Device Payload Transfer ]
• Move data from hardware registers into kernel memory buffers.
                      │
                      ▼
[ Context Restoration & Return ]
• Restore general-purpose registers from kernel stack.
• Disable maskable interrupts globally; clear current interrupt state.
• Restore original system priority mask; execute RTI (Return from Interrupt).
                      │
                      ▼
[ Re-enter User Mode space; resume execution at saved PC ]
```

## Section 7: Thread Lifecycle Initialization (`ThreadFork` & `ThreadRoot`)

To spawn a new thread, the operating system executes an internal allocation and initialization routine called `ThreadFork()`.

### 7.1 The `ThreadFork` Routine
`ThreadFork()` takes three primary arguments: a pointer to the function routine (`fcnPtr`), a pointer to its argument array (`fcnArgPtr`), and the requested stack allocation size.

The allocation routine executes the following steps:
1. Validates the arguments in user space, traps into kernel mode, and performs the security checks again.
2. Allocates a new physical memory block to serve as the thread's execution stack and instantiates a new TCB.
3. Configures the saved register values within the TCB so that its stack pointer points to the newly allocated stack block.
4. Sets the TCB's saved program counter (`retpc`) to point to the address of the kernel assembly stub function `ThreadRoot`.
5. Places the target function address (`fcnPtr`) and argument address (`fcnArgPtr`) into the TCB fields corresponding to the system's argument registers (e.g., `r0` and `r1` or `a0` and `a1`).
6. Appends the newly initialized TCB to the scheduler's ready queue.

### 7.2 Bootstrapping via `ThreadRoot`
The new thread remains suspended on the ready queue until the scheduler invokes `run_new_thread()` and selects it. The assembly context switch loads the prepared values into the CPU, changing the stack pointer and branching directly into `ThreadRoot`.


```c
// Conceptual execution flow of the ThreadRoot bootstrap stub
ThreadRoot(fcnPtr, fcnArgPtr) {
    DoStartupHousekeeping(); // Record start times and log thread statistics
    UserModeSwitch();         // Flip the mode bit to enter standard User Mode
    
    // Execute the user's application function with its arguments
    Call fcnPtr(fcnArgPtr);   // The stack grows and shrinks dynamically below this frame
    
    ThreadFinish();           // Clean up resources and wake up any waiting threads
}
```

## Section 8: Core Challenges of Concurrency, Nondeterminism, & Atomic Operations

To build correct multi-threaded applications, systems programmers must understand the differences between concurrency and true physical parallelism.

### 8.1 Concurrency vs. Parallelism
* **Concurrency:** The logical overlap of multiple execution streams over time. On a single CPU core, threads run concurrently through time-slicing and interleaving. The scheduler is free to run instructions in any order, meaning programs must be designed to work correctly under any possible timeline.
* **Parallelism:** The physical execution of multiple instruction streams at the exact same instant, which requires multiple processing cores or processors.


### 8.2 Shared State Corruption: The ATM Bank Server Case Study
Consider an un-synchronized bank server designed to handle balance modifications across an automated teller network.

```c
// Vulnerable balance modification routine
void Deposit(acctId, amount) {
    acct = GetAccount(acctId);   // Fetches account details (triggers blocking disk I/O)
    acct->balance += amount;     // Critical Section
    StoreAccount(acct);          // Saves updated account details back to disk
}
```

#### The Assembly-Level Breakdown of Lost Updates
While `acct->balance += amount` looks like an atomic assignment in C, it compiles down to three independent assembly instructions:

```assembly
# Assembly sequence for a balance update
LOAD  r1, acct->balance  # 1. Read balance from memory into register r1
ADD   r1, amount         # 2. Add deposit value to register r1
STORE r1, acct->balance  # 3. Write updated value back to memory
```

If two threads attempt to deposit money into the same account simultaneously, a context switch can interleave these instructions and cause updates to be lost:

```
Initial State: acct->balance = 1000
Thread 1 (Deposits 100)               Thread 2 (Deposits 200)
┌──────────────────────────────────┐  ┌──────────────────────────────────┐
│ LOAD r1, balance (r1=1000)       │  │                                  │
│ ADD  r1, 100     (r1=1100)       │  │                                  │
│               [ --- Context Switch Intercepts Execution --- ]          │
│                                  │  │ LOAD r1, balance (r1=1000)       │
│                                  │  │ ADD  r1, 200     (r1=1200)       │
│                                  │  │ STORE balance, r1 (balance=1200) │
│               [ --- Context Switch Restores Thread 1 --- ]             │
│ STORE balance, r1 (balance=1100) │  │                                  │
└──────────────────────────────────┘  └──────────────────────────────────┘
Final State: acct->balance = 1100  (The 200 deposit is completely lost!)
```

### 8.3 Mathematical Interleaving Analysis Examples

#### Mathematical Interleaving Case Study 1
*Initial State: $y = 12$.*
* **Thread A instructions:** $I_{A1}: x = 1; \quad I_{A2}: x = y + 1;$
* **Thread B instructions:** $I_{B1}: y = 2; \quad I_{B2}: y = y \times 2;$

Depending on how the scheduler interleaves these instructions, the variable $x$ can result in three different final values:
* **Scenario A ($x = 13$):** Thread A runs to absolute completion before Thread B starts.
  $$x = y + 1 \implies x = 12 + 1 = \mathbf{13}$$
* **Scenario B ($x = 5$):** Thread B runs to absolute completion before Thread A executes its second instruction.
  $$y = 2 \implies y = 2 \times 2 = 4$$
  $$x = y + 1 \implies x = 4 + 1 = \mathbf{5}$$
* **Scenario C ($x = 3$):** Thread B executes its first instruction ($y = 2$), then Thread A preempts it and runs completely ($x = y + 1$), and finally Thread B finishes ($y = y \times 2$).
  $$x = 2 + 1 = \mathbf{3}$$

#### Mathematical Interleaving Case Study 2 (Word Aggregation Bounds)
*Initial State: Shared variable $x = 0$.*
* **Thread A:** `x = 1;`
* **Thread B:** `x = 2;`

On modern architectures with atomic word writes, the final value of $x$ will non-deterministically be either `1` or `2`. 

However, on older or non-standard architectures with non-atomic multi-word writes, a split write sequence could produce a garbled middle value. For example, if Thread A writes the upper bits and is interrupted by Thread B before writing the lower bits, the interleaved execution could result in an impossible final value like `3`.

### 8.4 Atomic Operations
To prevent these corruptions, concurrent programs must be built on top of **Atomic Operations**. An atomic operation is an indivisible unit of execution that always runs completely to success or behaves as if it never ran at all; it cannot be stopped mid-execution, and its intermediate state cannot be modified by other threads. 

On most modern processors, basic memory references and word assignments (standard loads and stores) are guaranteed to be atomic by the hardware, which prevents the garbled bit-interleaving example described above. However, complex instructions—such as double-precision floating-point operations or hardware-level array copies—are often not atomic, requiring developers to build synchronization structures using high-level **Lock APIs** (`acquire()` and `release()`).

* **Deadline:** Due Tuesday, February 17th. The course staff enforces a strict policy: **no extensions of any sort are granted for design documents**.
* **Design Review Format:** Teams must attend an upcoming high-level design review with their TA to defend their technical approach. Students do not need to present fully finalized code with every semicolon in place, but they must prove they thoroughly understand the complexities of their design.

#### Group Verification Checklist
1. **Algorithmic Clarity:** Teams must explicitly detail exactly what functions they intend to modify and outline the core algorithms they will use.
2. **Structural Mapping:** Explain how different system modules will link together and trace data flows across kernel layers.
3. **Shared Contribution:** Every single team member must fully understand their portion of the code. Individual understanding is verified through direct questioning during the review.
4. **Testing Strategy:** Teams must explain their test cases and prove they understand the testing framework, including identifying edge cases that are not covered by the baseline tests provided by the staff.
5. **Academic Honesty:** All work must be original. The course staff scans submissions and monitors repositories for any code copied from historic semesters or generated by AI tools.
