## Section 1: Mesa Monitor Architecture & Pintos Thread Layout Review

### 1.1 Algorithmic Framework of Mesa Monitor Programs
An operating system uses the **Monitor** abstraction as a primary paradigm to manage concurrent access to shared data structures. A Mesa monitor combines a mutual exclusion lock with one or more condition variables. 

The structural pattern of a Mesa monitor relies on synchronization logic where threads check state variables, wait if a resource is unavailable, and signal when a state change allows other threads to proceed.


The standard programmatic implementation of the Mesa monitor pattern is structured as follows :

```c
/* Standard Mesa Monitor Synchronization Design Pattern */

void Monitor_Wait_Routine() {
    acquire(&lock); // Establish mutual exclusion over monitor state variables 
    
    // Critical Mesa Semantic: The condition MUST be verified inside a while loop.
    // When a thread is woken up by signal(), it is simply placed on the ready queue.
    // Another thread could be scheduled first and change the state variables, 
    // making the condition false again by the time the waiter executes.
    while (need_to_wait) { 
        condvar.wait(&cond, &lock); // Atomically release lock and go to sleep 
        // Upon waking, the thread automatically re-acquires the lock before returning 
    }
    
    // Check and/or update state variables safely 
    UpdateStateVariables();
    
    release(&lock); // Relinquish monitor control 
}

void Monitor_Signal_Routine() {
    acquire(&lock); // Establish mutual exclusion 
    
    // Modify shared state variables to eliminate the wait condition 
    ModifyStateVariables();
    
    // Under Mesa semantics, the signaler keeps the lock and the processor 
    // The waiter is placed on the ready queue with no special priority 
    condvar.signal(&cond); // Wake up at most one waiter, if any exist 
    
    release(&lock); // Release monitor control 
}
```

### 1.2 Pintos / x86 Multi-Threaded Kernel Single Thread Process Layout
In the Pintos operating system running on x86 hardware, every active user process or thread is directly bound to an associated kernel thread. 


This execution context is implemented using a unified, fixed-size hardware structure:
* **The 4KB Allocation Boundary:** Each kernel thread is described by a single **4KB page memory object**. This single page co-locates both the Thread Control Block (TCB) and the kernel execution stack.
* **The Bottom Element (TCB):** The Thread Control Block sits at the lowest physical address of the 4KB page allocation. It tracks vital thread parameters, including:
  * `magic #`: A guard value positioned at the edge of the TCB struct to detect stack overflows. If the kernel stack pointer moves past its boundary, it overwrites this magic number, allowing the kernel to catch the violation before scheduling the thread.
  * `list`: Kernel list pointers used to insert the thread into scheduler queues (such as the `ready_list` or device wait lists).
  * `priority`: An integer value tracking the thread's scheduling precedence.
  * `stack`: A pointer tracking the top of the kernel stack when the thread is suspended.
  * `status`: The current execution lifecycle state (e.g., `RUNNING`, `READY`, `BLOCKED`).
  * `tid`: The unique Thread Identifier integer.
* **The Top Element (Kernel Stack):** The thread's kernel execution stack occupies the remainder of the 4KB page, growing downwards from the top of the page toward the TCB.
* **Registers:** The processor maps execution contexts using hardware registers, such as the Program Counter (`PC`) and Stack Pointer (`SP`).

## Section 2: x86 Privilege Escalation, Kernel Crossings, & Interrupt Processing

Operating systems isolate applications within user space and handle privileged hardware operations within the kernel using strict hardware boundaries.


### 2.1 Hardware Execution Modes
* **User Mode (Privilege Level 3):** Applications execute instructions with restricted access. The CPU uses user-space stacks (`ss:esp`) and maps virtual memory using the Page Table Base Register (`PTBR` or register `CR3`). Direct access to physical memory blocks or hardware control structures is blocked.
* **Kernel Mode (Privilege Level 0):** The CPU runs with unrestricted supervisor privileges. It operates on protected kernel stacks while the user thread's state is preserved, standing by until execution returns to user space.
* **Pure Kernel Threads:** These specialized execution contexts run entirely within kernel space at `PL: 0`. They execute using a small stack within their thread structure and have no corresponding user-mode component.

### 2.2 Privilege Transition Lifecycle (`User` $\rightarrow$ `Kernel`)
When a user application triggers a system call, trap, or hardware interrupt, the CPU executes a hardware-enforced transition sequence:

1. **Vector Indexing:** The interrupt or trap indexes into a system lookup table called the **Interrupt Vector Table** (also known as the Interrupt Descriptor Table or IDT). Each slot in this table maps to a specific exception type (such as vector `0x20` for the system timer).
2. **Stack Switching:** The x86 hardware reads the **Task-State Segment (TSS)** to find the kernel-space stack pointer (`ss:esp`) assigned to the incoming privilege level (`PL 0`). 
3. **Context Preservation:** The CPU pushes the suspended user-space stack registers (`old SS:ESP`) onto the newly activated kernel stack. It then pushes the user execution state, including the flags register and instruction pointer (`old EFLAGS`, `CS:EIP`), alongside any optional hardware error codes.
4. **Handler Branching:** The CPU sets the code segment and instruction pointer (`CS:EIP`) to point to the kernel target handler address defined in the IDT slot, shifting execution to kernel space.

### 2.3 Pintos Interrupt Vector Ingestion Routing (`stubs.S` & `interrupt.c`)
In Pintos, low-level trap and interrupt vectors are routed through a combination of assembly stubs and generic C dispatch handlers:

```assembly
# Conceptual low-level assembly layout inside stubs.S 

.global intr20_stub
intr20_stub:
    push $0x20          # 1. Push the unique hardware interrupt vector ID onto the stack 
    jmp intr_entry      # 2. Jump to the unified kernel entry point wrapper 

intr_entry:
    # 3. Save all general-purpose registers as an explicit execution frame 
    pusha 
    push %ds; push %es; push %fs; push %gs
    
    # 4. Initialize the protected kernel execution segment environment
    mov $0x10, %ax; mov %ax, %ds; mov %es, %ax
    
    # 5. Call the generic C handler, passing a pointer to the saved frame on the stack 
    pushl %esp
    call intr_handler   # Transfers execution to interrupt.c 
    
.global intr_exit
intr_exit:
    # 6. Restore the preserved register context and return from the interrupt 
    popa
    iret                # Executes iret instruction to return to user space 
```

Inside `interrupt.c`, the generic `intr_handler(struct intr_frame *frame)` function processes the incoming request:
1. **Classification & Triage:** It parses the frame vector ID, classifies the interrupt type, and acknowledges the hardware Interrupt Request (IRQ) line.
2. **Polymorphic Dispatch:** It forwards the frame to the registered callback handler. For a timer interrupt (`0x20`), control jumps directly to `timer_interrupt()` inside `timer.c`.
3. **Yield Verification:** On the return path, the handler checks if a context switch is required. If the active thread's time slice has expired, it invokes `thread_yield()` to switch threads.

## Section 3: Pintos Rescheduling Framework & Historical Context Switches

### 3.1 The Thread Rescheduling Mechanism
Pintos manages thread execution states through a structured sequence of function calls during a timer interrupt:

```
[ timer_interrupt() ] ──> Calls thread_tick() to update counters and check the quantum.
                                │
                                ▼
[ thread_yield() ] ───────> Moves current thread to READY and appends it to the ready list.
                                │
                                ▼
[ schedule() ] ───────────> Selects the next thread and invokes switch_threads().
                                │
                                ▼
[ switch_threads() ] ─────> Executes assembly context switch to swap stacks and registers.
```

* **`thread_tick(void)`:** Increments the active thread's runtime counters. If the current thread has exhausted its allocated time quantum, the function sets an internal `yield` flag to signal that preemption is required.
* **`thread_yield(void)`:** Executed on the return path from an interrupt handler if preemption was flagged. It changes the active thread's status back to `READY`, appends its TCB back onto the system `ready_list`, and invokes `schedule()` to select the next thread.
* **`schedule(void)`:** Runs the kernel's scheduling algorithm to select the next thread to run. It changes the selected thread's status to `RUNNING` and calls the low-level assembly routine `switch_threads` to swap execution contexts.
* **`switch_threads(struct thread *cur, struct thread *next)`:** A low-level assembly routine in `switch.S` that saves the active registers onto the current thread's kernel stack, updates the CPU's stack pointer register (`SP`) to point to the next thread's stack, and restores the new thread's saved register context.

### 3.2 Historical Perspective: Dennis Ritchie's Unix V6 Context Switch Context
The underlying complexity of low-level context switching is highlighted by a famous comment written by Dennis Ritchie in the source code of early **Unix V6 (`slp.c`)**:

```c
/*
 * If the new process paused because it was
 * swapped out, set the stack level to the last call
 * to savu(u.ssav).
 * This means that the return which is executed immediately
 * after the call to aretu
 * actually returns from the last routine which did
 * the savu.
 *
 * You are not expected to understand this.
 */
```
In early Unix systems, `savu` and `aretu` performed the historical equivalent of saving and restoring thread stack pointers during a context switch. 

Because the code manually manipulated activation frames to make the processor return to a completely different function context, the execution flow was incredibly counter-intuitive. Ritchie added the famous warning *"You are not expected to understand this"* as an ironic acknowledgement of how difficult it can be to reason about assembly-level stack manipulations.

## Section 5: Foundations of Scheduling & The CPU-I/O Burst Execution Model

### 5.1 Introduction to Scheduling
A core responsibility of an operating system is **Scheduling**: deciding which threads are given access to shared resources from moment to moment. While typically evaluated in terms of CPU time allocations, scheduling principles apply equally to other shared system resources, such as network bandwidth allocation or persistent disk access.


### 5.2 The CPU-I/O Burst Execution Model
The design of scheduling algorithms is driven by the physical reality that programs alternate between phases of computation and hardware waits:
* **CPU Burst:** A phase where the thread executes instructions sequentially on the processor core (such as loops, data updates, or memory calculations).
* **I/O Burst:** A phase where the thread blocks to wait for a hardware device operation to complete (such as reading a file from disk or waiting for a network packet).
* **The Scheduling Context:** A scheduling decision is made whenever the active thread finishes a CPU burst and blocks on an I/O request, freeing up the processor to run another thread. Under a time-slicing system, the operating system can also use timer interrupts to preempt a thread mid-burst, forcing it to yield the CPU before its computation finishes.
* **Temporal Locality and Distribution:** Real-world workloads are heavily **weighted toward small bursts**. Most threads execute brief computational phases before quickly blocking on an I/O operation or a user interaction. Long, continuous CPU bursts are rare and typically isolated to heavy numerical or background rendering tasks.

### 5.3 Core Optimization Criteria
An operating system balances several conflicting metrics when allocating CPU time:

1.  **Minimize Response Time:** The elapsed time required to complete a specific operation or job. This metric represents what the user experiences, such as the delay to echo a keystroke in a text editor or the time required to compile a program. Real-time systems face strict deadlines where tasks must finish within specific windows imposed by the physical world.
2.  **Maximize Throughput:** The total number of operations or jobs completed per unit of time. Throughput is closely related to response time but has different optimization requirements. For example, minimizing average response time often requires frequent context switching to keep systems responsive, which introduces performance overhead that reduces total throughput. Maximizing throughput requires minimizing context-switching overhead and keeping hardware resources highly utilized.
3.  **Fairness:** Ensuring that the CPU is shared among competing users or processes in an equitable way. Fairness often conflicts with average response time optimizations; a system can regularly achieve a lower average response time across all tasks by intentionally choosing to be less fair to long-running background jobs.

## Section 6: Classic Scheduling Policies: FCFS vs. Round-Robin

### 6.1 First-Come, First-Served (FCFS) / FIFO Scheduling
**First-Come, First-Served** is a non-preemptive scheduling policy where the thread that requests the CPU first retains control until it either blocks or runs to absolute completion.


#### Mathematical Trace Example 1 (Standard Arrival)
Consider three processes arriving in the system in the exact order $P_1, P_2, P_3$ at time index 0:
* $P_1$: Burst duration = 24 ms 
* $P_2$: Burst duration = 3 ms 
* $P_3$: Burst duration = 3 ms 

The resulting Gantt chart traces execution as follows:
```
Gantt Chart Layout:
├───────────────────────────────P1───────────────────────────────┼───P2───┼───P3───┤
0                                                               24       27       30
```
We evaluate the performance metrics for this schedule:
* **Waiting Times:** $P_1 = 0\text{ ms}$; $P_2 = 24\text{ ms}$; $P_3 = 27\text{ ms}$ 
* **Average Waiting Time:** $$\text{AWT} = \frac{0 + 24 + 27}{3} = \mathbf{17.0\text{ ms}}$$ 
* **Completion Times:** $P_1 = 24\text{ ms}$; $P_2 = 27\text{ ms}$; $P_3 = 30\text{ ms}$ 
* **Average Completion Time:**
    $$\text{ACT} = \frac{24 + 27 + 30}{3} = \mathbf{27.0\text{ ms}}$$ 

#### Mathematical Trace Example 2 (Altered Arrival Order)
Now consider the same three processes, but suppose they happen to arrive in the alternative order $P_2, P_3, P_1$ at time index 0. The Gantt chart shifts:
```
Gantt Chart Layout:
├───P2───┼───P3───┼───────────────────────────────P1───────────────────────────────┤
0        3        6                                                                30
```
Re-evaluating our performance metrics:
* **Waiting Times:** $P_1 = 6\text{ ms}$; $P_2 = 0\text{ ms}$; $P_3 = 3\text{ ms}$ 
* **Average Waiting Time:**
    $$\text{AWT} = \frac{6 + 0 + 3}{3} = \mathbf{3.0\text{ ms}}$$ 
* **Completion Times:** $P_2 = 3\text{ ms}$; $P_3 = 6\text{ ms}$; $P_1 = 30\text{ ms}$ 
* **Average Completion Time:**
    $$\text{ACT} = \frac{3 + 6 + 30}{3} = \mathbf{13.0\text{ ms}}$$ 

#### The Convoy Effect Fault
These examples demonstrate that FCFS performance varies wildly based on the exact order tasks are submitted. 

Because FCFS is non-preemptive, it suffers from a performance bottleneck known as the **Convoy Effect**: a long, CPU-bound process can monopolize the processor, forcing a long line of short, I/O-bound processes to sit completely idle in the ready queue. This behaves like a Safeway supermarket line where a customer buying a single carton of milk is trapped behind a large cart full of items, dragging down average response times across the entire system (though the upside is you get to read about Space Aliens in the tabloids while waiting!).

### 6.2 Round-Robin (RR) Preemptive Scheduling
To protect short jobs from the convoy effect, **Round-Robin** scheduling introduces preemption. Each thread is allocated a small, fixed window of CPU time called a **Time Quantum** (or time slice) $q$, typically ranging between 10 and 100 milliseconds. 

When a thread's time slice expires, a hardware timer interrupt intercepts execution, and the kernel moves the thread to the back of a First-In-First-Out ready queue to let the next thread run.

#### Invariant Allocation Guarantees
If there are $n$ active processes waiting in the ready queue with a configured time quantum $q$:
* Each process receives a strict, equitable $1/n$ share of the total CPU time.
* The CPU time is allocated in continuous chunks of at most $q$ time units.
* No individual process ever waits more than $(n - 1)q$ time units before its next execution slice.

#### Mathematical Trace Example 3 (Round Robin Processing)
Consider four processes arriving simultaneously at time 0, executed under a Round-Robin policy with a time quantum $q = 20\text{ ms}$:
* $P_1$: Burst duration = 53 ms 
* $P_2$: Burst duration = 8 ms 
* $P_3$: Burst duration = 68 ms 
* $P_4$: Burst duration = 24 ms 

The system steps through time slices, tracking the remaining burst requirements for each process:
* `Interval 0-20`: $P_1$ runs for $20\text{ ms}$ (requires $33\text{ ms}$ remaining).
* `Interval 20-28`: $P_2$ runs for $8\text{ ms}$ and terminates cleanly.
* `Interval 28-48`: $P_3$ runs for $20\text{ ms}$ (requires $48\text{ ms}$ remaining).
* `Interval 48-68`: $P_4$ runs for $20\text{ ms}$ (requires $4\text{ ms}$ remaining).
* `Interval 68-88`: $P_1$ returns to run for $20\text{ ms}$ (requires $13\text{ ms}$ remaining).
* `Interval 88-108`: $P_3$ returns to run for $20\text{ ms}$ (requires $28\text{ ms}$ remaining).
* `Interval 108-112`: $P_4$ returns to run for its final $4\text{ ms}$ and terminates.
* `Interval 112-125`: $P_1$ returns to run for its final $13\text{ ms}$ and terminates.
* `Interval 125-145`: $P_3$ runs for $20\text{ ms}$ (requires $8\text{ ms}$ remaining).
* `Interval 145-153`: $P_3$ runs for its final $8\text{ ms}$ and terminates.

```
Gantt Chart Layout:
├─P1─┼─P2─┼─P3─┼─P4─┼─P1─┼─P3─┼─P4─┼─P1─┼─P3─┼─P3─┤
0    20   28   48   68   88   108  112  125  145  153
```
We calculate the precise process waiting times by subtracting execution intervals from completion marks:
* **Waiting Time $P_1$:** $(68 - 20) + (112 - 88) = \mathbf{72\text{ ms}}$ 
* **Waiting Time $P_2$:** $(20 - 0) = \mathbf{20\text{ ms}}$ 
* **Waiting Time $P_3$:** $(28 - 0) + (88 - 48) + (125 - 108) = \mathbf{85\text{ ms}}$ 
* **Waiting Time $P_4$:** $(48 - 0) + (108 - 68) = \mathbf{88\text{ ms}}$ 
* **Average Waiting Time:**
    $$\text{AWT} = \frac{72 + 20 + 85 + 88}{4} = \mathbf{66.25\text{ ms}}$$ 
* **Completion Times:** $P_1 = 125\text{ ms}$; $P_2 = 28\text{ ms}$; $P_3 = 153\text{ ms}$; $P_4 = 112\text{ ms}$ 
* **Average Completion Time:**
    $$\text{ACT} = \frac{125 + 28 + 153 + 112}{4} = \mathbf{104.5\text{ ms}}$$ 

### 6.3 Choosing the Optimal Time Quantum
Selecting the correct time quantum requires carefully balancing application responsiveness against system overhead:
* **Quantum is configured too large ($q \rightarrow \infty$):** The Round-Robin policy degrades into standard non-preemptive First-Come, First-Served scheduling, making short tasks vulnerable to convoy effects again.
* **Quantum is configured too small ($q \rightarrow 0$):** Total system throughput drops sharply due to high overhead. Because the CPU spends a large percentage of its time executing context switches rather than running application instructions, the system bogs down in pure overhead.
* **The 1% Engineering Standard:** In modern production kernels, a typical time quantum is configured between **10ms and 100ms**. Given that hardware context-switching overhead typically requires between **0.1ms and 1.1ms**, this ratio ensures that context switches consume roughly **1% or less** of the total CPU capacity, keeping the system highly efficient.

## Section 8: Priority-Based Scheduling, Starvation, & Priority Inversion

Real-world operating systems use **Strict Priority Scheduling** to ensure that high-importance tasks are executed ahead of background processes.


### 8.1 Multi-Level Priority Queue Framework
The scheduler categorizes threads into distinct priority tiers. The system enforces a strict rule: **always execute the highest-priority runnable jobs to completion** before allocating CPU cycles to lower priority tiers. Inside each individual priority queue, threads are multiplexed using a Round-Robin policy with an allocated time quantum.

### 8.2 Primary Structural Failures
1.  **Starvation:** If the system faces a continuous stream of high-priority tasks, lower-priority threads will be completely starved of CPU cycles and may never get a chance to run.
2.  **Deadlock via Priority Inversion:** A severe synchronization failure that occurs when a low-priority thread acquires a shared mutual exclusion lock needed by a high-priority thread. Because the high-priority thread blocks waiting for the lock, it yields the CPU. However, if an intermediate-priority thread arrives that does not need the lock, the scheduler will run it ahead of the low-priority thread. As a result, the low-priority thread is blocked from executing and cannot release the lock, permanently stalling the high-priority task.

### 8.3 Fixes and Mitigation Strategies (Scheduling Fairness)
Strict fixed-priority scheduling between queues is inherently unfair. Long-running jobs may never get CPU time (an urban legend claims that when the Multics machine was shut down, a 10-year-old job was found still waiting in the queue). Therefore, systems must balance fairness against average response time.

To implement fairness and prevent starvation, systems use **Dynamic Priorities** and fractional CPU allocations:
* **Fractional CPU Allocation:** Systems can give each queue a specific fraction of the CPU. However, if one queue has a single long-running job and another has 100 short-running ones, this behaves like express lanes in a supermarket—sometimes express lanes get so long that you get better service by going to a standard line!
* **Aging Interactivity Heuristics:** Some operating systems monitor threads and dynamically boost the priority of jobs that have been starved of CPU time. 
* **The Limitations of Ad-Hoc Adjustments:** These adjustments can be difficult to tune. If the system becomes overloaded, every thread's priority eventually gets boosted, which means interactive jobs begin to suffer and cause responsive applications to stutter.
* **Priority Inheritance (Pintos Protocol):** To solve priority inversion, when a high-priority thread blocks on a lock held by a lower-priority thread, the lock holder temporarily inherits the high priority level of the blocked thread, allowing it to run, finish its critical section, and quickly release the lock.

## Section 9: Predictive Scheduling: SJF and SRTF

If an operating system could predict the future, it could optimize average response times using task duration metrics.

### 9.1 Core Policies
* **Shortest Job First (SJF):** A non-preemptive policy that selects the runnable thread with the shortest upcoming CPU burst duration. SJF is provably optimal for minimizing average waiting times among non-preemptive algorithms.
* **Shortest Remaining Time First (SRTF):** The preemptive variant of Shortest Job First. If a new job arrives with a shorter total time to completion than the remaining time on the currently running job, the scheduler immediately preempts the CPU and switches to the new job. SRTF is provably optimal among preemptive scheduling models.

### 9.2 Complex Three-Job Allocation Scenario
To see the benefits of SRTF, consider a system with three threads and a single shared storage disk:
* **Job A and Job B:** Completely CPU-bound tasks that require a full week of continuous calculation.
* **Job C:** A heavily I/O-bound task that alternates between 1 ms of CPU calculation and 9 ms of disk read operations.

#### Policy Comparison
* **Under FCFS:** If Job A or Job B captures the CPU first, it will run continuously for a week. Job C is trapped in the ready queue, forcing the physical disk to sit completely idle and dragging down system utilization.
* **Under Round-Robin (with a 100ms quantum):** Job A runs for 100ms, then Job B runs for 100ms. Job C finally gets the CPU, executes its 1ms of calculation, and blocks for 9ms to read the disk. Once the disk read completes, Job C must wait for the CPU-bound jobs to finish their 100ms time slices before it can run again. Total disk utilization drops to roughly **4.5%** ($\frac{9\text{ ms}}{201\text{ ms}}$), wasting disk bandwidth.
* **Under Round-Robin (with a 1ms quantum):** Interleaving threads at a 1ms interval improves disk utilization to **90%**, but it forces the system to execute thousands of context switches per second, creating heavy kernel overhead.
* **Under SRTF:** The scheduler recognizes that Job C has the shortest remaining CPU burst requirement (only 1 ms). It gives Job C the CPU immediately. Job C runs for 1ms and blocks on its 9ms disk read, freeing up the CPU. The scheduler then runs Job A or Job B. As soon as Job C's disk operation finishes, it is placed back on the ready queue; because its remaining burst is shorter than the background jobs, the scheduler immediately preempts the CPU-bound task to let Job C run. This pattern achieves **90% disk utilization** with minimal context-switching overhead.

### 9.3 SRTF Limitations and Yardstick Application
* **Starvation Risk:** SRTF can lead to severe starvation if there is a constant influx of many small jobs. Large jobs may never get to run.
* **Predicting the Future:** Implementing true SRTF requires predicting a job's future runtime. Some early systems asked the user to specify how long a job would take upon submission, killing the job if it exceeded its estimate to prevent cheating. However, it is generally hard to predict a job's runtime even for non-malicious users.
* **The SRTF Yardstick:** Because we cannot truly know how long a job will take, perfect SRTF is impossible to implement. However, because it is provably optimal for average response time, it is used as a theoretical **yardstick** to measure and compare the effectiveness of other scheduling policies.

> **SRTF Pros & Cons Summary:**
> * **Pros (+):** Provably optimal for average response time.
> * **Cons (-):** Inherently unfair to long jobs (starvation), and practically impossible to implement accurately since it requires predicting the future.

## Section 10: Adaptive Burst Forecasting: The Exponential Moving Average (EMA)

Because an operating system cannot truly know a job's future runtime, it must predict upcoming CPU bursts by observing past execution history (changing policy based on past behavior). This adaptive approach works because programs exhibit predictable behavior over time; a thread that was I/O bound in the past is highly likely to remain I/O bound in the near future (if computer behavior were truly random, historical prediction wouldn't help). 

This adaptive estimation principle is used everywhere in operating systems, including CPU scheduling, virtual memory management, and file systems.

While exponential averaging is the most common estimator, the estimation function ($f$) could theoretically be one of many different time-series estimation schemes, such as **Kalman filters**.

### 10.1 Mathematical Formulation
The kernel tracks a thread's historical runtimes and forecasts its next CPU burst length using an **Exponential Moving Average (EMA)** formula:

$$\tau_{n} = \alpha t_{n-1} + (1 - \alpha)\tau_{n-1}$$

Where:
* $t_{n-1}$ is the actual duration of the most recent physical CPU burst.
* $\tau_{n-1}$ is the historical predicted guess for that most recent burst.
* $\tau_{n}$ is the newly calculated prediction for the *next* upcoming CPU burst duration.
* $\alpha$ is a smoothing weight factor bounded strictly between 0 and 1 ($0 < \alpha \le 1$).

### 10.2 Expanding the Recurrence Relation
To understand how the smoothing factor weights past history, we can expand the recurrence relation by substituting past terms:

$$\tau_{n} = \alpha t_{n-1} + (1-\alpha)[\alpha t_{n-2} + (1-\alpha)\tau_{n-2}]$$
$$\tau_{n} = \alpha t_{n-1} + \alpha(1-\alpha)t_{n-2} + \alpha(1-\alpha)^2 t_{n-3} + \dots + (1-\alpha)^k \tau_{n-k}$$

Because $(1-\alpha)$ is less than 1, its exponent increases with each step backward in time, meaning older CPU bursts have an exponentially smaller impact on the current prediction. 
* Configuring $\alpha = 0.5$ balances recent history and long-term trends equally.
* Configuring $\alpha = 1.0$ makes the formula completely forget long-term history, baseing the prediction entirely on the single most recent burst duration ($t_{n-1}$).

### Question 2: Exponential Moving Average Prediction Analysis
A thread's scheduling history has an estimated burst duration $\tau_4 = 8\text{ ms}$. The smoothing weight factor is configured to $\alpha = 0.4$. The thread executes its next burst, and its actual physical runtime turns out to be $t_4 = 15\text{ ms}$.

Calculate the predicted burst duration estimate ($\tau_5$) for the thread's next upcoming execution phase.

#### Solution
We apply the Exponential Moving Average formula directly:

$$\tau_{n} = \alpha t_{n-1} + (1 - \alpha)\tau_{n-1}$$

Substituting our parameters into the equation ($n = 5$):

$$\tau_{5} = (0.4 \times t_{4}) + (1 - 0.4) \times \tau_{4}$$
$$\tau_{5} = (0.4 \times 15) + (0.6 \times 8)$$
$$\tau_{5} = 6.0 + 4.8 = \mathbf{10.8\text{ ms}}$$

The system adapts to the long burst by adjusting its next prediction upward to **10.8 ms**.
