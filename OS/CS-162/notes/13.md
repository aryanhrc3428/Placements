## Section 1: Multi-Level Feedback Queue Review & Shifting Scheduling Horizons

### 1.1 Algorithmic Framework of Multi-Level Feedback Queues (MLFQ)
The **Multi-Level Feedback Queue (MLFQ)** is an adaptive scheduling framework designed to analyze an application's past execution history to predict its upcoming CPU burst requirements. First implemented in the Compatible Time-Sharing System (CTSS), MLFQ addresses a core systems problem: optimizing performance without forcing applications to pre-declare their compute parameters.


#### Core Structural Invariants
* **Priority-Tiered Layout:** The scheduler maintains multiple distinct ready queues, where each queue is assigned a unique priority level. Queues near the top handle high-priority foreground workloads, while queues at the bottom process low-priority background workloads.
* **Divergent Scheduling Disciplines:** Each individual queue can run a different scheduling policy. For instance, high-priority foreground queues typically use a preemptive **Round-Robin (RR)** policy to ensure responsiveness, while the lowest-priority background queue uses a non-preemptive **First-Come, First-Served (FCFS)** policy to maximize raw instruction throughput.
* **Exponential Quantum Scaling:** To prevent long-running tasks from causing frequent context switches, the time quantum increases exponentially at lower priority levels (e.g., top queue $Q = 8\text{ ms}$, middle queue $Q = 16\text{ ms}$, cascading down to an un-quantized FCFS baseline).
* **Dynamic Priority Adjustments:** 1. A newly created or awakened thread is placed at the back of the highest-priority queue.
  2. If the thread consumes its entire allocated time quantum without blocking for I/O, the scheduler assumes it is a compute-bound task and demotes it down one level to a lower priority queue.
  3. If the thread blocks for I/O or yields the CPU voluntarily before its time slice expires, the scheduler identifies it as an interactive or I/O-bound task, retaining or promoting its position near the top of the priority tiers.

## Section 2: Proportional-Share Paradigms: Lottery & Stride Scheduling

### 2.1 Philosophy of Proportional Sharing
Traditional priority-based scheduling algorithms always favor high-priority jobs, which means lower-priority background tasks can be starved of CPU cycles indefinitely if high-priority tasks continue to arrive. However, in many systems, priority is simply a tool to keep the system responsive rather than an absolute rule. 

**Proportional-Share Scheduling** replaces strict preemption hierarchies with a resource-sharing model. Instead of completely blocking lower-priority tasks, the scheduler guarantees each job a specific, predictable percentage of total CPU cycles based on its assigned weight. Because every active job is guaranteed a non-zero portion of the CPU, starvation is structurally eliminated, ensuring that all tasks make forward progress. This model is closely related to the principle of **Fair Queueing** used in computer networking.

---

### 2.2 Lottery Scheduling Mechanics
**Lottery Scheduling** is a randomized approach to proportional-share scheduling. The operating system represents CPU allocation fractions as virtual **Lottery Tickets**. 


#### Core Selection Logic
* Every active job is assigned a specific number of lottery tickets based on its importance or weight.
* At each new time slice interval, the scheduler generates a random number within the range of total outstanding tickets.
* The task that holds the winning ticket number is awarded the CPU core for that time slice.
* **Statistical Convergence:** While any individual draw is random, over time, the total CPU allocation converges mathematically so that each job's runtime is directly proportional to its number of tickets.

#### Approximating Shortest Remaining Time First (SRTF)
To maximize responsiveness without running into starvation issues, the system can adjust ticket counts dynamically based on execution characteristics:
* Short, interactive, or I/O-bound jobs are awarded a large pool of tickets, which gives them a high probability of gaining the CPU quickly and keeps the system responsive.
* Long-running, compute-bound jobs are given a smaller pool of tickets. They run less frequently but are still guaranteed to make progress since they retain at least one ticket.
* **Graceful Scaling under Load:** Unlike strict priority systems, adding or deleting jobs in a lottery scheduler scales resource allocation gracefully and proportionally across all active tasks, entirely independent of the total number of outstanding tickets in the system.

---

### 2.3 Mathematical Allocation Analysis under Variable Workload Mixes
To calculate a task's expected CPU share, we use the following probability formula:
$$\text{Expected CPU Share } U_i = \frac{N_i}{\sum_{j=1}^{M} N_j}$$

Where $N_i$ represents the number of tickets allocated to target job $i$, and $M$ tracks the total number of active threads in the ready queue.

Suppose the system uses a configuration where short, interactive tasks are assigned **10 tickets** and long, compute-bound tasks are assigned **1 ticket**. The table below analyzes the expected CPU allocation across different workload combinations:

| Active Job Mix Layout <br> (`# Short / # Long`)  | Expected CPU Share <br> (Per Short Job)  | Expected CPU Share <br> (Per Long Job)  | Derivation and Probability Formulas |
| :---: | :---: | :---: | :--- |
| **$1 \text{ / } 1$**  | **$91\%$**  | **$9\%$**  | $\text{Total} = 10 + 1 = 11 \text{ tickets}$. <br> $\text{Short Share} = \frac{10}{11} \approx 90.9\%$; $\text{Long Share} = \frac{1}{11} \approx 9.09\%$. |
| **$0 \text{ / } 2$**  | **N/A**  | **$50\%$**  | $\text{Total} = 1 + 1 = 2 \text{ tickets}$. <br> $\text{Long Share} = \frac{1}{2} = 50\%$ per process. |
| **$2 \text{ / } 0$**  | **$50\%$**  | **N/A**  | $\text{Total} = 10 + 10 = 20 \text{ tickets}$. <br> $\text{Short Share} = \frac{10}{20} = 50\%$ per process. |
| **$10 \text{ / } 1$**  | **$9.9\%$**  | **$0.99\%$**  | $\text{Total} = (10 \times 10) + 1 = 101 \text{ tickets}$. <br> $\text{Short Share} = \frac{10}{101} \approx 9.9\%$; $\text{Long Share} = \frac{1}{101} \approx 0.99\%$. |
| **$1 \text{ / } 10$**  | **$50\%$**  | **$5\%$**  | $\text{Total} = 10 + (10 \times 1) = 20 \text{ tickets}$. <br> $\text{Short Share} = \frac{10}{20} = 50\%$; $\text{Long Share} = \frac{1}{20} = 5\%$. |

#### Handling System Overload Bounds
If the system faces extreme overload (for example, a load average of 100 or more concurrent short tasks), individual time allocations drop significantly, making it difficult for any single application to make reasonable forward progress. To protect system stability in this scenario, administrative policies must intervene to drop the active load, such as force-terminating non-critical background jobs or logging off inactive users.

### 2.4 Code Example: Lottery Ticket Selection Loop
This code shows how a lottery scheduler selects the next winning task by traversing an internal array of active threads.

```c
#include <stdio.h>
#include <stdlib.h>

typedef struct thread {
    char name[16];
    int ticket_count;
} thread_t;

// Selects the winning thread by accumulating tickets sequentially 
thread_t* schedule_lottery_winner(thread_t* active_threads, int dynamic_task_count) {
    int total_system_tickets = 0; // 
    
    // Accumulate the sum of all outstanding tickets across active tasks 
    for (int i = 0; i < dynamic_task_count; i++) {
        total_system_tickets += active_threads[i].ticket_count; // 
    }
    
    if (total_system_tickets == 0) return NULL;

    // Pick a random "dart" value d within the range [1, total_system_tickets] 
    int dart = (rand() % total_system_tickets) + 1; // 
    int running_ticket_sum = 0;
    
    // Step through the tasks sequentially; select the first task that pushes the sum past the dart 
    for (int j = 0; j < dynamic_task_count; j++) {
        running_ticket_sum += active_threads[j].ticket_count; // 
        if (running_ticket_sum >= dart) { // 
            return &active_threads[j]; // Return pointer to the winning thread 
        }
    }
    return &active_threads[dynamic_task_count - 1];
}
```

### 2.5 Stride Scheduling: Non-Random Proportional Sharing
While lottery scheduling is simple, it suffers from the **Law of Small Numbers**: over short time frames, the actual CPU allocation can deviate significantly from the intended proportions, leading to short-term unfairness. To eliminate this randomness while maintaining proportional allocation, systems developers use **Stride Scheduling**.

```
                          [ STRIDE SCHEDULING FORMULATION ]
                                          │
       Compute Stride length for each active task: Stride = Big# W / Tickets
                                          │
       • High-ticket tasks receive a small stride.
       • Low-ticket tasks receive a large stride.
                                          │
                                          ▼
     [ Selection Step: Choose the active task with the lowest Pass Counter ]
                                          │
       • Run selected task for a single time quantum.
       • Update tracking register: Pass_Counter = Pass_Counter + Stride.
```

#### Step-by-Step Mathematical Walkthrough
The scheduler calculates an execution **Stride** for each job, which is inversely proportional to its ticket allocation:
$$\text{Stride } S_i = \frac{\text{Large Constant } W}{N_i}$$

Where $N_i$ represents the number of tickets assigned to job $i$. Every job tracks its execution progress using a local register called a **Pass Counter**. At each scheduling interval, the scheduler selects the job with the lowest pass counter, runs it for a single time slice, and increments its pass counter by its stride value. This mechanism ensures that jobs with smaller strides (more tickets) are selected more frequently, matching their target allocations exactly over short time frames.

Consider a system with a constant $W = 10,000$ and three active jobs with the following ticket allocations:
* Job A: $100\text{ tickets} \implies \text{Stride } S_A = \frac{10,000}{100} = \mathbf{100}$ 
* Job B: $50\text{ tickets} \implies \text{Stride } S_B = \frac{10,000}{50} = \mathbf{200}$ 
* Job C: $250\text{ tickets} \implies \text{Stride } S_C = \frac{10,000}{250} = \mathbf{40}$ 

#### Chronological Scheduling Execution Matrix
Assuming all pass counters are initially zero, the table below traces the first 6 scheduling intervals:

| Step Allocation | Job Selected <br> (Lowest Pass)  | Pass Counter A <br> (Before $\rightarrow$ After) | Pass Counter B <br> (Before $\rightarrow$ After) | Pass Counter C <br> (Before $\rightarrow$ After) | Invariant Status Summary |
| :---: | :--- | :---: | :---: | :---: | :--- |
| **Initial** | — | $0$ | $0$ | $0$ | All pass counters tied at 0. |
| **Step 1** | **Job A** *(Tied-break)* | $0 \rightarrow \mathbf{100}$ | $0$ | $0$ | Job A runs; its counter increases by its stride ($+100$). |
| **Step 2** | **Job B** *(Tied-break)* | $100$ | $0 \rightarrow \mathbf{200}$ | $0$ | Job B runs; its counter increases by its stride ($+200$). |
| **Step 3** | **Job C** | $100$ | $200$ | $0 \rightarrow \mathbf{40}$ | Job C runs; its counter increases by its stride ($+40$). |
| **Step 4** | **Job C** | $100$ | $200$ | $40 \rightarrow \mathbf{80}$ | Job C still has the lowest counter ($80 < 100$); it runs again. |
| **Step 5** | **Job C** | $100$ | $200$ | $80 \rightarrow \mathbf{120}$ | Job C runs for a third time ($120$). |
| **Step 6** | **Job A** | $100 \rightarrow \mathbf{200}$ | $200$ | $120$ | Job A has the lowest counter ($100 < 120$); it is selected. |

This deterministic execution flow ensures that Job C runs exactly twice as often as Job A, and Job A runs exactly twice as often as Job B, matching their ticket ratios ($250 : 100 : 50 \implies 5 : 2 : 1$) without any of the statistical variance seen in lottery scheduling.

## Section 3: The Linux Completely Fair Scheduler (CFS) Deep Dive

### 3.1 Core Philosophy and the Virtualization Model
The Linux **Completely Fair Scheduler (CFS)** is an elegant, multi-tier implementation of proportional-share scheduling used in modern production kernels. 

```
Idealized Multi-Threading Model (Simultaneous Execution):
Core: [ Thread 1 (1/N) ][ Thread 2 (1/N) ][ Thread 3 (1/N) ] ──> Simultaneous continuous cycles

Hardware Multi-Threading Model (Time-Slices):
Core: [  Thread 1 (Q)  ][  Thread 2 (Q)  ][  Thread 3 (Q)  ] ──> Discrete interleaved chunks
```
*The fluid conceptual multi-threading model vs. discrete hardware execution.*

CFS is built around a specific ideal model: a **perfectly subdivided hardware processor** where $N$ active threads run simultaneously on the same core, with each thread receiving an exact, continuous $\frac{1}{N}$ fraction of total CPU instruction cycles. 

Since real, commodity hardware cannot execute an arbitrary number of threads simultaneously in infinitely small subdivisions, the operating system must allocate the full CPU to a single thread for discrete time slices. The core goal of CFS is to minimize execution variance, tracking how much time each thread has run to keep them roughly in sync with the idealized model.

---

### 3.2 CFS Responsiveness Boundaries: Target Latency & Minimum Granularity
To maintain predictability and prevent starvation, CFS manages context switches using two key bounds:

#### 1. Target Latency (`sched_latency`)
The target latency is the fixed window of time over which **every single active runnable process must be scheduled at least once**. Under standard equal sharing with $n$ active processes, the base time quantum $Q_i$ allocated to a task is calculated as:
$$Q_i = \frac{\text{Target\_Latency}}{n}$$

* Consider a system with a target latency of $20\text{ ms}$ running $4$ active processes. Each process is awarded an equal time slice of $\frac{20\text{ ms}}{4} = \mathbf{5\text{ ms}}$.
* If the workload surges to $200$ active processes, a naive application of the formula drops each individual time slice to $\frac{20\text{ ms}}{200} = \mathbf{0.1\text{ ms}}$. This micro-slice causes severe performance issues, as the system spends almost all its time executing context switches rather than running application instructions.

#### 2. Minimum Granularity (`sched_min_granularity`)
To protect system throughput from context-switching overhead, CFS enforces a lower bound called the **Minimum Granularity**. This parameter defines the absolute shortest time slice any individual thread can be assigned during a scheduling cycle.

* If a system has a target latency of $20\text{ ms}$ and a minimum granularity floor of $1\text{ ms}$:
* When handling $200$ concurrent processes, CFS drops the target latency constraint and overrides the time slice calculation, awarding each process the minimum floor of **$1\text{ ms}$**. This pushes the total scheduling period out to $200\text{ ms}$ ($200 \times 1\text{ ms}$), protecting system throughput at the cost of longer response times under heavy load.

---

### 3.3 Nice Values, Weights, and the `vruntime` Transformation
To support proportional-share allocation, CFS maps traditional UNIX **nice values** (which range from -20 for high priority to 19 for low priority) to an exponential scale of internal weights:

$$\text{Weight } W_i = \frac{1024}{(1.25)^{\text{nice}}}$$

This exponential mapping ensures that a relative difference of 5 nice points always translates to a fixed, predictable ratio of CPU cycles, regardless of the absolute nice values chosen. Because $(1.25)^5 \approx 3$, two tasks whose nice values differ by 5 will always maintain a **$3:1$ ratio of weights and CPU cycles**.

Using these weights, the scheduler shifts from tracking raw physical CPU runtime to tracking a normalized metric called **Virtual Runtime (`vruntime`)**:

$$\text{vruntime}_i = \text{Real Physical CPU Execution Time} \times \frac{\text{Baseline Weight } (1024)}{W_i}$$

#### Operational Implications of the Formula
* **High-Priority Tasks (Low Nice Value / Large Weight):** For a high-priority task, $W_i > 1024$, which means its virtual runtime accumulates **slower** than real physical time.
* **Low-Priority Tasks (High Nice Value / Small Weight):** For a low-priority task, $W_i < 1024$, which means its virtual runtime accumulates **faster** than real physical time.
* **Selection Policy:** The scheduler always selects the thread with the **absolute lowest virtual runtime** to run next. This automatically rewards high-priority tasks with longer and more frequent physical time slices while keeping all tasks synchronized over time.

### 3.4 C++ Simulation: CFS Allocation Loop
This simulation demonstrates how CFS calculates time slices and virtual runtime accumulation for a group of competing threads.

```cpp
#include <iostream>
#include <vector>
#include <cmath>
#include <string>

struct CFSTask {
    std::string name;
    int nice_value;
    double weight;
    double physical_cpu_time;
    double vruntime;
};

void calculate_cfs_quantums(std::vector<CFSTask>& tasks, double target_latency, double min_granularity) {
    double total_system_weight = 0.0;
    for (const auto& task : tasks) {
        total_system_weight += task.weight;
    }

    std::cout << "--- CFS Scheduling Period Allocation Loop ---\n";
    for (auto& task : tasks) {
        // Compute proportional time slice quantum based on relative task weight 
        double quantum = target_latency * (task.weight / total_system_weight); // 
        
        // Enforce the minimum granularity floor to protect system throughput 
        if (quantum < min_granularity) {
            quantum = min_granularity; // 
        }
        
        // Simulate execution: add quantum to raw physical time
        task.physical_cpu_time += quantum;
        
        // Update virtual runtime: vruntime = real_time * (baseline_weight / task_weight) 
        task.vruntime += quantum * (1024.0 / task.weight); // 
        
        std::cout << "Task: " << task.name 
                  << " | Nice: " << task.nice_value 
                  << " | Allocated Quantum: " << quantum << " ms"
                  << " | New vruntime: " << task.vruntime << " ms\n";
    }
}

int main() {
    std::vector<CFSTask> tasks = {
        // Weight = 1024 / (1.25)^nice 
        {"Thread_A", 0,  1024.0, 0.0, 0.0},                  // Nice 0 baseline 
        {"Thread_B", 5,  1024.0 / std::pow(1.25, 5), 0.0, 0.0} // Nice 5 (~3x less weight) 
    };
    
    calculate_cfs_quantums(tasks, 20.0, 1.0); // Target Latency = 20ms, Min Granularity = 1ms 
    return 0;
}
```

### 3.5 Caching and Lookups via Red-Black Trees
To track and select tasks efficiently, CFS stores all runnable threads in a self-balancing **Red-Black Tree**, sorted strictly by their virtual runtimes (`vruntime`).


* **Algorithmic Complexity:** Inserting a thread or removing it from the tree requires balanced path traversals that run in logarithmic time, **$O(\log N)$**, where $N$ is the number of active threads.
* **Leftmost Node Caching:** Because the scheduler needs to access the task with the absolute lowest `vruntime` at every interval, modern implementations optimize lookups by **caching a pointer directly to the leftmost leaf node**. This allows the scheduler to select the next thread to run in instant constant time, **$O(1)$**, avoiding the need to traverse the tree structure on every context switch.
* **Automatic Interactivity Credits:** When an interactive thread blocks for I/O, it is removed from the tree, and its `vruntime` stops accumulating. While it sleeps, the `vruntime` values of active, compute-bound threads continue to increase. When the interactive thread wakes up, its historical `vruntime` is significantly lower than its competitors. As a result, it is immediately placed at the far left of the Red-Black tree, allowing it to preempt the compute-bound tasks and run immediately. This elegant design handles interactive tasks automatically without requiring complex heuristics.

## Section 4: Real-Time Scheduling Systems & Workload Models

### 4.1 Philosophy of Real-Time Systems
In conventional operating systems, the scheduler optimizes for system metrics like maximum throughput or average response times. 

In a **Real-Time System (RTS)**, the primary goal is **Predictability of Performance**. The system must guarantee with complete confidence that critical operations will always complete within strict, worst-case timing deadlines imposed by the physical world. Real-time scheduling is not about raw processing speed; it is about enforcing absolute timing predictability.

* **Hard Real-Time Systems:** Used for mission-critical, safety-oriented systems (such as medical pacemakers, automotive brake controllers, or automated flight systems). Missing a single deadline is treated as a fatal system failure. The system must prove *a priori* (before deployment) that all tasks will meet their deadlines under worst-case scenarios.
* **Soft Real-Time Systems:** Used for applications like live multimedia streaming or video playback. Missing a deadline drops a frame but does not cause a fatal failure; the system simply aims to meet deadlines with high probability.

---

### 4.2 Real-Time Workload Formulations
A real-time workload model represents tasks as a series of repeating execution requirements defined by specific constraints:


* Tasks are assumed to be independent, fully preemptable, and can arrive at arbitrary release times.
* Each task $i$ is defined by a tuple: **$(P_i, C_i)$**, where **$P_i$** represents its recurring **Period** window, and **$C_i$** specifies its maximum **Worst-Case Computation Time**.
* **Periodic Arrivals:** Tasks arrive cyclically at fixed intervals: $A_i^{t+1} = A_i^t + P_i$.
* **Relative Deadlines:** Each task must finish its computation burst before the end of its current period: $D_i^t = A_i^t + P_i$.

---

### 4.3 Why Conventional Scheduling Fails Real-Time Deadlines
Traditional scheduling policies like Round-Robin are completely unsuited for real-time systems because they make scheduling choices without considering task deadlines. 

If multiple real-time tasks are multiplexed using a uniform time quantum, a task with a long period can easily starve a short-period task with an immediate deadline, causing the system to fail. To handle real-time workloads safely, scheduling policies must factor deadlines directly into their selection logic.

## Section 5: Hard Real-Time Policies: Rate Monotonic & Earliest Deadline First

### 5.1 Rate Monotonic Scheduling (RMS)
**Rate Monotonic Scheduling (RMS)** is a foundational static priority scheduling policy for periodic real-time workloads.

#### Algorithmic Assignment Invariant
* Priorities are assigned statically based on the length of each task's period: **the shorter the period, the higher its scheduling priority**.
* Mathematically: $P_i < P_j \implies \pi_i > \pi_j$, where $\pi$ represents priority.
* At any scheduling interval, the scheduler always executes the active task with the highest priority.

#### The Liu & Layland Schedulability Bound Theorem
To determine if a set of periodic tasks can be scheduled safely under RMS without missing deadlines, we evaluate the **Liu & Layland Utilization Bound Test**. The processor utilization $U$ of a task set is the sum of the ratios of computation time to period for all tasks:

$$U = \sum_{i=1}^{n} \left( \frac{C_i}{P_i} \right)$$

A set of $n$ periodic tasks is guaranteed to be fully schedulable under RMS if its total processor utilization satisfies the following inequality:

$$\sum_{i=1}^{n} \left( \frac{C_i}{P_i} \right) \le n \left( 2^{1/n} - 1 \right)$$

* **Sufficient but Not Necessary:** This utilization test is *sufficient* but not *necessary*. If a task set passes the test, it is guaranteed to meet all deadlines. If it fails, it might still be schedulable, depending on the exact alignment of task periods. For example, if all periods are perfectly harmonic (meaning each period evenly divides the larger ones), the utilization bound scales up to a full **100%**.
* **Asymptotic Limit:** As the number of independent tasks grows toward infinity, the utilization bound converges to a fixed mathematical limit:
  $$\lim_{n \rightarrow \infty} n \left( 2^{1/n} - 1 \right) = \ln 2 \approx \mathbf{69.3\%}$$

---

### 5.2 RMS Execution Failure Case Study
Consider a real-time system running three periodic tasks with parameters $T_i = (P_i, C_i)$:
* Task 1: $T_1 = (4, 1) \implies U_1 = \frac{1}{4} = \mathbf{0.25}$ 
* Task 2: $T_2 = (5, 2) \implies U_2 = \frac{2}{5} = \mathbf{0.40}$ 
* Task 3: $T_3 = (7, 2) \implies U_3 = \frac{2}{7} \approx \mathbf{0.286}$ 

#### Load Verification
We sum the individual task utilizations to check the total processor load:
$$U_{\text{Total}} = 0.25 + 0.40 + 0.286 = \mathbf{0.936}$$
Because $U_{\text{Total}} \le 1.0$, the workload does not physically overload the processor, meaning a valid schedule is theoretically possible.

#### Applying the Schedulability Test
Next, we evaluate the Liu & Layland bound for $n = 3$ tasks:
$$\text{Bound} = 3 \times \left( 2^{1/3} - 1 \right) \approx 3 \times (1.2599 - 1) \approx \mathbf{0.780}$$

Since $0.936 > 0.780$, the task set fails the sufficient utilization test. This tells us that under a static priority RMS policy, deadlines are likely to be missed.

#### Step-by-Step Chronological Execution Trace
Under RMS, priorities are assigned statically based on period lengths ($P_1=4 < P_2=5 < P_3=7$), resulting in the fixed priority hierarchy: **$\pi_1 > \pi_2 > \pi_3$**.

```
RMS Failure Interleaving Timeline:
├─T1─┼───T2───┼──T3──┼─T1─┼─T2─┼──T3──┼─T1─┤
0    1        3      5    6    8      10   11
                     ▲         ▲      ▲
                     │         │      └─ T1 arrives; hits deadline block!
                     │         └─ T2 deadline missed at T=5!
                     └─ T1 deadline missed at T=4!
```
*Chronological trace of the static priority execution failure.*

* **Interval `0 to 1`:** $T_1$ has the highest priority and runs for $1\text{ ms}$, completing its computation cleanly.
* **Interval `1 to 3`:** With $T_1$ finished, $T_2$ runs for its required $2\text{ ms}$ burst, completing at $3\text{ ms}$.
* **Interval `3 to 4`:** $T_3$ runs for $1\text{ ms}$. At time $4\text{ ms}$, the period for $T_1$ expires, prompting it to arrive again and immediately preempt $T_3$ due to its higher priority.
* **Interval `4 to 5`:** $T_1$ runs for $1\text{ ms}$. At time $5\text{ ms}$, the period for $T_2$ expires, causing it to arrive again.
* **Interval `5 to 7`:** $T_2$ runs for its $2\text{ ms}$ burst, completing at $7\text{ ms}$.
* **Interval `7 to 8`:** At time $7\text{ ms}$, **Task 3 ($T_3$) hits its relative deadline!** However, it has only executed for $1\text{ ms}$ out of its required $2\text{ ms}$ computation burst. 
* **The Failure:** Because static priorities forced $T_3$ to be preempted by arriving higher-priority tasks, it misses its deadline at $7\text{ ms}$, causing a hard real-time system failure.

### 5.3 Earliest Deadline First (EDF) Scheduling
To avoid the utilization limitations of static priority policies, real-time systems use **Earliest Deadline First (EDF)**. EDF is a dynamic priority scheduling policy where priorities are updated on the fly based on changing task deadlines.

#### Algorithmic Selection Invariant
At any scheduling interval, the scheduler evaluates all active tasks and allocates the CPU to the task with the **closest absolute deadline** ($D_i^t$).

#### The Schedulability Bound Theorem
A set of periodic real-time tasks is guaranteed to meet all deadlines under an EDF policy if and only if its total processor utilization satisfies the following constraint:

$$\sum_{i=1}^{n} \left( \frac{C_i}{P_i} \right) \le \mathbf{1.0}$$

This is an **exact schedulability test** (both *necessary* and *sufficient*). As long as the workload does not physically overload the processor, EDF is mathematically guaranteed to find a valid schedule that meets every deadline.

Applying this to our previous counter-example workload ($U_{\text{Total}} = 0.936 \le 1.0$), EDF successfully schedules the tasks without missing any deadlines.

---

### 5.4 The Domino Effect Performance Anomaly
While EDF is optimal for underloaded systems ($U \le 1.0$), it exhibits a critical performance bottleneck under short-term overloads ($U > 1.0$) known as the **Domino Effect**.


If an unexpected hardware spike or an extra task ($T_{\text{new}}$) pushes the total processor utilization past 1.0, the closest-deadline task will consume extra cycles, causing it to miss its deadline slightly. 

Because EDF always prioritizes the task whose deadline is closest, the scheduler will cascade this delay down the line. It switches to the next closest task, which now misses its deadline as well, triggering a chain reaction where **every subsequent task in the queue misses its deadline sequentially**. A minor overload can thus cause a complete collapse of deadline compliance across the entire system.

## Section 6: Methodologies for Scheduling Evaluation & System Load Curves

### 6.1 Evaluation Frameworks
Systems engineers use three complementary techniques to evaluate and validate scheduling algorithms:


1. **Deterministic Modeling:** Takes a fixed, predetermined workload profile and calculates the exact performance metrics for each policy. While simple and useful for core validation, it only applies to that specific workload profile.
2. **Queueing Models:** Uses stochastic probability distributions (such as Poisson arrival rates and exponential service times) to analyze system behavior mathematically. This provides useful long-term averages but can struggle with complex, non-linear real-world workloads.
3. **Trace-Driven Implementation and Simulation:** Records actual execution logs from production servers onto a "trace tape" and replays them inside a simulator to test different scheduling algorithms. This provides a flexible and realistic environment to evaluate policies against real-world data before deploying them to production kernels.

---

### 6.2 Strategic Selection Synthesis Table
The optimal scheduling policy depends entirely on the primary performance goals of the system:

| Primary Optimization Core Target  | Recommended Scheduling Policy Selection  | Architectural Justification and Mechanics |
| :--- | :---: | :--- |
| **Maximize Raw CPU Throughput**  | **FCFS**  | Eliminates all preemption and context-switching overhead, keeping the CPU focused entirely on application code. |
| **Minimize Average Response Time**  | **SRTF Approximation**  | Provably optimal framework that gets short jobs out of the system quickly. |
| **Maximize I/O Device Throughput**  | **SRTF Approximation**  | Prioritizes interactive tasks, allowing them to issue I/O requests quickly and keep peripheral devices fully utilized. |
| **Enforce Strict CPU Time Fairness**  | **Linux CFS**  | Uses virtual runtime tracking (`vruntime`) to keep all executing threads synchronized with a perfectly subdivided processor model. |
| **Enforce Fair Ready-Queue Wait Times**  | **Round-Robin**  | Uses fixed time slices to cycle through the ready queue sequentially, ensuring no thread is left waiting for long. |
| **Guarantee Real-Time Deadlines**  | **EDF**  | Dynamic priority policy that scales up to 100% utilization, always prioritizing the task closest to its deadline. |
| **Favor High-Importance System Tasks**  | **Strict Priority**  | Enforces a rigid execution hierarchy, guaranteeing that high-priority system tasks always run ahead of background processes. |

---

### 6.3 The Non-Linear System Load Curve
A common misconception in systems engineering is that a hardware resource (such as a CPU, network link, or storage interface) should be driven to 100% utilization to maximize efficiency. 


In reality, response times follow a highly non-linear load curve : as resource utilization approaches **100%**, response times escape toward **infinity**. This occurs because as the system fills up, any minor variation in task arrivals causes long queues to build up faster than they can drain. 

Most scheduling algorithms work fine within the stable, linear portion of the curve, but performance degrades rapidly once load passes the **"Knee" of the curve** (typically around 70% to 80% utilization). When a system hits this inflection point, the correct solution is not to tune the scheduling policy, but to expand the physical hardware capacity to return the system to its stable operating range.

## Section 7: Deadlock Architecture: Principles, Case Studies, & Multi-Resource Traps

### 7.1 Deadlock vs. Starvation
* **Starvation:** A progress anomaly where a fully operational, runnable thread is delayed indefinitely because the scheduler continuously chooses to allocate resources to other tasks. Starvation is a policy issue and *can* resolve on its own if system conditions change.
* **Deadlock:** A structural failure where a set of threads is permanently blocked because they are caught in a circular dependency, each waiting for a resource held by another. Deadlocks are unresolvable and can never break on their own without external intervention, such as force-terminating a process or restarting the system.

---

### 7.2 Case Study: The Single-Lane Bridge
The single-lane bridge crossing is a classic physical analogy for resource deadlocks:


* **Resource Mapping:** Each distinct section of the single-lane road acts as a exclusive hardware resource. A vehicle must control the segment beneath it and successfully acquire the upcoming segment before it can move forward.
* **Deadlock State:** If two vehicles traveling in opposite directions meet in the middle of the bridge, a deadlock occurs. Each vehicle controls one half of the bridge and is waiting for the other half to clear; neither can move forward, creating a permanent stall.
* **Recovery via Preemption:** To break this deadlock, an external force must intervene and force one of the vehicles to back up, rolling back its execution state and preempting its resource so the other vehicle can pass. If a long line of cars has built up behind the moving vehicle, multiple cars may have to be backed up sequentially to clear the jam.
* **Starvation Contradiction:** If a continuous stream of east-bound vehicles crosses the bridge at high speed, west-bound vehicles will sit stalled at the bank indefinitely. This is an example of **Starvation**, not deadlock; the system is still making forward progress, and the stall will clear naturally once the east-bound traffic thin out.

---

### 7.3 Non-Deterministic Lock Deadlocks
In multithreaded applications, deadlocks regularly manifest as non-deterministic timing bugs that depend on the exact order in which instructions interleave.

```c
// Cross-Locking Deadlock Pattern Architecture

pthread_mutex_t lock_x = PTHREAD_MUTEX_INITIALIZER;
pthread_mutex_t lock_y = PTHREAD_MUTEX_INITIALIZER;

void *Thread_A_Routine(void *arg) {
    pthread_mutex_lock(&lock_x);   // Acquire Lock X successfully 
    // If a context switch occurs here, Thread B can run and grab Lock Y 
    pthread_mutex_lock(&lock_y);   // Attempt to acquire Lock Y; stalls if held by B 
    
    /* Execute Critical Work */
    pthread_mutex_unlock(&lock_y);
    pthread_mutex_unlock(&lock_x);
    return NULL;
}

void *Thread_B_Routine(void *arg) {
    pthread_mutex_lock(&lock_y);   // Acquire Lock Y successfully 
    pthread_mutex_lock(&lock_x);   // Attempt to acquire Lock X; stalls if held by A 
    
    /* Execute Critical Work */
    pthread_mutex_unlock(&lock_x);
    pthread_mutex_unlock(&lock_y);
    return NULL;
}
```

This code pattern creates a classic **"Unlucky Case"** race condition. If Thread A acquires Lock X and is immediately context-switched out before it can request Lock Y, Thread B can run and acquire Lock Y. 

When both threads resume, Thread A will stall waiting for Lock Y, and Thread B will stall waiting for Lock X. Neither thread can make forward progress, locking up both paths permanently. This bug is exceptionally difficult to diagnose because it is non-deterministic; under a **"Lucky"** scheduling sequence, one thread may clear both locks before the second thread starts, allowing the program to complete successfully without any signs of a fault.

---

### 7.4 Multi-Resource Vectors: Space and Sockets
Deadlocks are not limited to software locks; they can occur across any finite system resources, including physical memory or network connections.

#### Memory Space Deadlocks
Consider a server with exactly $2\text{ MB}$ of available physical memory running two threads:
* **Thread A:** Calls `AllocateOrWait(1 MB)`, succeeds, and is context-switched out.
* **Thread B:** Calls `AllocateOrWait(1 MB)`, succeeds, and consumes the remaining memory.
* Next, both threads attempt to allocate an additional $1\text{ MB}$ of memory before releasing their initial allocations. Because the system's memory is fully exhausted, both threads enter a blocked state waiting for space to clear. Since neither thread can proceed to call `Free()`, the system enters a permanent memory deadlock.

## Section 8: Coffman Conditions & Resource-Allocation Graphs (RAG)

### 8.1 The Four Coffman Conditions
For a deadlock to occur, all four of the following physical conditions must hold simultaneously within the system:

1. **Mutual Exclusion:** Only one thread can use a specific resource at any given time. If a resource is fully shareable, it cannot participate in a deadlock.
2. **Hold and Wait:** A thread must currently hold at least one resource while waiting to acquire additional resources that are held by other active threads.
3. **No Preemption:** Resources cannot be forcefully reclaimed from a thread. A resource can only be released voluntarily by the thread holding it after its task finishes.
4. **Circular Wait:** There must exist a closed, circular loop of waiting threads, $\{T_1, T_2, \dots, T_n\}$, where $T_1$ is waiting for a resource held by $T_2$, $T_2$ is waiting for a resource held by $T_3$, and $T_n$ is waiting for a resource held by $T_1$.

---

### 8.2 Resource-Allocation Graph (RAG) Modeling
The operating system can model and detect deadlocks at runtime using a directed graph structure called a **Resource-Allocation Graph (RAG)**.


#### Mathematical Graph Formulation
* The graph consists of a set of vertices $V$ and directed edges $E$.
* The vertices are partitioned into two distinct classes:
  * Thread Nodes: $T = \{T_1, T_2, \dots, T_n\}$, represented as circles.
  * Resource Type Nodes: $R = \{R_1, R_2, \dots, R_m\}$, represented as squares. Internal dots within a square indicate the number of available instances of that resource type.
* **Directed Request Edge ($T_i \rightarrow R_j$):** Indicates that thread $T_i$ is currently blocked and waiting to acquire an instance of resource type $R_j$.
* **Directed Assignment Edge ($R_j \rightarrow T_i$):** Indicates that a specific instance of resource type $R_j$ has been allocated to and is currently held by thread $T_i$.

#### Analyzing Cycles and Deadlocks
* **Single-Instance Resources:** If every resource type node contains exactly one instance, **the presence of a cycle in the RAG is a necessary and sufficient condition for a deadlock**. If a cycle exists, the system is deadlocked; if no cycle exists, the system is safe.
* **Multi-Instance Resources:** If resource nodes contain multiple instances, a cycle is a *necessary* but *not sufficient* condition for a deadlock. If a cycle forms but an external thread outside the loop holds an instance of the same resource, that external thread can complete its work, release its instance, and break the cycle, resolving the stall without deadlock.

## Section 9: Algorithmic Deadlock Detection & Strategic Management Frameworks

### 9.1 The Deadlock Detection Vector Algorithm
For systems with multi-instance resources, the kernel uses a vector-based bank detection algorithm to scan the system for deadlocks. The algorithm attempts to find a valid execution sequence that allows all current threads to finish.

#### Vector Parameters
Let $m$ be the number of resource types in the system.
* `[Available]`: A vector of length $m$ tracking the current count of free, unallocated resources of each type.
* `[Allocation_x]`: A vector of length $m$ tracking the resources currently held by thread $X$.
* `[Request_x]`: A vector of length $m$ tracking the additional resources currently requested by thread $X$.

#### Algorithmic Execution Steps
```
1. Initialize a working tracking vector: [Avail] = [FreeResources].
2. Construct a list named UNFINISHED containing all active system nodes.

3. LOOP:
   done = true
   Foreach node in UNFINISHED {
       // Check if the node's resource request can be satisfied by the available pool
       if ([Request_node] <= [Avail]) { 
           remove node from UNFINISHED; // Assume the task can finish and return its resources 
           [Avail] = [Avail] + [Allocation_node]; // Reclaim all resources held by this node 
           done = false
       }
   }
   if (done == false) GOTO LOOP; // Continue iterating if progress was made

4. EVALUATION:
   if (UNFINISHED list is NOT empty) {
       The remaining nodes left inside the UNFINISHED list are formally DEADLOCKED.
   }
```

---

### 9.2 Complete Step-by-Step Matrix Simulation
Consider a system running three threads ($T_1, T_2, T_3$) and managing two resource types ($R_1, R_2$). Let's trace the deadlock detection algorithm against a specific resource allocation state.

#### Initial Vector Configurations
```
[FreeResources] = [0, 1]  (0 instances of R1, 1 instance of R2 are free)

Current Allocation State:
[Alloc_T1] = [1, 0] | [Request_T1] = [0, 1]
[Alloc_T2] = [0, 1] | [Request_T2] = [1, 0]
[Alloc_T3] = [0, 1] | [Request_T3] = [0, 0]
```

We step through the detection algorithm to evaluate system safety:

| Iteration Step | `[Avail]` Vector State | `UNFINISHED` List Contents | Node Evaluated | Conditional Check Status | Verification Results & Actions |
| :---: | :---: | :--- | :---: | :---: | :--- |
| **Step 0** | `[0, 1]` | `[T1, T2, T3]` | — | — | Initialization; copy free resources to `[Avail]`. |
| **Step 1** | `[0, 1]` | `[T1, T2, T3]` | **$T_1$** | `[0, 1] <= [0, 1]` <br> **(TRUE)** | $T_1$'s request can be satisfied. Move $T_1$ out of the list and reclaim its resources: <br> `[Avail] = [0,1] + [1,0] = [1, 1]`. |
| **Step 2** | `[1, 1]` | `[T2, T3]` | **$T_2$** | `[1, 0] <= [1, 1]` <br> **(TRUE)** | $T_2$'s request can be satisfied. Move $T_2$ out of the list and reclaim its resources: <br> `[Avail] = [1,1] + [0,1] = [1, 2]`. |
| **Step 3** | `[1, 2]` | `[T3]` | **$T_3$** | `[0, 0] <= [1, 2]` <br> **(TRUE)** | $T_3$'s request is empty and passes. Move $T_3$ out of the list: <br> `[Avail] = [1,2] + [0,1] = [1, 3]`. |
| **Step 4** | `[1, 3]` | **`Empty`** | — | — | Loop completes. Since `UNFINISHED` is completely empty, **the system is safe and free of deadlocks**. |

---

### 9.3 Strategic Management Frameworks
Operating systems use four core paradigms to manage the risk of resource deadlocks:

1. **Deadlock Prevention:** Design the system's code and software structures so that it is physically impossible for at least one of the four Coffman conditions to form. For example, forcing all threads to acquire locks in a strict, pre-defined global order completely eliminates the risk of circular waits.
2. **Deadlock Avoidance:** The kernel monitors resource requests dynamically at runtime. If a request is correct but has the potential to lead to a circular wait down the line, the kernel deliberately delays the thread's request until safety is restored. The **Banker's Algorithm** is the classic algorithmic framework used to implement deadlock avoidance.
3. **Deadlock Recovery:** Allows deadlocks to happen naturally, using a background detection thread to spot cycles in the resource-allocation graph. Once a deadlock is detected, the system recovers by force-terminating the deadlocked threads or rolling back their execution states to free up resources.
4. **Deadlock Denial (The Ostrich Algorithm):** Completely ignores the possibility of deadlocks. If deadlocks are rare and the engineering cost of prevention or avoidance is high, the system simply assumes deadlocks won't happen, relying on users to restart their applications if a stall occurs. Nearly all major general-purpose operating systems, including Linux, macOS, and Windows, use this pragmatic approach.

## Section 10: High-Yield Practice Exam Problems with Detailed Solutions

### Problem 1: Hard Real-Time Schedulability Testing
A hard real-time system is configured to coordinate three independent periodic tasks with parameters $T_i = (P_i, C_i)$:
* Task 1: $T_1 = (5, 2)$
* Task 2: $T_2 = (6, 1)$
* Task 3: $T_3 = (10, 1)$

1. Calculate the total processor utilization $U_{\text{Total}}$ for this task set.
2. Determine if this task set is guaranteed to meet all deadlines under **Rate Monotonic Scheduling (RMS)**.
3. Determine if this task set is guaranteed to meet all deadlines under **Earliest Deadline First (EDF)**.

#### Solution
1. **Calculate Total Processor Utilization:**
   We compute individual utilizations ($U_i = C_i / P_i$) and sum them up:
   $$U_1 = \frac{2}{5} = 0.400$$
   $$U_2 = \frac{1}{6} \approx 0.167$$
   $$U_3 = \frac{1}{10} = 0.100$$
   $$U_{\text{Total}} = 0.400 + 0.167 + 0.100 = \mathbf{0.667}$$

2. **Evaluate Rate Monotonic Schedulability:**
   We apply the Liu & Layland utilization bound test for $n = 3$ tasks:
   $$\text{Bound} = 3 \times \left( 2^{1/3} - 1 \right) \approx 3 \times (1.2599 - 1) \approx \mathbf{0.780}$$
   * **RMS Analysis:** Comparing our values, we see that $U_{\text{Total}} = 0.667 \le \text{Bound} = 0.780$. Because the total utilization is below the sufficient threshold, **the task set is guaranteed to meet every deadline under RMS**.

3. **Evaluate Earliest Deadline First Schedulability:**
   We check the EDF schedulability condition ($\sum U_i \le 1.0$):
   * **EDF Analysis:** Since $U_{\text{Total}} = 0.667 \le 1.0$, the workload satisfies the exact condition. **The task set is guaranteed to be fully schedulable under EDF**.

### Problem 2: Tracking Strides and Pass Counters
A stride scheduler coordinates three active threads (Thread A, Thread B, Thread C) using a global constant $W = 60,000$. The threads are assigned the following ticket counts:
* Thread A: $300\text{ tickets}$
* Thread B: $100\text{ tickets}$
* Thread C: $200\text{ tickets}$

1. Calculate the execution stride length for each thread.
2. Assuming all pass counters are initially zero, trace the scheduler's choices and update the counters step-by-step for the first 4 scheduling intervals.

#### Solution
1. **Calculate Stride Lengths:**
   Using the formula $S_i = W / N_i$, we compute the stride lengths:
   $$\text{Stride } S_A = \frac{60,000}{300} = \mathbf{200}$$
   $$\text{Stride } S_B = \frac{60,000}{100} = \mathbf{600}$$
   $$\text{Stride } S_C = \frac{60,000}{200} = \mathbf{300}$$

2. **Trace Scheduling Decisions and Counter Updates:**
   At each interval, the scheduler selects the thread with the lowest pass counter, runs it for a single time quantum, and adds its stride to its counter.

   * **Interval 1:** All pass counters are tied at $0$ (`Pass_A = 0, Pass_B = 0, Pass_C = 0`). 
     * *Selection:* **Thread A** is chosen (resolving the tie arbitrarily).
     * *Update:* `Pass_A = 0 + 200 = 200`. 
     * *State:* `[Pass_A = 200, Pass_B = 0, Pass_C = 0]`.

   * **Interval 2:** Evaluating counters `[200, 0, 0]`, Thread B and Thread C are tied for the lowest value ($0$).
     * *Selection:* **Thread B** is chosen.
     * *Update:* `Pass_B = 0 + 600 = 600`.
     * *State:* `[Pass_A = 200, Pass_B = 600, Pass_C = 0]`.

   * **Interval 3:** Evaluating counters `[200, 600, 0]`, Thread C has the lowest value ($0$).
     * *Selection:* **Thread C** is chosen.
     * *Update:* `Pass_C = 0 + 300 = 300`.
     * *State:* `[Pass_A = 200, Pass_B = 600, Pass_C = 300]`.

   * **Interval 4:** Evaluating counters `[200, 600, 300]`, Thread A has the lowest value ($200$).
     * *Selection:* **Thread A** is chosen.
     * *Update:* `Pass_A = 200 + 200 = 400`.
     * *State:* `[Pass_A = 400, Pass_B = 600, Pass_C = 300]`.

This deterministic tracking ensures that over time, the frequency of thread execution matches their ticket ratios ($300:100:200 \implies 3:1:2$) without any statistical variance.
