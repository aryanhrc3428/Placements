## Module 1: Core Scheduling Concepts and Metrics Review

### 1.1 Formal Definition of Scheduling
In computer systems architecture, **Scheduling** refers to the set of algorithmic decisions that dictate how shared resources are allocated among competing execution threads from moment to moment. While scheduling is most commonly discussed in the context of central processing unit (CPU) allocation, the fundamental engineering principles apply directly to other key hardware infrastructure components, including shared communication networks (network bandwidth) and persistent storage storage devices (disk read/write access).


The core design problem for an operating system's scheduler is managing the transition of tasks across various system queues as resources become available:

```
                     ┌──────────────────┐
                     │    Ready Queue   │ <───────────────────┐
                     └──────────────────┘                     │
                               │                               │ Time Slice Expired /
                       Scheduler Dispatch                      │ Hardware Preemption
                               │                               │
                               ▼                               │
                     ┌──────────────────┐                      │
                     │    Active CPU    │ ─────────────────────┤
                     └──────────────────┘                      │
                               │                               │
                          I/O Request                          │
                               │                               │
                               ▼                               │
                     ┌──────────────────┐                      │
                     │    I/O Queue     │                      │
                     └──────────────────┘                      │
                               │                               │
                         I/O Completion                        │
                               │                               │
                               └───────────────────────────────┘
```
*The workflow of thread transitions managed by a work-conserving kernel scheduler.*

### 1.2 Mathematical Analysis of Round-Robin Scheduling
**Round-Robin (RR)** scheduling is a preemptive policy where each process is given a small, fixed window of CPU time called a **Time Quantum** ($Q$). If a process's computation burst exceeds this quantum, a hardware timer interrupt fires, the process is preempted, and its thread control block is appended to the back of the ready queue.

#### Workload Specification
Consider a set of four processes arriving simultaneously at time $T = 0$, requiring the following CPU burst times:
* Process 1 ($P_1$): $53\text{ ms}$ 
* Process 2 ($P_2$): $8\text{ ms}$ 
* Process 3 ($P_3$): $68\text{ ms}$ 
* Process 4 ($P_4$): $24\text{ ms}$ 

Executing this workload under a Round-Robin policy with a time quantum $Q = 20\text{ ms}$ produces the following execution timeline:

```
Gantt Chart Execution Timeline:
├────P1────┼──P2──┼────P3────┼────P4────┼────P1────┼────P3────┼──P4──┼──P1──┼────P3────┼──P3──┤
0          20     28         48         68         88         108    112    125        145    153
```
*Chronological tracking of process allocations.*

#### Chronological Breakdown of Time Allocations
1.  `0 to 20 ms`: $P_1$ runs for the full time quantum. Remaining burst = $53 - 20 = 33\text{ ms}$.
2.  `20 to 28 ms`: $P_2$ runs. Because its burst requirement ($8\text{ ms}$) is less than the quantum, it completes early and yields the CPU at $28\text{ ms}$.
3.  `28 to 48 ms`: $P_3$ runs for the full quantum. Remaining burst = $68 - 20 = 48\text{ ms}$.
4.  `48 to 68 ms`: $P_4$ runs for the full quantum. Remaining burst = $24 - 20 = 4\text{ ms}$.
5.  `68 to 88 ms`: $P_1$ returns to run for its second quantum. Remaining burst = $33 - 20 = 13\text{ ms}$.
6.  `88 to 108 ms`: $P_3$ returns to run for its second quantum. Remaining burst = $48 - 20 = 28\text{ ms}$.
7.  `108 to 112 ms`: $P_4$ returns to execute its remaining $4\text{ ms}$, terminating at $112\text{ ms}$.
8.  `112 to 125 ms`: $P_1$ returns to execute its final $13\text{ ms}$, terminating cleanly at $125\text{ ms}$.
9.  `125 to 145 ms`: $P_3$ returns to run for its third quantum. Remaining burst = $28 - 20 = 8\text{ ms}$.
10. `145 to 153 ms`: $P_3$ finishes its remaining $8\text{ ms}$ of calculation, terminating the workload at $153\text{ ms}$.

#### Mathematical Calculation of Waiting Times
The waiting time for each process is calculated by tracking the total time it spent sitting in the ready queue:
$$\text{Waiting Time} = \text{Completion Time} - \text{Arrival Time} - \text{Total CPU Burst Time}$$

$$\text{Wait}(P_1) = 125 - 0 - 53 = (68 - 20) + (112 - 88) = \mathbf{72\text{ ms}} \text{ }$$
$$\text{Wait}(P_2) = 28 - 0 - 8 = (20 - 0) = \mathbf{20\text{ ms}} \text{ }$$
$$\text{Wait}(P_3) = 153 - 0 - 68 = (28 - 0) + (88 - 48) + (125 - 108) = \mathbf{85\text{ ms}} \text{ }$$
$$\text{Wait}(P_4) = 112 - 0 - 24 = (48 - 0) + (108 - 68) = \mathbf{88\text{ ms}} \text{ }$$

#### Average Workload Metrics
* **Average Waiting Time (AWT):**
    $$\text{AWT} = \frac{72 + 20 + 85 + 88}{4} = \frac{265}{4} = \mathbf{66\frac{1}{4}\text{ ms}} \text{ }$$
* **Average Completion Time (ACT):**
    $$\text{ACT} = \frac{125 + 28 + 153 + 112}{4} = \frac{418}{4} = \mathbf{104\frac{1}{2}\text{ ms}} \text{ }$$

#### Architectural Trade-offs
* **Advantages:** Round-Robin is fair and prevents long-running jobs from monopolizing the processor, significantly improving responsiveness for short, interactive tasks.
* **Disadvantages:** If the time quantum is small relative to the system's hardware limits, performance drops sharply because the CPU spends too much time saving and restoring registers during context switches.

## Module 2: Predictive and Adaptive Scheduling Algorithms

### 2.1 Shortest Job First (SJF) and Shortest Remaining Time First (SRTF)
If an operating system could predict the future, it could optimize waiting times by factoring task durations directly into scheduling decisions.

[Image illustrating Predictive Scheduling via Shortest Job First versus Shortest Remaining Time First models]

* **Shortest Job First (SJF):** A non-preemptive scheduling policy that selects the runnable task with the shortest total computation requirement. SJF is provably optimal for minimizing average waiting times among non-preemptive algorithms. It is also known as Shortest Time to Completion First (STCF).
* **Shortest Remaining Time First (SRTF):** The preemptive variant of Shortest Job First. If a newly arrived task has a shorter remaining execution time than the task currently running on the CPU, the scheduler immediately preempts the running task. SRTF is provably optimal for minimizing average response times across all preemptive models. It is also known as Shortest Remaining Time to Completion First (SRTCF).

#### Workload Invariant Behavior
* If all tasks in the ready queue have identical execution lengths, SRTF results in the exact same scheduling timeline as standard non-preemptive FCFS.
* When task lengths vary, SRTF maximizes responsiveness for short tasks by preventing them from getting stuck behind long-running calculations.

#### Predictability and Practical Considerations
* **Predicting the Future:** To implement SRTF perfectly, the system needs to know job runtimes. Some early systems asked the user for time estimates (and killed jobs that exceeded them to prevent cheating), but this is hard to predict even for non-malicious users.
* **The Yardstick:** Because we cannot perfectly know task durations, SRTF cannot be implemented exactly in practice. Instead, it serves as an optimal theoretical yardstick for measuring other scheduling policies.
* **SRTF Pros & Cons:**
  * **Pros:** Provably optimal for average response time (+).
  * **Cons:** Hard to predict the future (-), inherently unfair (-), and can lead to starvation for long jobs.

### 2.2 Case Study: Maximizing System and Device Utilization via SRTF
To see the benefits of SRTF, consider a multi-device system running three processes:
* **Process A and Process B:** Completely CPU-bound tasks, where each requires an entire week of continuous execution.
* **Process C:** A heavily I/O-bound task that executes in a short loop: $1\text{ ms}$ of CPU computation followed immediately by $9\text{ ms}$ of disk storage operations.

If these processes run independently without competition, Process C utilizes 90% of the disk device ($\frac{9\text{ ms}}{1\text{ ms} + 9\text{ ms}}$), while Process A or Process B can saturate 100% of the processor core.

#### 1. First-Come, First-Served Execution Behavior
If either Process A or Process B captures the CPU first under an FCFS policy, it will run continuously for a full week. As a result, the interactive Process C sits starved in the ready queue, forcing the system's physical storage disk to sit completely idle for the entire week and severely degrading hardware utilization.

#### 2. Round-Robin Execution Behavior with a $100\text{ ms}$ Time Quantum
If the system uses a standard Round-Robin policy with a $100\text{ ms}$ time quantum, execution alternates as follows:
* Process A runs for $100\text{ ms}$, then Process B runs for $100\text{ ms}$.
* Process C finally gains the CPU, executes its $1\text{ ms}$ calculation burst, and blocks on its $9\text{ ms}$ disk I/O operation.
* When the disk operation completes, Process C is moved back to the ready queue. However, it must wait for both Process A and Process B to complete their full $100\text{ ms}$ time slices before it can run again.
* **Hardware Efficiency:** The disk device can only ingest data for $9\text{ ms}$ out of every $201\text{ ms}$ cycle ($100\text{ ms} + 100\text{ ms} + 1\text{ ms}$). This drops the overall disk utilization to a meager **4.5%** ($\frac{9}{201}$), wasting significant hardware bandwidth.

#### 3. Round-Robin Execution Behavior with a $1\text{ ms}$ Time Quantum
To improve disk utilization, the time quantum could be dropped to $1\text{ ms}$. This allows Process C to run almost immediately when its disk I/O completes, increasing disk utilization back to **90%**. However, this configuration forces the system to execute a massive number of context switches per second, creating heavy kernel overhead that bogs down CPU throughput.

#### 4. Shortest Remaining Time First Execution Behavior
Under an SRTF policy, the scheduler recognizes that Process C has the shortest remaining CPU burst requirement (only $1\text{ ms}$). 

```
SRTF Optimal Pipeline Interleaving Timeline:
├─C─┼──────A──────┼─C─┼──────A──────┼─C─┼──────A──────┤
0   1             10  11            20  21            30
    └─── C's I/O ─┘   └─── C's I/O ─┘   └─── C's I/O ─┘
```
*Process C preempts the background calculation immediately upon completing its I/O burst.*

When Process C's disk operation finishes, it enters the ready queue. Because its upcoming computational burst ($1\text{ ms}$) is significantly shorter than the remaining runtimes of Process A or Process B, the scheduler immediately preempts the running background job. Process C runs for $1\text{ ms}$ and blocks on its next disk operation, freeing up the CPU. This pattern achieves **90% disk utilization** with minimal context-switching overhead, keeping the hardware highly efficient.

### 2.3 Adaptive Burst Forecasting via Exponential Moving Averages
Because an operating system cannot truly know an application's future runtime, it must predict upcoming CPU bursts by observing past execution history. This adaptive approach works because programs exhibit predictable behavior over time; a thread that was recently I/O bound is highly likely to remain I/O bound in its next execution phase.

```
                 [ ADAPTIVE BURST LENGTH ESTIMATION ]
                                  │
         Historical Record of Past Burst Lengths: tn-1, tn-2...
                                  │
                                  ▼
                    [ Exponential Moving Average ]
             • Formulates next predicted guess: τn
             • Mathematically blends recent and old data
                                  │
                                  ▼
      [ Smoothing Factor (α) balances current vs past trends ]
```

The scheduler tracks an application's execution history and forecasts its next CPU burst length ($\tau_n$) using an **Exponential Moving Average (EMA)** recurrence relation:

$$\tau_{n} = \alpha t_{n-1} + (1 - \alpha)\tau_{n-1}$$

Where:
* $t_{n-1}$ is the actual measured duration of the most recent CPU burst phase.
* $\tau_{n-1}$ is the historical predicted estimate calculated for that most recent burst.
* $\tau_{n}$ is the newly calculated prediction for the *next* upcoming CPU burst duration.
* $\alpha$ is a smoothing weight factor bounded strictly between 0 and 1 ($0 < \alpha \le 1$).

#### Expanding the Recurrence Relation
To understand how the smoothing factor weights past history, we expand the mathematical relation by substituting past terms:
$$\tau_{n} = \alpha t_{n-1} + (1-\alpha)[\alpha t_{n-2} + (1-\alpha)\tau_{n-2}]$$
$$\tau_{n} = \alpha t_{n-1} + \alpha(1-\alpha)t_{n-2} + \alpha(1-\alpha)^2 t_{n-3} + \dots + (1-\alpha)^k \tau_{n-k}$$

Because $(1-\alpha)$ is less than 1, its exponent increases with each step backward in time. This means older CPU bursts have an exponentially smaller impact on the current prediction, allowing the scheduler to adapt quickly to recent changes in application behavior.

## Module 3: High-Level Schedulers: MLFQ and the Linux O(1) Architecture

### 3.1 Multi-Level Feedback Queue (MLFQ) Architecture
The Multi-Level Feedback Queue is an adaptive scheduling framework designed to automatically distinguish between interactive, I/O-bound tasks and long-running, compute-bound tasks without requiring explicit statements from the user.

#### Identifying App Types
Before utilizing MLFQ, a core question arises: **How do we handle a simultaneous mix of different types of apps (interactive vs. high throughput)?**
* Can we trust an app to self-report as "interactive"?
* Should a scheduler treat identical apps the same across servers, workstations, tablets, and phones?
* Schedulers typically encode the assumption that apps which sleep a lot and have short bursts are interactive and need high priority, whereas compute-heavy apps get lower priority. However, this is imperfect (e.g., what about apps that sleep long then compute long, or periodic mandatory tasks?).

#### Core Structural Invariants
* **Priority Tiers:** The scheduler maintains multiple distinct ready queues, where each queue is assigned a different priority level. Queues near the top are designated for high-priority foreground tasks, while queues at the bottom handle low-priority background tasks.
* **Dynamic Promotion and Demotion Loops:**
    1.  Every newly created or awakened task starts at the highest priority queue.
    2.  If a task uses up its entire allocated time quantum without blocking for I/O, the scheduler assumes it is a compute-bound job and demotes it down one level to a lower priority queue.
    3.  If a task blocks for I/O and yields the CPU before its time quantum expires, the scheduler assumes it is an interactive task and retains or promotes its position near the top of the priority tiers.
* **Scheduling Between Queues:** 
    * **Fixed Priority:** Serve all tasks from the highest priority queue, then next, etc.
    * **Time Slice Allocation:** Give each queue a certain percentage of CPU time (e.g., 70% to the highest queue, 20% to the next, 10% to the lowest) to prevent starvation.
* **Variable Quantum Scaling:** To maximize throughput, the length of the time quantum increases exponentially at lower priority levels (e.g., top queue $Q=1\text{ ms}$, next $Q=2\text{ ms}$, next $Q=4\text{ ms}$, bottom uses FCFS execution).

#### System Gaming Exploits and Countermeasures
Because MLFQ adjusts priorities based on time slice expirations, a clever application can "game" the scheduler to maintain artificially high priority. 

If a compute-bound task executes for 99% of its time quantum and then issues a brief, meaningless I/O operation (such as writing a single character to a terminal log via a `printf` wrapper), it yields the CPU voluntarily. The scheduler sees that the task yielded before timeout and retains its position in the highest-priority foreground tier. This allows the task to run significantly faster than its competitors by monopolizing the top queue. 

> **Example:** In a class Othello programming competition, a clever student realized that to win, their AI needed more compute time than competitors. By strategically inserting meaningless `printf` statements, the program constantly yielded just before its quantum expired, maintaining top-tier interactive priority and running much faster overall!

To prevent this exploit, modern MLFQ implementations track a task's *total accumulated runtime* across its time slice rather than resetting the window on every yield.

#### Scheduling Fairness Tradeoffs
Implementing strict fixed-priority scheduling between queues is inherently unfair. There is an old urban legend that when a Multics mainframe was finally shut down, operators found a 10-year-old background job still waiting in the lowest priority queue! Ultimately, fairness is gained by hurting average response time.
To handle this, some variants of UNIX periodically increase the priority of jobs that haven't received service (treating queues like express lanes at a supermarket—if the express lane is too long, jobs get better service moving to other lines). However, this can backfire under heavy load: if no job gets CPU time, everyone's priority increases, and true interactive jobs suffer.

### 3.2 Case Study: The Linux $O(1)$ Kernel Scheduler
The Linux $O(1)$ scheduler is a priority-based scheduling architecture designed to make scheduling choices in absolute constant time ($O(1)$ complexity), regardless of how many active tasks are running on the system.

```
Linux O(1) Priority Address Range:
┌───────────────────────────────────────────┬───────────────────────────────────────────┐
│     Realtime/Kernel Tasks (0 to 99)       │        Standard User Tasks (100 to 139)   │
└───────────────────────────────────────────┴───────────────────────────────────────────┘
◄── Higher Priority (Lower Value)                                   Lower Priority ───►
```
*The strict division of priority ranges inside the O(1) scheduling layout.*

#### Core Mechanical Components
* **Priority Array Layout:** The scheduler supports exactly 140 unique priority levels. Levels 0 through 99 are strictly reserved for high-priority real-time and kernel tasks, while levels 100 through 139 map to standard user-space applications (configured via the system `nice` utility).
* **Constant-Time Selection Table:** To achieve $O(1)$ selection speeds, the scheduler maps active tasks onto a **140-bit hardware bitmask table**. Each bit tracks whether its corresponding priority queue contains runnable tasks. The CPU uses a specialized hardware instruction (such as `sched_find_first_bit` or find-first-set asm) to locate the highest-priority active bit in constant time, entirely avoiding the need to scan through long runqueues.
* **Dual Runqueue Swapping:** The scheduler maintains two separate priority arrays: the **Active Array** and the **Expired Array**.
    1.  The scheduler always selects and executes tasks from the highest-priority queue within the active array.
    2.  When a task consumes its entire allocated time quantum, the scheduler calculates its new priority and appends it to the appropriate queue inside the expired array.
    3.  Once the active array's bitmask hits zero (meaning all runnable tasks have exhausted their time slices), the scheduler performs a constant-time pointer swap, turning the expired array into the new active array and vice versa.

#### Interactive Heuristics Formulas
To keep user-space applications responsive, the scheduler adjusts a task's base priority level by up to $\pm 5$ points based on its execution history. The scheduler tracks interactivity using an internal tracking metric:

$$\Delta_{\text{Priority}} = f(\text{sleep\_avg})$$
$$\text{sleep\_avg} = \text{Time}_{\text{Biologically Asleep}} - \text{Time}_{\text{Actively Running}} \text{ }$$

Tasks with a high `sleep_avg` are classified as I/O-bound or interactive, prompting the scheduler to reduce their priority number (boosting their execution priority). 

To prevent temporary changes in behavior from causing rapid priority shifts, the scheduler uses an **Interactive Credit (IC)** tracking metric. Tasks earn credit when they sleep for extended periods and spend credit when they execute long calculations. This credit acts as a buffer (hysteresis), stabilizing the task's priority mapping. Furthermore, interactive tasks receive "special dispensation": they are placed back into the active queue instead of the expired queue when their time slice ends, unless some other task has been starved for too long.

#### Real-Time Tasks Scheduling
Real-time tasks (priorities 0-99) bypass these heuristics:
* They always preempt non-RT tasks.
* There is no dynamic adjustment of their priorities.
* They use specialized scheduling schemes:
  * **`SCHED_FIFO`:** Preempts other tasks and runs with no timeslice limit (runs until it blocks or yields).
  * **`SCHED_RR`:** Preempts normal tasks and uses round-robin scheduling amongst other RT tasks of the exact same priority.

## Module 4: Multiprocessor and Multi-Core Coordination Layouts

### 4.1 Thread versus Process Scheduling Costs
Modern operating systems schedule individual execution **threads** or task groups rather than full processes. When a core switches contexts between tasks, the performance cost depends heavily on whether the tasks share an address space:
* **Switching Threads within the Same Process:** The kernel only needs to save and restore general-purpose registers, program counters, and stack pointers. This is fast and efficient because the active memory mapping remains unchanged.
* **Switching Processes:** The kernel must completely change the active memory mapping by reloading page table pointers (e.g., register `CR3` on x86). This is expensive because it invalidates local Translation Lookaside Buffers (TLBs) and flushes hardware caches, causing cache misses that degrade system throughput.

> **Note on Simultaneous Multithreading (Hyperthreading):** Different threads can be interleaved on a cycle-by-cycle basis and can even be in different processes (different address spaces). This is handled via hardware-supported interleaving, allowing execution parallelization directly within a single CPU pipeline.

---

### 4.2 Multi-Core Optimization Vectors
Multiprocessor and multi-core scheduling introduces unique performance challenges, particularly around cache line contention and data sharing.


* **Core-Local Runqueues:** Rather than using a single global ready queue—which would cause heavy lock contention as cores compete to pull tasks—modern schedulers assign a private, independent set of priority queues to each individual CPU core. This allows each core to schedule tasks independently, using background load-balancing algorithms to periodically redistribute work across cores if one becomes overloaded. Scheduling of these per-core queues can be synchronized safely using local interrupt enable/disable.
* **Affinity Scheduling:** When a thread finishes an I/O operation and returns to the ready queue, the scheduler attempts to re-schedule it onto the **exact same physical CPU core** it was running on previously. This is known as affinity scheduling, and it improves performance by maximizing local cache reuse and preserving branch prediction histories.
* **Gang Scheduling & Space Sharing:** Tightly-coupled parallel applications often split work across multiple threads that communicate frequently via synchronization barriers. If the scheduler picks these threads independently, a thread can easily waste its time slice busy-waiting on a barrier for a peer thread that isn't currently running. To prevent this, the scheduler implements **Gang Scheduling**, identifying related threads and scheduling them to run *simultaneously* across separate physical CPU cores. This keeps spin-locks efficient and significantly improves parallel execution throughput.
  * **Scheduler Activations:** The OS informs a parallel program exactly how many physical processors its threads are scheduled on. The application can then adapt its parallelization strategy to the number of cores it actually has.
  * **Space Sharing:** Sharing space with other parallel programs is often more efficient because parallel speedup is frequently sublinear with respect to the number of cores.

#### Gang Scheduling Visual Interleaving Pattern
```
Without Gang Scheduling (Inefficient):
Core 1: [ Thread 1 (Active)  ][ Thread X (Compute) ]
Core 2: [ Thread Y (Compute) ][ Thread 2 (Active)  ] 
Result: Thread 1 wastes its time slice spinning on a barrier because Thread 2 is suspended.

With Gang Scheduling (Optimal Integration):
Core 1: [ Thread 1 (Active)  ]
Core 2: [ Thread 2 (Active)  ]
Result: Both threads run concurrently, allowing them to pass the barrier immediately.
```

### 4.3 Advanced Synchronization Primitive Code Implementation
When building synchronization primitives for multiprocessor systems, systems programmers must choose between letting a thread sleep or forcing it to busy-wait based on the expected wait time.

#### Code Example: The Basic Multiprocessor Spinlock
A basic spinlock uses an atomic hardware instruction to lock a variable without entering the kernel.

```c
// Basic test&set Spinlock Verification Implementation
int lock_state = 0; // 0 = FREE, 1 = BUSY 

void Spinlock_Acquire(int *lock_ptr) {
    // Every call to test&set executes an atomic write instruction.
    // Under cache coherence protocols, a write requires exclusive access, 
    // which invalidates that cache line across all other CPU cores.
    while (test_and_set(lock_ptr)) {
        // Busy-wait loop: consumes 100% CPU cycles 
    }; 
}

void Spinlock_Release(int *lock_ptr) {
    // Atomic store resets the lock value
    *lock_ptr = 0; // 
}
```

#### Code Example: The Optimized `test&test&set` Spinlock
If multiple cores compete for the basic spinlock above, their constant write attempts force the cache line to bounce back and forth between core-local caches ("cache line ping-ponging"), consuming massive interconnect network bandwidth. 

To eliminate this overhead, systems programmers use an optimized pattern called a **`test&test&set` lock**.

```c
// Optimized test&test&set Spinlock Implementation
void TestAndTestAndSet_Acquire(volatile int *lock_ptr) {
    do {
        // 1. Pure Read Spin: Loop locally on a plain read instruction.
        // This allows the cache line to remain in a shared, read-only state across cores.
        // Caller reads the value directly from its local L1/L2 cache without generating bus traffic.
        while (*lock_ptr) {
            // Spin locally in hardware cache line space 
        }
        
        // 2. Atomic Try: Only attempt an actual write when the lock appears to be free.
    } while (test_and_set(lock_ptr)); // 
}
```

## Module 5: Starvation and Progress Anomalies

An operating system's scheduler must ensure that all runnable tasks make forward progress over time. 

### 5.1 Starvation versus Deadlock Definitions
* **Starvation:** A progress anomaly where a fully operational, runnable thread is prevented from making forward progress for an indefinite period because the scheduler continuously chooses to allocate resources to other tasks. Starvation is distinct from a deadlock because it *can* resolve on its own if system conditions change (e.g., if high-priority traffic clears).
* **Deadlock:** A structural system failure where two or more threads are permanently blocked because they are caught in a circular dependency, each waiting for a resource held by another. Deadlocks are unresolvable and cannot be cleared without manual intervention or a system reset.

---

### 5.2 Comparative Policy Starvation Vulnerability Matrix
The table below analyzes the starvation risks of the different scheduling policies covered across lectures.

| Scheduling Policy / Model | Prone to Starvation? | Primary Cause of Starvation / Progress Anomalies |
| :--- | :---: | :--- |
| **Non-Work-Conserving Scheduler**  | **Yes**  | Leaves the CPU core idle even when there are valid, runnable tasks waiting in the system queues. |
| **Last-Come, First-Served (LCFS / Stack)**  | **Yes**  | If the task arrival rate exceeds the system's processing speed, late-arriving jobs saturate the CPU, trapping early-arriving tasks at the bottom of the stack indefinitely. |
| **First-Come, First-Served (FCFS / FIFO)**  | **Yes**  | If a task falls into an infinite loop or encounters a bug in a non-preemptive environment, it will hog the CPU forever, starving all other tasks in the queue. |
| **Round-Robin (RR)**  | **No**  | Guaranteed fairness. With $N$ active processes and a time quantum $Q$, every task is guaranteed to run again within a predictable window of at most $(N-1) \times Q\text{ ms}$. |
| **Shortest Remaining Time First (SRTF)**  | **Yes**  | If the system faces a continuous stream of short, rapid execution tasks, long-running compute jobs will be starved indefinitely. |
| **Multi-Level Feedback Queue (MLFQ)**  | **Yes**  | Because MLFQ uses runtime metrics to approximate SRTF, long-running compute tasks are continuously demoted to the bottom queues, where they starve if top-tier jobs keep arriving. |
| **Strict Priority Scheduling**  | **Yes**  | The scheduler always selects the thread with the highest priority. Lower-priority background tasks are blocked from executing as long as high-priority tasks are active. |

## Module 6: Priority Inversion and Priority Donation Protocols

### 6.1 The Mechanics of Priority Inversion
**Priority Inversion** is a critical scheduling failure that occurs in priority-based systems when a high-priority thread is indirectly blocked from running by a lower-priority task, violating the system's priority rules.

#### Step-by-Step Breakdown of a Priority Inversion
Consider three jobs running under a strict priority scheduler:
* **Job 3 ($J_3$):** High Priority 
* **Job 2 ($J_2$):** Medium Priority 
* **Job 1 ($J_1$):** Low Priority 

```
Execution Step Timeline:
1. J1 (Low) runs naturally and acquires a mutual exclusion lock.
                               │
                               ▼
2. J3 (High) wakes up, preempts J1, and attempts to acquire the same lock.
   Because J1 holds the lock, J3 blocks and yields the CPU.
                               │
                               ▼
3. J2 (Medium) wakes up. Since J2's priority is higher than J1, the scheduler
   runs J2 immediately. J2 completely blocks J1 from executing.
                               │
                               ▼
[ CRITICAL BREAKDOWN ]: Because J1 cannot run, it can never complete its critical
                       section to release the lock. As a result, the medium-priority
                       J2 successfully starves the high-priority J3 via J1.
```

---

### 6.2 The Priority Donation (Inheritance) Mitigation Protocol
To resolve priority inversion, modern operating systems implement a coordination protocol called **Priority Donation** (or Priority Inheritance).

[Image tracing the Priority Donation flow where Job 3 temporarily boosts Job 1's scheduling priority]

#### The Donation Workflow
1.  When a high-priority thread ($J_3$) blocks on a mutual exclusion lock currently held by a low-priority thread ($J_1$), the scheduler steps in.
2.  The scheduler temporarily boosts $J_1$'s priority, granting it the **high priority level of $J_3$**.
3.  This temporary boost allows $J_1$ to run ahead of any intermediate medium-priority tasks ($J_2$), ensuring it can execute its critical section without interference.
4.  As soon as $J_1$ completes its critical section and calls `Release()`, its priority drops back down to its base level. $J_3$ immediately acquires the lock and resumes execution, maintaining the correct priority sequence.

### 6.3 Case Study: The Martian Pathfinder Mission Recovery
On July 4, 1997, NASA’s Martian Pathfinder rover successfully landed on the surface of Mars, delivering the first mobile exploration vehicle to the planet using a novel air-filled balloon system. 

However, a few days into the mission, the rover began experiencing random, unprovoked system resets that caused the real-time operating system (VxWorks) to reboot, losing critical scientific progress and telemetry data.


#### The Failure Core (Priority Inversion)
The rover's software tasks competed for access to a shared tracking memory architecture (the information bus) protected by a mutual exclusion lock inside the VxWorks kernel:
* **High-Priority Task:** The Data Distribution Task, responsible for routing telemetry and mission critical system parameters. This task executed on a strict real-time deadline.
* **Medium-Priority Tasks:** A collection of standard background flight controls, telemetry encoders, and communication handlers.
* **Low-Priority Task:** The ASI/MET atmospheric instrument data collector task, which periodically logged basic meteorological sensor readings into the memory bus.

The system encountered the exact priority inversion sequence described in Section 6.1:
1.  The low-priority atmospheric task acquired the lock on the shared memory bus.
2.  The high-priority data distribution task awoke and attempted to acquire the bus lock, forcing it to block and enter a suspended state.
3.  In the meantime, a collection of medium-priority background flight tasks awoke. Since their priority was higher than the atmospheric task, the scheduler ran them immediately, completely starving the low-priority task.
4.  Because the atmospheric task was blocked from running, it could never release the memory bus lock. 
5.  **The Watchdog Trigger:** The real-time system's **Watchdog Timer** noticed that the critical high-priority data distribution task had not run for an extended period, missing its real-time deadline. Recognizing this lack of forward progress as a critical system hang, the watchdog forced a hard hardware reset to return the rover to a safe state, wiping out active calculations.

#### The Engineering Fix
The software team traced the issue back to a specific design choice: the original developers had explicitly **turned off priority donation** inside the VxWorks lock configuration because they were worried about the slight memory and processing overhead required to update priority metadata. 

To fix the rover, the engineering team uploaded a remote software patch that re-enabled priority inheritance for the memory bus lock, allowing the atmospheric task to quickly clear its critical section and preventing further system resets.

### Problem 1: Priority Inversion Cascade Analysis
Suppose a strict priority scheduler coordinates five active jobs with priorities ranging from 1 (lowest) to 5 (highest):
* Job $J_1$: Priority = 1
* Job $J_2$: Priority = 2
* Job $J_3$: Priority = 3
* Job $J_4$: Priority = 4
* Job $J_5$: Priority = 5

The system contains two distinct mutual exclusion locks, `LockA` and `LockB`. Consider the following execution timeline:
1.  At $T = 0$, $J_1$ is the only active thread. It runs and calls `Acquire(&LockA)`.
2.  At $T = 10$, $J_2$ wakes up, preempts $J_1$, and calls `Acquire(&LockB)`.
3.  At $T = 20$, $J_5$ wakes up, preempts $J_2$, and calls `Acquire(&LockA)`. It blocks immediately because `LockA` is held by $J_1$.
4.  At $T = 30$, $J_4$ wakes up and wants to execute a long computation burst. It does not require any locks.
5.  At $T = 40$, $J_3$ wakes up and calls `Acquire(&LockB)`. It blocks immediately because `LockB` is held by $J_2$.

#### Question
If the system does **not** implement priority donation, trace which job the scheduler will choose to run at time $T = 45$, and explain which tasks are currently starved.

#### Solution
To find the active thread at $T = 45$, we analyze the state of each job sequentially:
* **Job 5 ($J_5$, Priority 5):** Blocked since $T = 20$, waiting on `LockA` which is held by $J_1$.
* **Job 4 ($J_4$, Priority 4):** Fully runnable and ready to execute its computation burst.
* **Job 3 ($J_3$, Priority 3):** Blocked since $T = 40$, waiting on `LockB` which is held by $J_2$.
* **Job 2 ($J_2$, Priority 2):** Fully runnable but currently preempted. It holds `LockB`.
* **Job 1 ($J_1$, Priority 1):** Fully runnable but currently preempted. It holds `LockA`.

The scheduler evaluates the ready tasks ($J_4, J_2, J_1$) and selects the one with the highest priority, which is **Job 4 ($J_4$)**.

#### Starvation Impact Analysis
* $J_5$ (the highest priority task) is starved because it is blocked on $J_1$, which is completely preempted by $J_4$.
* $J_3$ is starved because it is blocked on $J_2$, which is also preempted by $J_4$.
* $J_2$ and $J_1$ are starved because $J_4$ has a higher priority and will continue to hog the CPU until its long computation completes.
* **Conclusion:** This is a classic priority inversion failure. The un-synchronized, un-related $J_4$ is successfully starving the highest priority task $J_5$ by blocking $J_1$ from running to release the lock.

### Problem 2: Exponential Moving Average Multi-Step Prediction Trace
A scheduler uses an Exponential Moving Average (EMA) to predict the length of a thread's next CPU burst. The smoothing factor is configured to $\alpha = 0.5$, and the initial prediction guess is set to $\tau_1 = 10\text{ ms}$.

The thread executes three sequential CPU burst phases, resulting in the following actual physical runtimes:
* Actual burst 1 ($t_1$): $6\text{ ms}$
* Actual burst 2 ($t_2$): $4\text{ ms}$
* Actual burst 3 ($t_3$): $14\text{ ms}$

#### Question
Calculate the step-by-step predictions ($\tau_2, \tau_3, \tau_4$) and determine the final estimated burst length $\tau_4$ that the scheduler will use to evaluate the thread's next execution phase.

#### Solution
We calculate the adaptive predictions step-by-step using the EMA formula:
$$\tau_{n} = \alpha t_{n-1} + (1 - \alpha)\tau_{n-1} \text{ }$$

Given $\alpha = 0.5$, the formula simplifies to a straightforward average of the most recent actual burst and the previous prediction:
$$\tau_{n} = 0.5 t_{n-1} + 0.5 \tau_{n-1}$$

#### Step 1: Calculate Prediction $\tau_2$
Using the actual length of the first burst ($t_1 = 6\text{ ms}$) and the initial guess ($\tau_1 = 10\text{ ms}$):
$$\tau_{2} = (0.5 \times t_{1}) + (0.5 \times \tau_{1})$$
$$\tau_{2} = (0.5 \times 6) + (0.5 \times 10) = 3 + 5 = \mathbf{8.0\text{ ms}}$$

#### Step 2: Calculate Prediction $\tau_3$
Using the actual length of the second burst ($t_2 = 4\text{ ms}$) and the previous prediction ($\tau_2 = 8.0\text{ ms}$):
$$\tau_{3} = (0.5 \times t_{2}) + (0.5 \times \tau_{2})$$
$$\tau_{3} = (0.5 \times 4) + (0.5 \times 8.0) = 2 + 4 = \mathbf{6.0\text{ ms}}$$

#### Step 3: Calculate Prediction $\tau_4$
Using the actual length of the third burst ($t_3 = 14\text{ ms}$) and the previous prediction ($\tau_3 = 6.0\text{ ms}$):
$$\tau_{4} = (0.5 \times t_{3}) + (0.5 \times \tau_{3})$$
$$\tau_{4} = (0.5 \times 14) + (0.5 \times 6.0) = 7 + 3 = \mathbf{10.0\text{ ms}}$$

The adaptive scheduler adjusts its final estimate to **$10.0\text{ ms}$**, factoring in the sudden long burst while smoothing out short-term spikes.
