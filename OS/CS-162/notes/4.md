## Section 1: Concurrency vs. Parallelism Revisited

To build high-performance systems, we must distinguish between how threads are executed logically versus physically.

* **Concurrency:** The logical grouping and management of multiple tasks over overlapping time frames. It is about **handling multiple things at once (MTAO)**. On a single-core processor, concurrency is achieved by interleaved execution (time-slicing), where the operating system's scheduler dynamically switches between threads.
* **Parallelism:** The physical execution of multiple tasks at the exact same physical instant. It is about **doing multiple things simultaneously**. Parallelism requires multiple physical processor cores or execution units.

```
not Concurrent       not Parallel         [ Thread A ] ───────────────>
                                          [ Thread B ] ───────────────> (Sequential execution)

Concurrent           not Parallel         [ A ][ B ][ A ][ B ][ A ] ──> (Interleaved on 1 Core)

Concurrent           and Parallel         Core 1: [ Thread A ] ───────>
                                          Core 2: [ Thread B ] ───────> (Simultaneous on 2 Cores)
```
*Based on the conceptual execution models of threads.*

### Schedulability and Time-Slicing
When running threads concurrently, the scheduler is free to execute them in any order and with any granularity of interleaving. A thread might run to absolute completion without interruption, or it might be preempted and time-sliced in large or extremely small instruction chunks. Programs must be designed to behave correctly regardless of the scheduling order.

## Section 2: Thread Memory Layout & Stack Management

In a multi-threaded process, all threads reside inside a single, shared address space. While they share global data and the heap, each thread must have its own private execution context, which includes its own set of CPU registers and its own private execution stack.


### Key Engineering Challenges in Multi-threaded Layouts
When laying out multiple thread stacks inside a single process address space, systems programmers must address several design challenges:

1. **Stack Positioning:** Where do we position the stacks relative to each other? In a single-threaded process, the stack grows downwards from the top of the address space, and the heap grows upwards from the bottom. With multiple threads, the OS must allocate multiple stacks inside the address space, leaving virtual memory gaps between them.
2. **Determining Stack Size:** What maximum size should we choose for each stack? If the allocated stack size is too small, a thread executing deeply nested or recursive function calls will quickly run out of stack space. If it is too large, the process's virtual address space becomes fragmented, limiting the maximum number of threads that can be spawned.
3. **Handling Stack Violations:** What happens if a thread violates its stack boundary? If a thread's stack grows too large, it can overflow into the memory region allocated for another thread's stack or the heap. This leads to silent data corruption, which is exceptionally difficult to debug.
4. **Catching Violations (Guard Pages):** How can we catch stack overflows? Modern operating systems place a read-and-write-protected virtual memory page—called a **Guard Page**—directly below each thread's stack. If a thread exceeds its stack limit and touches this page, the MMU triggers a page fault exception, allowing the kernel to catch the violation immediately and terminate the offending thread (producing a Segmentation Fault) rather than allowing silent memory corruption.

## Section 3: Thread Interleaving, Nondeterminism, & Race Conditions

### 3.1 The Illusion vs. Physical Reality
The programmer is presented with the illusion of an infinite number of processors, where each thread executes sequentially on its own dedicated virtual core. 

In reality, the physical processor core(s) multiplex threads dynamically. Threads execute at variable "speeds" because of hardware interrupts, cache misses, page faults, and scheduling decisions. Programs must be designed to work correctly under any possible scheduling timeline.

### 3.2 Programmer vs. Processor View of Execution
Consider the following sequence of instructions:
```c
x = x + 1;
y = y + x;
z = x + 5 * y;
```

To the programmer, these instructions appear to execute atomically and sequentially. However, the processor can suspend a thread at any point (even mid-instruction at the assembly level) to run other threads, resuming it later.

```
Possible Execution Timeline #1:
[ Thread 1: x = x+1 ] ──> [ Thread 1: y = y+x ] ──> [ Thread 1: z = x+5y ] (Runs uninterrupted)

Possible Execution Timeline #2:
[ Thread 1: x = x+1 ] ──> (Thread 1 Suspended / Other Threads Run) ──> [ Thread 1: y = y+x ] ──> [ Thread 1: z = x+5y ]

Possible Execution Timeline #3:
[ Thread 1: x = x+1 ] ──> [ Thread 1: y = y+x ] ──> (Thread 1 Suspended / Other Threads Run) ──> [ Thread 1: z = x+5y ]
```
*Thread suspension can occur between any two instructions, altering the interleaving of operations.*

### 3.3 Independent vs. Cooperating Threads
* **Independent Threads:** These threads do not share any state or memory with other threads. Their execution is completely deterministic and reproducible, meaning the same inputs will always produce the same outputs.
* **Cooperating Threads:** These threads share memory, data structures, or hardware resources. They are non-deterministic and their behavior depends on the exact scheduling order of instructions, making them highly susceptible to race conditions if not properly synchronized.
* **The Goal:** Build cooperating systems with **Correctness by Design**.

### 3.4 Race Condition Case Studies

#### Case Study 1: Independent Threads
*Initially, $x = 0$ and $y = 0$.*
* **Thread A:** `x = 1;`
* **Thread B:** `y = 2;`

**Question:** What are the possible values of $x$ after both threads finish?  
**Answer:** $x$ must be $1$. Because Thread A and Thread B are independent and do not write to the same memory locations, Thread B cannot interfere with the value of $x$.

#### Case Study 2: Cooperating Threads (Race Condition)
*Initially, $x = 0$ and $y = 0$.*
* **Thread A:** `x = y + 1;`
* **Thread B:** `y = 2; y = y * 2;`

**Question:** What are the possible values of $x$ after both threads finish?  
**Answer:** $x$ can non-deterministically be **$1$, $3$, or $5$** depending on instruction interleaving:
* **Scenario 1 ($x = 1$):** Thread A executes before Thread B starts.
  $$x = y + 1 \implies x = 0 + 1 = 1$$
* **Scenario 2 ($x = 3$):** Thread B executes its first instruction (`y = 2`), then Thread A preempts Thread B and runs (`x = y + 1`), and finally Thread B finishes (`y = y * 2`).
  $$x = 2 + 1 = 3$$
* **Scenario 3 ($x = 5$):** Thread B runs to completion (`y` becomes $4$), and then Thread A runs.
  $$x = 4 + 1 = 5$$

This is a classic **Race Condition**, where Thread A races against Thread B to access the shared variable $y$.

## Section 4: Mutual Exclusion & Locks

To prevent race conditions, cooperating threads must synchronize their access to shared data structures.


### 4.1 Key Synchronization Definitions
* **Synchronization:** The coordination of thread execution to ensure correct behavior when accessing shared data.
* **Mutual Exclusion:** A mechanism that ensures only one thread can perform a specific operation or access a shared resource at any given time.
* **Critical Section:** A block of code that accesses a shared resource and must not be entered by more than one thread concurrently.
* **Lock:** A software object that provides mutual exclusion, allowing only one thread to hold it at a time.

### 4.2 The Lock Interface
A lock exposes two fundamental atomic operations:
* `Lock.acquire()`: Attempts to acquire the lock. If the lock is currently free, the calling thread marks it as busy and continues executing. If the lock is busy, the calling thread is blocked and must wait until the lock becomes free.
* `Lock.release()`: Marks the lock as free, allowing one of the waiting threads to acquire it. This operation should only be called by the thread currently holding the lock.

### 4.3 Synchronizing a Concurrent Insert
In the tree-based set data structure shown above, we can guarantee thread safety by wrapping our operations in acquire and release calls:

```
Thread A                              Thread B
Lock.acquire();                       Lock.acquire(); (blocks if Thread A has it)
Insert(3) into tree;                  Insert(4) into tree;
Lock.release();                       Lock.release();
```
*Ensuring sequential execution of inserts through mutual exclusion.*

## Section 5: POSIX Mutex API & Fixing Race Conditions in C

### 5.1 The pthreads Mutex Interface
The standard POSIX threads library provides mutual exclusion objects called **mutexes**:

```c
// Dynamic initialization of a mutex
int pthread_mutex_init(pthread_mutex_t *mutex, const pthread_mutexattr_t *attr); //

// Locking and unlocking operations
int pthread_mutex_lock(pthread_mutex_t *mutex); //
int pthread_mutex_unlock(pthread_mutex_t *mutex); //
```

### 5.2 Analyzing a Vulnerable Concurrent Program
The following program spawns a user-defined number of threads that concurrently increment a shared global variable without locking:

```c
#include <stdio.h>
#include <stdlib.h>
#include <pthread.h>
#include <string.h>

int common = 162; // Shared global variable

void *threadfun(void *threadid) {
    long tid = (long)threadid; //
    
    // VULNERABLE CRITICAL SECTION: 'common++' is not atomic!
    // It compiles down to: Read common, Increment register, Write back common.
    printf("Thread #%lx stack: %lx common: %lx (%d)\n", 
           tid, (unsigned long)&tid, (unsigned long)&common, common++); //
           
    pthread_exit(NULL); //
}

int main(int argc, char *argv[]) {
    long t; //
    int nthreads = 2; //
    if (argc > 1) {
        nthreads = atoi(argv[1]); //
    }
    
    pthread_t *threads = malloc(nthreads * sizeof(pthread_t)); //
    
    printf("Main stack: %lx, common: %lx (%d)\n", 
           (unsigned long)&t, (unsigned long)&common, common); //
           
    for (t = 0; t < nthreads; t++) {
        int rc = pthread_create(&threads[t], NULL, threadfun, (void *)t); //
        if (rc) {
            printf("ERROR; return code from pthread_create() is %d\n", rc); //
            exit(-1); //
        }
    }
    
    for (t = 0; t < nthreads; t++) {
        pthread_join(threads[t], NULL); //
    }
    
    pthread_exit(NULL); //
}
```

#### Why This Code Is Broken
Because `common++` is not atomic, the compiled code reads the value from memory into a register, increments the register, and writes it back. If Thread 0 and Thread 1 interleave during these steps, they can read the same initial value and write back the same incremented value, causing one of the increments to be lost.

This leads to non-deterministic outputs like the following:
```
Main stack: 7ffee2c6b6b8, common: 10cf95048 (162)
Thread #1 stack: 70000d83bef8 common: 10cf95048 (162)  <-- Thread 1 read 162
Thread #3 stack: 70000d941ef8 common: 10cf95048 (164)  <-- Increments are lost/unordered!
Thread #2 stack: 70000d8beef8 common: 10cf95048 (165)
Thread #0 stack: 70000d7b8ef8 common: 10cf95048 (163)
```
*The output depends entirely on the scheduler's order.*

### 5.3 Correcting the Program Using pthreads Mutexes
To fix this race condition, we must serialize access to the critical section using a mutex:

```c
#include <stdio.h>
#include <stdlib.h>
#include <pthread.h>

int common = 162; //

// Statically initialize the mutex
pthread_mutex_t common_lock = PTHREAD_MUTEX_INITIALIZER; //

void *threadfun(void *threadid) {
    long tid = (long)threadid; //
    
    // Acquire the lock to enter the critical section
    pthread_mutex_lock(&common_lock); //
    
    // CRITICAL SECTION (Protected)
    int my_common = common++; //
    
    // Release the lock immediately after completing the shared access
    pthread_mutex_unlock(&common_lock); //
    
    printf("Thread #%lx stack: %lx common: %lx (%d)\n", 
           tid, (unsigned long)&tid, (unsigned long)&common, my_common); //
           
    pthread_exit(NULL); //
}
```

By wrapping the read and modification of `common` inside `pthread_mutex_lock` and `pthread_mutex_unlock` statements, we ensure that only one thread can execute this block at a time, guaranteeing correct and deterministic behavior.

## Section 6: User-to-Kernel Mode Transitions

Operating systems isolate user applications within a restricted environment (User Mode) and handle privileged operations within a secure boundary (Kernel Mode). 


### 6.1 The Three Transition Vectors
There are three distinct mechanisms that trigger a transfer of control from user space to kernel space:

| Vector | Trigger Type | Conceptual Model | Key Examples |
| :--- | :--- | :--- | :--- |
| **System Call (Syscall)** | Synchronous / Intentional | The process explicitly requests a system service. This is like a function call but operates outside the process's address space. | `fork()`, `exec()`, `write()`, `exit()` |
| **Hardware Interrupt** | Asynchronous / External | An external hardware event triggers a context switch. This occurs independently of the currently running process. | Timer expiration, network packet arrival, disk write completion |
| **Trap or Exception** | Synchronous / Error | An internal processor event or error within the process triggers a context switch. | Segmentation faults, division by zero, page faults |

## Section 7: Implementing Safe Kernel Transfers & Separate Stacks

To prevent user programs from compromising system security or stability, the operating system must strictly enforce safe kernel transitions.

### 7.1 Security Requirements for Kernel Transfers
* **Atomic Mode Switches:** Entering kernel mode must automatically switch the mode bit and jump to a pre-defined kernel instruction handler in a single, atomic step.
* **Indirection Tables:** The CPU must restrict entry points into the kernel. It does this using lookup tables—such as the **System Call Table** and **Interrupt Vector Table**—which map syscall or interrupt IDs to specific kernel functions. This prevents user programs from jumping to arbitrary locations in kernel memory.
* **Preserving Process State:** The kernel handler must save the entire hardware state (registers, program counter, and flags) of the suspended process before starting its own work.

### 7.2 The Two-Stack Model
The kernel **cannot** write to or use the user-space stack for its own execution:
* **Security risk:** The kernel cannot trust the user's stack pointer to point to a valid, aligned memory location. If it did, a malicious user could pass an invalid pointer to trigger a kernel crash.
* **Concurrency risk:** In a multi-threaded application, other running threads could modify data on the user stack while the kernel is using it, leading to TOCTOU (Time-of-Check to Time-of-Use) vulnerabilities.

As a result, each thread has both a **User Stack** (in user space) and a **Kernel Stack** (in protected kernel space).


### 7.3 Hardware Register Transitions During a Transition
When an interrupt or system call occurs, the hardware automatically saves the user process's state onto the thread's **Kernel Stack**:

```
Before Interrupt:                    During Interrupt (Saved onto Kernel Stack):
[ Registers ]                        ┌──────────────────────────────┐
  SS:ESP   ──> User Stack pointer    │             SS               │ (User Stack Segment)
  CS:EIP   ──> User Instruction      ├──────────────────────────────┤
  EFLAGS   ──> CPU flags             │             ESP              │ (User Stack Pointer)
                                     ├──────────────────────────────┤
                                     │           EFLAGS             │ (User CPU flags)
                                     ├──────────────────────────────┤
                                     │             CS               │ (User Code Segment)
                                     ├──────────────────────────────┤
                                     │             EIP              │ (User Instruction Pointer)
                                     ├──────────────────────────────┤
                                     │          Error Code          │ (Optional)
                                     └──────────────────────────────┘
```
*The hardware automatically pushes these registers onto the kernel stack to preserve the user-space context.*

### 7.4 Handling Arguments Safely
* **Locating Arguments:** The kernel locates system call arguments passed in the registers or on the user stack.
* **Copying Arguments:** The kernel copies these arguments from user memory into kernel memory before validation. This protects the kernel from malicious programs trying to alter arguments mid-check.
* **Validating Arguments:** The kernel validates all inputs to prevent errors in user code from causing kernel crashes.
* **Copying Results Back:** After executing the system call, the kernel copies any output values back into user memory.

### 7.5 Interrupt Control (Safe Entry into Kernel)
Similar to system calls, interrupts also vector through a table of handlers and use the kernel interrupt stack. Key properties include:
* **Invisible to User Process:** Interrupt processing occurs between instructions and is restarted transparently, resulting in no change to the process state. (Question: What can be observed even with perfect interrupt processing? Answer: Time elapsed).
* **Interrupts Disabled:** The interrupt handler is invoked with interrupts disabled. They are re-enabled upon completion.
* **Non-blocking Execution:** The handler must run to completion without waiting (non-blocking). 
* **Deferred Work:** For heavy processing, the handler packs the required work into a queue and passes it off to an OS thread for the hard work (by waking up an existing OS thread).

## Section 8: Hardware Interrupt Routing & Controllers

Hardware interrupts are routed from external devices to the CPU through a dedicated hardware component called the **Interrupt Controller**.


### 8.1 Key Hardware Components
* **Interrupt Lines:** Physical wires connected to hardware devices (like network cards, storage controllers, or timers).
* **Software Interrupts:** Special registers that allow the operating system to trigger software-defined interrupt paths.
* **Interrupt Mask:** A bitmask inside the controller that allows the operating system to selectively enable or disable individual interrupt lines.
* **Priority Encoder:** Evaluates all active, enabled interrupt lines and selects the one with the highest priority to forward to the CPU.
* **Global Interrupt Disable Flag:** An internal flag in the CPU that can disable all maskable interrupts.
* **Non-Maskable Interrupt (NMI):** A dedicated hardware line for critical errors (such as memory parity faults or hardware failures) that cannot be disabled by the CPU.

### 8.2 The Interrupt Vector Dispatch Pattern
When an interrupt is accepted by the CPU, the hardware uses the **Interrupt ID** as an index into the **Interrupt Vector Table** to look up the address of the corresponding handler and jump to it.


*This lookup pattern is also used to dispatch system calls and exceptions.*

## Section 9: The Process Lifecycle & Management API

Everything running outside the kernel executes within a process. Processes are managed dynamically through a set of core operating system APIs.

### 9.1 Bootstrapping: The "INIT" Process
Since all processes are created by other processes, how does the very first process start?
* **The First Process:** The kernel starts the first user-space process directly when the system boots.
* **INIT Process:** This process is called `init` or the "system bootstrap" process. All other processes on the system are spawned as descendants of this initial process.

### 9.2 The Process Management API
* `fork()`: Copies the current process to create a new child process.
* `exec()`: Replaces the current process's program image with a new executable.
* `wait()`: Suspends the calling process until one of its child processes terminates.
* `exit()`: Terminates the current process.
* `kill()`: Sends an asynchronous signal to a process.
* `sigaction()`: Registers a custom signal handler function within a process.

## Section 10: Process Management Case Studies & Code Walkthroughs

### 10.1 Case Study 1: The OS Library Exit Wrapper
Consider the following program (`pid.c`):

```c
#include <sys/types.h>
#include <unistd.h>
#include <stdio.h>
#include <stdlib.h>

int main(int argc, char *argv[]) {
    pid_t pid = getpid(); // Retrieve process ID
    printf("My pid: %d\n", pid); //
    exit(0); //
}
```

**Question:** What happens if `main` returns without explicitly calling `exit()`?  
**Answer:** The program still terminates cleanly. This is because standard executables are wrapped by system libraries (like `libc`). The library defines the true entry point of the executable, which calls `main`:

```c
// Conceptual runtime bootstrap code inside standard system library (libc)
void _start() {
    // ... initialize environment ...
    int status = main(argc, argv);
    exit(status); // If main returns, the library calls exit automatically
}
```
*The OS library wraps the main function to guarantee clean termination.*

### 10.2 Case Study 2: Process Creation via `fork()`
The `fork()` system call duplicates the calling process, creating an identical child process.

```c
pid_t cpid = fork(); //
```

* **Return Values:**
  * **In the Parent Process:** `fork()` returns the positive Process ID (PID) of the newly created child process.
  * **In the Child Process:** `fork()` returns `0`.
  * **Error:** If the call fails, `fork()` returns a negative value, and no child process is created.
* **Shared State:** The child process starts as a single-threaded copy of the parent, duplicating its address space (memory), open file descriptors, and register values.

#### Walkthrough of `fork1.c`
This program demonstrates how to use `fork()` to create a child process and run separate code paths in the parent and child:

```c
#include <stdlib.h>
#include <stdio.h>
#include <unistd.h>
#include <sys/types.h>

int main(int argc, char *argv[]) {
    pid_t cpid, mypid;
    pid_t pid = getpid(); //
    
    printf("Parent pid: %d\n", pid); //
    
    cpid = fork(); // Process splits into Parent and Child
    
    if (cpid > 0) {
        /* Parent Process Path */ //
        mypid = getpid(); //
        printf("[%d] parent of [%d]\n", mypid, cpid); //
    } else if (cpid == 0) {
        /* Child Process Path */ //
        mypid = getpid(); //
        printf("[%d] child\n", mypid); //
    } else {
        perror("Fork failed"); //
    }
    return 0;
}
```

**Output Analysis:**
```
Parent pid: 45210
[45210] parent of [45211]
[45211] child
```
*Note: The parent and child processes execute concurrently, so the order of their print statements is non-deterministic and depends on the scheduler.*

### 10.3 Case Study 3: Concurrent Loops (`fork_race.c`)
This program demonstrates how parent and child processes run concurrently as independent execution streams after a `fork()`:

```c
#include <stdio.h>
#include <unistd.h>
#include <sys/types.h>

int main() {
    int i;
    pid_t cpid = fork(); //
    
    if (cpid > 0) {
        for (i = 0; i < 10; i++) {
            printf("Parent: %d\n", i); //
        }
    } else if (cpid == 0) {
        for (i = 0; i > -10; i--) {
            printf("Child: %d\n", i); //
        }
    }
    return 0;
}
```

**Question 1:** What does this print?  
**Answer:** The output will be a non-deterministic mix of "Parent: X" and "Child: Y" print statements. This is because the parent and child are scheduled independently.

**Question 2:** Would adding a call to `sleep(1)` inside each loop change the behavior?  
**Answer:** Yes. Calling `sleep(1)` forces the running process to yield the CPU and enter a blocked state. This allows the scheduler to switch to the other process, resulting in a more evenly interleaved output (typically alternating between parent and child prints).

### 10.4 Case Study 4: Starting New Programs (`execv` & `wait`)
To run a different program inside a child process, we use the `exec()` family of system calls alongside `wait()` to synchronize execution.

#### Process Management Example (`fork2.c`)
This program shows how a parent process forks a child, the child executes a new program (the `ls` command), and the parent waits for the child to finish before exiting:

```c
#include <stdio.h>
#include <stdlib.h>
#include <sys/types.h>
#include <sys/wait.h>
#include <unistd.h>

int main() {
    int status; //
    pid_t tcpid; //
    pid_t cpid = fork(); //
    
    if (cpid > 0) {
        /* Parent Process Path */ //
        pid_t mypid = getpid(); //
        printf("[%d] parent of [%d]\n", mypid, cpid); //
        
        // Wait for the child process to terminate and collect its exit status
        tcpid = wait(&status); //
        printf("[%d] bye %d (%d)\n", mypid, tcpid, WEXITSTATUS(status)); //
        
    } else if (cpid == 0) {
        /* Child Process Path */ //
        pid_t mypid = getpid(); //
        printf("[%d] child\n", mypid); //
        
        // Prepare arguments for the 'ls' command
        char *args[] = {"ls", "-l", NULL}; //
        
        // Replace the child's program image with '/bin/ls'
        execv("/bin/ls", args); //
        
        // execv only returns if the call fails
        perror("execv failed"); //
        exit(1); //
    }
    return 0;
}
```

#### Why `execv` Does Not Return
On success, `execv()` replaces the process's address space, code, and stack with the new program. The original code after the `execv` call is completely gone, so it only executes if the system call fails (e.g., if the file path is incorrect).

## Section 11: The Shell Pattern (Fork-Exec-Wait)

Command-line shells (such as Bash) use a fundamental process management pattern called the **Fork-Exec-Wait** loop.


### Steps of the Shell Loop
1. **Fork:** The shell forks a copy of itself, creating a child process.
2. **Exec:** The child process calls `exec()` to replace its program image with the command entered by the user (e.g., `ls` or `grep`).
3. **Wait:** The parent shell process calls `wait()` to block and yield the CPU until the child process finishes executing.
4. **Resume:** Once the child process terminates, the `wait()` call returns, and the parent shell resumes to prompt the user for the next command.

## Section 12: Asynchronous Signals & Signal Handling

Signals are asynchronous notifications sent by the kernel to a user-space process. They behave like software interrupts, preempting the application's normal execution to run a signal handler.

### 12.1 Registering Custom Signal Handlers
We can use the `sigaction` API to catch specific signals and run custom handler functions:

```c
#include <stdio.h>
#include <stdlib.h>
#include <sys/types.h>
#include <unistd.h>
#include <signal.h>

// Custom signal handler callback function
void signal_callback_handler(int signum) { //
    printf("Caught signal %d!\n", signum); //
    exit(1); // Terminate process safely
}

int main() {
    struct sigaction sa; //
    sa.sa_flags = 0; //
    
    // Initialize the signal mask to block other signals during execution
    sigemptyset(&sa.sa_mask); //
    
    // Assign our custom callback function to the handler
    sa.sa_handler = signal_callback_handler; //
    
    // Register the handler for SIGINT (Control-C)
    sigaction(SIGINT, &sa, NULL); //
    
    while (1) {
        // Infinite loop waiting for a signal
    }
    return 0;
}
```

**Question:** What happens if a process receives a `SIGINT` (Control-C) but has not registered a custom signal handler?  
**Answer:** The process is terminated immediately. The operating system defines default handler behaviors for all signals, and the default action for `SIGINT` is termination.

### 12.2 Common POSIX Signals
* `SIGINT`: Sent when the user presses Control-C. Its default action is process termination.
* `SIGTERM`: The standard termination signal sent by administrative tools (like the `kill` command) to request a clean exit.
* `SIGTSTP`: Sent when the user presses Control-Z. Its default action is to suspend (stop) process execution.
* `SIGKILL`: Forcefully terminates a process.
* `SIGSTOP`: Forcefully suspends a process.

### 12.3 Uncatchable Signals: SIGKILL and SIGSTOP
The `SIGKILL` and `SIGSTOP` signals **cannot** be caught, blocked, or ignored using `sigaction`.

#### The Rationale
If processes could catch or block `SIGKILL` and `SIGSTOP`, buggy or malicious programs could run forever without any way for system administrators to stop them. Restricting these signals ensures that administrators can always forcefully terminate or suspend runaway processes, maintaining control over system resources.

## Section 13: High-Yield Exam Concept Review

Verify your understanding of these core concepts before the exam:

### 1. Concurrency vs. Parallelism
* Concurrency is about logical execution overlap (handling multiple tasks at once). Parallelism is about physical execution overlap (running multiple tasks simultaneously on different cores).

### 2. Thread Stack Layout
* While threads share heap and global data, they must have private execution stacks to maintain their own function call contexts. Overflows are caught using protected virtual memory zones called **Guard Pages**.

### 3. Mutual Exclusion
* Protecting shared data structures requires mutual exclusion. POSIX mutexes (`pthread_mutex_lock`/`unlock`) serialize access to critical sections, eliminating non-deterministic race conditions.

### 4. Separate Kernel Stacks
* The kernel must use a separate, secure kernel stack for each thread because it cannot trust user-space stack pointers. On an interrupt, the hardware automatically saves the user state (SS, ESP, EFLAGS, CS, EIP) onto the kernel stack.

### 5. Process Lifecycle APIs
* `fork()` copies the calling process and returns `0` in the child and the child's PID in the parent. `exec()` replaces the process's code image and never returns on success. `wait()` suspends the parent process until the child terminates.
