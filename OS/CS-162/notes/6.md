## Module 1: High-Level vs. Low-Level I/O Abstractions & Kernel Buffering

Operating systems bridge the gap between user-level application logic and physical hardware devices. This is managed through a layered system architecture, isolating applications from hardware details.


### 1.1 Core API Execution Models
Systems programmers distinguish between the direct, raw system call interface and standard user-space buffering libraries.

#### Low-Level File API (`read`, `write`, `open`, `close`)
* **Interface Mechanics:** Operates directly on raw integer **File Descriptors (FD)**.
* **Kernel Ownership:** File descriptors are integer indices into a per-process **File Descriptor Table** (also called a Process Descriptor Table). This table contains pointers to a system-wide kernel-level structure called the **Open File Description**.
* **Performance Profile:** Every single low-level read or write call forces a system call trap into the kernel, regardless of the size of the request (even a single byte). If an application reads a file 4 bytes at a time in a loop using `read()`, each iteration incurs the performance overhead of a full kernel context switch.

#### High-Level File API (`fopen`, `fread`, `fwrite`, `fclose`)
* **Interface Mechanics:** Exposes a pointer to a **`FILE` structure** allocated in user space.
* **User-Space Buffering:** The `FILE` structure contains an internal, user-space chunk of memory acting as a buffer. It also wraps the underlying low-level raw integer file descriptor.
* **Performance Profile:** Library functions filter requests through this user-space buffer. For example, a loop reading 4 bytes at a time via `fread()` will call the underlying `read()` system call on its first iteration to fetch a much larger block (e.g., 1024 bytes) from the kernel. This data is loaded into the user-space buffer, and the first 4 bytes are returned to the user. Subsequent `fread()` iterations pull data directly from this buffer without invoking any system calls, only trapping into the kernel again once the buffer runs dry.

### 1.2 Under the Hood: Low-Level vs. High-Level API Calls
The following assembly-level and kernel-transition walkthrough highlights the path of both calls:

```
Low-Level API: read(fd, buf, 4)
User Space                                     Kernel Space
┌─────────────────────────────────┐            ┌──────────────────────────────────┐
│ 1. Move args into registers     │            │ 4. Extract args from registers   │
│ 2. Set EAX = 0x03 (sys_read)    │            │ 5. Dispatch to sys_read handler  │
│ 3. Execute trap/syscall instr   │ ─────────> │ 6. Access device / File System   │
│                                 │            │ 7. Write result to EAX           │
│ 9. Read data from %EAX & return │ <───────── │ 8. Return from syscall/interrupt │
└─────────────────────────────────┘            └──────────────────────────────────┘

High-Level API: fread(buf, 4, 1, stream)
User Space
┌────────────────────────────────────────────────────────────────────────┐
│ 1. Inspect internal stream buffer for requested 4 bytes                │
│ 2. IF data is present:                                                 │
│    - Copy 4 bytes directly to buf and return immediately (No Syscall!) │
│ 3. IF data is absent:                                                  │
│    - Execute trap/syscall to read a large block of data (e.g. 1024B)   │
│    - Populate the user-space stream buffer                             │
│    - Copy the first 4 bytes to buf and return to caller                │
└────────────────────────────────────────────────────────────────────────┘
```

## Module 2: Kernel Internals: Open File Descriptions and `vfs_read` Code Analysis

Every active process maintains an isolated view of files, which is translated by the kernel into concrete data layout records.


### 2.1 The Open File Description Structure
Within the kernel (as defined in `linux/fs.h`), each open file is represented by a `struct file` object. While this structure contains many technical parameters, its primary roles are tracking:
1. **Physical Location:** Where to find the file data on disk (via the `f_path` and `f_inode` parameters).
2. **File Pointer Position:** The current read/write offset position within the file (`f_pos`).

Key structural elements of `struct file` in the Linux kernel include:
* `f_path`: The file path representation.
* `f_inode`: Pointer to the unique kernel inode structure containing physical metadata.
* `f_op`: Pointer to a table of device-specific file operations (`struct file_operations`).
* `f_count`: The reference count of file descriptors pointing to this description.
* `f_flags`: Access and open status flags (e.g., `O_RDONLY`, `O_CREAT`, `O_NONBLOCK`).
* `f_mode`: The open mode permissions (e.g., `FMODE_READ`, `FMODE_WRITE`).
* `f_pos`: The current byte offset for read/write tracking.

### 2.2 Deep Dive: Kernel Source Code Walkthrough of `vfs_read`
The `vfs_read` function inside `fs/read_write.c` is the kernel's entry point for handling read operations. It validates arguments, checks permissions, and dispatches requests to device-specific file system drivers.

```c
ssize_t vfs_read(struct file *file, char __user *buf, size_t count, loff_t *pos)
{
    ssize_t ret;

    // 1. Validate that the file description was opened with read permissions
    if (!(file->f_mode & FMODE_READ)) 
        return -EBADF; // Bad File Descriptor error

    // 2. Validate that the file operations and read function pointers are populated
    if (!file->f_op || (!file->f_op->read && !file->f_op->aio_read)) 
        return -EINVAL; // Invalid argument error

    // 3. Verify that the destination user-space buffer is writable by the user process
    // unlikely() is a compiler optimization hint indicating this path is rarely taken
    if (unlikely(!access_ok(VERIFY_WRITE, buf, count))) 
        return -EFAULT; // Bad address fault

    // 4. Verify that the requested file range is valid and within security boundaries
    ret = rw_verify_area(READ, file, pos, count); //
    if (ret >= 0) {
        count = ret;

        // 5. Polymorphic Dispatch: If the driver provides a read method, use it;
        // otherwise, fall back to do_sync_read()
        if (file->f_op->read)
            ret = file->f_op->read(file, buf, count, pos); //
        else
            ret = do_sync_read(file, buf, count, pos); //

        // 6. Post-execution bookkeeping and event notifications
        if (ret > 0) {
            // Notify parent directory that this file was accessed (for monitoring)
            fsnotify_access(file->f_path.dentry); //

            // Update accounting metrics: bytes read by the current task
            add_rchar(current, ret); //
        }

        // Update accounting metrics: count of total read system calls by current task
        inc_syscr(current); //
    }

    return ret; // Return actual bytes read or negative error code
}
```

## Module 3: Device Drivers: Top Half vs. Bottom Half and I/O Lifecycles

### 3.1 Device Driver Concept
A **Device Driver** is device-specific code in the kernel that interacts directly with physical device controllers. Drivers register with the kernel to expose a standard, uniform interface to the virtual file system, allowing the same generic kernel I/O system to interact with completely different physical hardware. Special device configurations are handled via the polymorphic `ioctl()` system call.

### 3.2 Split-Half Design
To keep system calls responsive and handle asynchronous hardware, modern device drivers are split into two halves:

```
┌────────────────────────────────────────────────────────────────────────┐
│                          TOP HALF (Synchronous)                        │
│ - Runs in the execution path of system calls (process context).        │
│ - Implements standard file operations (open, read, write, ioctl).     │
│ - Initiates device operations and can put the calling thread to sleep. │
└────────────────────────────────────────────────────────────────────────┘
                                    │
                         Asynchronous Hardware I/O
                                    │
                                    ▼
┌────────────────────────────────────────────────────────────────────────┐
│                        BOTTOM HALF (Asynchronous)                      │
│ - Runs as an interrupt service routine (ISR) in interrupt context.     │
│ - Cannot block, sleep, or wait on resources.                           │
│ - Handles incoming data blocks or output completions from hardware.    │
│ - Wakes up sleeping threads waiting in the top half once I/O is done.  │
└────────────────────────────────────────────────────────────────────────┘
```

### 3.3 The File Operations Interface (`struct file_operations`)
Device driver methods are registered with the kernel using the `file_operations` struct:
```c
struct file_operations {
    struct module *owner;
    loff_t (*llseek) (struct file *, loff_t, int);
    ssize_t (*read) (struct file *, char __user *, size_t, loff_t *); //
    ssize_t (*write) (struct file *, const char __user *, size_t, loff_t *); //
    ssize_t (*aio_read) (struct kiocb *, const struct iovec *, unsigned long, loff_t); //
    ssize_t (*aio_write) (struct kiocb *, const struct iovec *, unsigned long, loff_t); //
    int (*readdir) (struct file *, void *, filldir_t); //
    unsigned int (*poll) (struct file *, struct poll_table_struct *); //
    int (*ioctl) (struct inode *, struct file *, unsigned int, unsigned long); //
    int (*mmap) (struct file *, struct vm_area_struct *); //
    int (*open) (struct inode *, struct file *); //
    int (*flush) (struct file *, fl_owner_t id); //
    int (*release) (struct inode *, struct file *); //
    int (*fsync) (struct file *, struct dentry *, int datasync); //
    int (*fasync) (int, struct file *, int); //
    int (*flock) (struct file *, int, struct file_lock *); //
};
```

### 3.4 Lifecycle of an I/O Request
When a user program reads or writes to a device, the request transitions through several layers of the operating system.


The lifecycle of an I/O request follows this path:
1. **Initiation:** The user program invokes a standard system call (such as `read()`), trapping into the **Kernel I/O Subsystem**.
2. **Subsystem Verification:** The kernel checks if the request can be satisfied immediately (e.g., using data cached in kernel memory).
   * *If Yes:* The data is copied to the process, and the system call returns immediately.
   * *If No:* The request is forwarded to the **Device Driver Top Half**.
3. **Driver Initiation:** The driver's top half translates the request into device-controller commands and configures the hardware controller.
4. **Process Suspension:** If the operation is blocking, the driver puts the calling thread to sleep to yield the CPU.
5. **Hardware Monitoring:** The physical **Device Controller** monitors the hardware and executes the read/write operation.
6. **Hardware Completion:** Once the hardware operation completes, the device controller asserts a physical interrupt signal line to the CPU.
7. **Interrupt Service (Bottom Half):** The **Device Driver Bottom Half** handles the interrupt. It retrieves the data from the device, stores it in a kernel-space buffer, and signals to unblock the sleeping process.
8. **Top Half Return:** The driver's top half wakes up, determines that the operation completed successfully, and passes status info to the I/O subsystem.
9. **Finalization:** The subsystem copies the data from kernel-space buffers into the user program's memory and returns control to the user process.

## Module 4: Inter-Process Communication (IPC) & The Socket Abstraction

### 4.1 Process Isolation vs. Communication
By default, processes run inside isolated, protected virtual address spaces. While this protection prevents bugs or crashes in one application from affecting other programs, it also means that communication between processes must be configured explicitly through operating system channels.

```
                     [ IPC PERSISTENCE COMPARISON ]
                                    │
    ┌───────────────────────────────┴───────────────────────────────┐
    ▼                                                               ▼
[ Disk-Backed Files (Local Only) ]               [ Transient Queues (Pipes/Sockets) ]
• Producer/Consumer separated in time.           • Data is produced, consumed once, and discarded.
• Requires explicit system call polling.         • Implements blocking producer-consumer queues.
• High physical storage overhead.                • Fast, lightweight, and memory-backed.
```

### 4.2 The Socket Interface
A **Socket** is a software abstraction representing an endpoint of a network or local connection. 


* **Unified Abstraction:** Sockets allow communication across the network to look like standard file I/O. Sockets are represented as file descriptors, allowing applications to use standard read and write system calls.
* **Historical Origins:** Standardized by POSIX, the socket interface was originally developed and introduced in **4.2 BSD (Berkeley Standard Distribution) UNIX**.
* **Network Independence:** Sockets expose the same abstraction regardless of the underlying network layer, whether local (Unix Domain Sockets) or external (TCP/IP, UDP/IP).
* **Queue Abstraction:** Sockets model bidirectional communication using two queues:
  * `write()` adds data to the local output queue (destined for the peer).
  * `read()` removes data from the local input queue (sent by the peer).
* **VFS Differences:** Unlike files, sockets do not support positioning operations; calling `lseek()` on a socket descriptor is invalid and returns an error.

## Module 5: Sockets Deep-Dive: Namespaces, 5-Tuples, and Connection Setup

### 5.1 Communication Abstractions Comparison
Operating systems provide several abstractions for I/O and communication:

| Dimension | Files | Pipes | Sockets |
| :--- | :--- | :--- | :--- |
| **Lifetime** | Permanent, exist independently of process lifecycles. | Transient, cleaned up when the process terminates. | Transient, cleaned up when closed. |
| **Name Space** | Structured Directory Tree. | No formal namespace; represented as anonymous descriptors. | IP addresses and Port numbers. |
| **Setup Mechanics** | Opened via path strings. | Created via `pipe()`; descriptors are inherited by child processes. | Configured via hostnames, IP addresses, and ports. |
| **Topology** | Many-to-Many shared storage. | One-way communication queue spanning related processes. | Bidirectional communication queues spanning arbitrary machines. |

### 5.2 IP Namespaces
To establish a network connection, applications must identify the target machine and service using the IP namespace:
* **Hostname:** Human-readable domain name (e.g., `www.eecs.berkeley.edu`).
* **IP Address:** Unique network identifier:
  * *IPv4:* 32-bit integer (e.g., `128.32.244.172`).
  * *IPv6:* 128-bit integer (e.g., `2607:f140:0:81::f`).
* **Port Numbers:** 16-bit integers (0 to 65535) that route network packets to specific application processes:
  * *System / Well-Known Ports (0-1023):* Reserved for standard system services (e.g., HTTP is port 80, HTTPS is port 443). Binding to these ports requires superuser privileges.
  * *Registered Ports (1024-49151):* Assigned by IANA for specific user-space services.
  * *Dynamic / Private / Ephemeral Ports (49152-65535):* Automatically allocated by the operating system for client connections.

### 5.3 TCP/IP Connections and the 5-Tuple
On the Internet, every active TCP connection is uniquely identified by a **5-Tuple**:
$$\{\text{Source IP}, \text{Destination IP}, \text{Source Port}, \text{Destination Port}, \text{Protocol}\}$$

This 5-Tuple allows a server to multiplex multiple client connections on the same port. For example, a web server running on port 80 can handle thousands of simultaneous client connections because each connection has a unique client IP or ephemeral port.

### 5.4 Server Socket Mechanics
To accept incoming connections, servers use a specialized **Server Socket**:
* **Interface Restrictions:** Although represented as a file descriptor, a server socket cannot be used to read or write data directly.
* **Core Operations:**
  1. `listen()`: Configures the server socket to listen for incoming connection requests.
  2. `accept()`: Pops an incoming request off the connection queue, creates a new connection socket for that client, and returns its file descriptor.

## Module 6: Socket Programming in C

### 6.1 Complete Code Example: The Echo Client
This program establishes a connection to a remote server, prompts the user for text, sends it via a TCP socket, and prints the echoed response.

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <sys/types.h>
#include <sys/socket.h>
#include <netdb.h>

#define MAXIN 1024
#define MAXOUT 1024

// Resolve hostnames to IP addresses
struct addrinfo *lookup_host(char *host_name, char *port_name) {
    struct addrinfo *server;
    struct addrinfo hints;

    memset(&hints, 0, sizeof(hints));
    hints.ai_family = AF_UNSPEC;     // Support both IPv4 and IPv6
    hints.ai_socktype = SOCK_STREAM; // Establish TCP byte streams

    int rv = getaddrinfo(host_name, port_name, &hints, &server); //
    if (rv != 0) {
        fprintf(stderr, "getaddrinfo failed: %s\n", gai_strerror(rv)); //
        return NULL;
    }
    return server;
}

void run_client(int sockfd) {
    char sndbuf[MAXIN];
    char rcvbuf[MAXOUT];
    int n;

    while (1) {
        printf("Enter string: ");
        if (fgets(sndbuf, MAXIN, stdin) == NULL) break; //

        // Send string (including null terminator) to server
        write(sockfd, sndbuf, strlen(sndbuf) + 1); //

        memset(rcvbuf, 0, MAXOUT); //
        
        // Block until echoed data is received
        n = read(sockfd, rcvbuf, MAXOUT); //
        if (n <= 0) {
            printf("Connection closed by server.\n");
            break;
        }

        // Print response to standard output
        write(STDOUT_FILENO, rcvbuf, n); //
    }
}

int main(int argc, char *argv[]) {
    if (argc < 3) {
        fprintf(stderr, "Usage: %s <host> <port>\n", argv[0]);
        return 1;
    }

    char *host_name = argv[1];
    char *port_name = argv[2];

    struct addrinfo *server = lookup_host(host_name, port_name); //
    if (server == NULL) return 1;

    // 1. Create client socket descriptor
    int sock_fd = socket(server->ai_family, server->ai_socktype, server->ai_protocol); //
    if (sock_fd < 0) {
        perror("Socket creation failed");
        freeaddrinfo(server);
        return 1;
    }

    // 2. Connect to server
    if (connect(sock_fd, server->ai_addr, server->ai_addrlen) < 0) { //
        perror("Connect failed");
        close(sock_fd);
        freeaddrinfo(server);
        return 1;
    }

    run_client(sock_fd); // Execute application logic

    close(sock_fd); //
    freeaddrinfo(server);
    return 0;
}
```

### 6.2 Stream Framing Considerations
Because TCP is stream-oriented, it does not preserve application message boundaries. 

If a client makes two sequential calls to `write()` to send strings "A" and "B", the operating system may package them into a single TCP segment. As a result, a single call to `read()` on the server side might retrieve both strings "AB" at once, or retrieve a partial chunk of "A".

To handle this, application protocols must implement **framing**:
* **Delimiter-based:** Scan incoming bytes sequentially for a terminator character (such as a null byte `\0` or a newline `\n`) to identify message boundaries.
* **Length-prefixed:** Prefix each message with its byte length, allowing the reader to read the exact length before parsing the message.

## Module 7: Concurrent Server Architectures

To handle multiple clients concurrently, servers use different execution models to keep blocking operations on one connection from stalling others.

### 7.1 Web Server: Iterative Model (Single Process, No Protection)
This model handles connections sequentially.

```c
//
struct addrinfo *setup_address(char *port) {
    struct addrinfo *server;
    struct addrinfo hints;
    memset(&hints, 0, sizeof(hints));
    hints.ai_family = AF_UNSPEC;
    hints.ai_socktype = SOCK_STREAM;
    hints.ai_flags = AI_PASSIVE; // Accept connections on any local IP address

    getaddrinfo(NULL, port, &hints, &server); //
    return server;
}

int main(int argc, char *argv[]) {
    char *port_name = "8080";
    struct addrinfo *server = setup_address(port_name); //

    int server_socket = socket(server->ai_family, server->ai_socktype, server->ai_protocol); //
    bind(server_socket, server->ai_addr, server->ai_addrlen); //
    listen(server_socket, 128); //

    while (1) {
        // Block until a client connects
        int conn_socket = accept(server_socket, NULL, NULL); //
        
        // Serve client sequentially; blocks other incoming requests
        serve_client(conn_socket); //
        close(conn_socket); //
    }
    close(server_socket);
    return 0;
}
```

### 7.2 Web Server: Multi-Process Model (With Protection, No Concurrency)
This model sandboxes client requests by handling each connection in a separate process. However, calling a blocking `wait()` in the main loop means the server still handles requests sequentially.

```c
    while (1) {
        int conn_socket = accept(server_socket, NULL, NULL); //
        pid_t pid = fork(); // Duplicate parent process

        if (pid == 0) {
            /* Child Process Context */
            close(server_socket);       // Close the inherited listening socket
            serve_client(conn_socket);  // Service client request in sandbox
            close(conn_socket);         // Close child's reference to connection socket
            exit(0);                    // Terminate child process
        } else {
            /* Parent Process Context */
            close(conn_socket);         // Close parent's reference to connection socket
            wait(NULL);                 // Blocks accept loop until child terminates (Iterative execution)
        }
    }
```

### 7.3 Web Server: Concurrent Multi-Process Model (With Protection & Concurrency)
To handle clients concurrently, the parent process skips the blocking `wait()` inside the loop. This allows the parent to immediately accept the next client while the child handles the previous one.

```c
    while (1) {
        int conn_socket = accept(server_socket, NULL, NULL); //
        pid_t pid = fork(); //

        if (pid == 0) {
            /* Child Process Context */
            close(server_socket); //
            serve_client(conn_socket); //
            close(conn_socket); //
            exit(0); //
        } else {
            /* Parent Process Context */
            close(conn_socket); //
            // wait(NULL); is commented out to prevent blocking the accept loop
        }
    }
```
*Note: To clean up child processes without blocking, the parent must register a non-blocking `SIGCHLD` signal handler to call `waitpid(-1, NULL, WNOHANG)` whenever a child terminates.*

### 7.4 Web Server: Concurrent Multi-Threaded Model (No Protection, Concurrency)
Spawning processes incurs significant memory and CPU overhead. To improve performance, the server can handle connections in separate threads using `pthread_create()`.

```c
void *thread_worker(void *arg) {
    int conn_socket = (int)(long)arg;
    pthread_detach(pthread_self()); // Automatically reclaim resources on thread exit
    serve_client(conn_socket);
    close(conn_socket);
    pthread_exit(NULL);
}

int main(void) {
    // Socket initialization elided...
    while (1) {
        int conn_socket = accept(server_socket, NULL, NULL);
        pthread_t tid;
        // Spawn thread to handle client; main loop returns immediately to accept
        pthread_create(&tid, NULL, thread_worker, (void *)(long)conn_socket); //
    }
}
```
* **Performance Profile:** Fast connection handling with low memory overhead.
* **Security Risk:** No protection isolation. A bug or crash in one client handler thread can corrupt shared memory or crash the entire server process.

---

### 7.5 Bounded Concurrency: The Thread Pool Model
While the concurrent multi-threaded model is fast, spawning unbounded threads under high traffic can overwhelm the system, causing memory exhaustion and thread thrashing. 

To prevent this, servers use the **Thread Pool** pattern, allocating a fixed number of worker threads to handle connections.


```
                       [ Incoming Connections ]
                                  │
                                  ▼
┌──────────────────────────────────────────────────────────────────┐
│                          MASTER THREAD                           │
│ - Accepts connections from clients sequentially.                 │
│ - Places connection handles on a shared queue.                   │
│ - Wakes up sleeping workers in the pool.  │
└──────────────────────────────────────────────────────────────────┘
                                  │
                         Shared FIFO Queue
                                  │
                                  ▼
┌──────────────────────────────────────────────────────────────────┐
│                        WORKER THREAD POOL                        │
│ - Fixed size pool of worker threads.                             │
│ - Worker threads wait on the queue if empty.                     │
│ - Dequeue connections to service requests.│
└──────────────────────────────────────────────────────────────────┘
```

#### Thread Pool Execution Algorithm
```c
// Master Thread Algorithm
master() {
    allocThreads(worker, queue); // Instantiate bounded worker pool
    while (TRUE) {
        con = AcceptCon();       // Wait for client connection
        Enqueue(queue, con);     // Place connection handle on shared queue
        wakeUp(queue);           // Signal sleeping worker threads
    }
}

// Worker Thread Algorithm
worker(queue) {
    while (TRUE) {
        con = Dequeue(queue);    // Attempt to pop connection off queue
        if (con == null)
            sleepOn(queue);      // Block if queue is empty
        else
            ServiceWebPage(con); // Process I/O request
    }
}
```

### Problem 1: File Descriptor Duplication and Reference Counting
A server process executes the following code block:
```c
int fd = open("output.txt", O_RDWR); // Returns File Descriptor index 3
pid_t pid = fork();
if (pid == 0) {
    dup2(fd, STDOUT_FILENO); // Duplicate fd to stdout (FD index 1)
    close(fd);
    exit(0);
} else {
    close(fd);
    wait(NULL);
}
```

1. Sketch the file descriptor tables for both the parent and child processes immediately after the `fork()` call, showing how they map to kernel structures.
2. Calculate the reference count (`f_count`) of the kernel's **Open File Description** at each of the following points in execution:
   * Immediately before the `fork()` call.
   * Immediately after the `fork()` call, but before `dup2()`.
   * After the child process executes `dup2()` and `close(fd)`.
   * After the child process terminates and the parent executes `close(fd)`.

#### Solution:
1. **File Descriptor Table Mapping:**
   Immediately after the `fork()` call, both processes have descriptor tables mapping files to the same shared Open File Description in the kernel:
   ```
   Parent FD Table:                        Kernel Open File Description:
   [ 0 ] (stdin)  ───> Terminal            ┌────────────────────────────────┐
   [ 1 ] (stdout) ───> Terminal            │ Struct File:                   │
   [ 2 ] (stderr) ───> Terminal            │ - Offset: 0                    │ <──┐
   [ 3 ] (fd)     ────────────────────────> │ - Inode: output.txt            │    │
                                           └────────────────────────────────┘    │
   Child FD Table (Duplicated):                                                  │
   [ 0 ] (stdin)  ───> Terminal                                                  │
   [ 1 ] (stdout) ───> Terminal                                                  │
   [ 2 ] (stderr) ───> Terminal                                                  │
   [ 3 ] (fd)     ───────────────────────────────────────────────────────────────┘
   ```

2. **Reference Count (`f_count`) Calculations:**
   * **Before `fork()`:** Only the parent process has descriptor `3` pointing to the file description.
     $$\text{f\_count} = 1$$
   * **After `fork()`:** The child process duplicates the parent's descriptor table, so both the parent's FD `3` and the child's FD `3` point to the same file description.
     $$\text{f\_count} = 2$$
   * **After child `dup2()` and `close()`:**
     * `dup2(3, 1)` duplicates FD `3` to FD `1` inside the child process. Both FD `3` and FD `1` in the child now point to the description, incrementing `f_count` to $3$.
     * `close(fd)` (equivalent to `close(3)`) removes the child's FD `3` reference, decrementing `f_count` to $2$.
     * The file description is still referenced by FD `1` in the child and FD `3` in the parent.
     $$\text{f\_count} = 2$$
   * **After child exit and parent `close()`:**
     * When the child process terminates, the operating system automatically closes its remaining open descriptors, removing FD `1`'s reference and decrementing `f_count` to $1$.
     * When the parent process executes `close(fd)` (equivalent to `close(3)`), the parent's reference is removed.
     * With `f_count` reaching $0$, the kernel deallocates the Open File Description and closes the file.
     $$\text{f\_count} = 0$$

### Problem 2: Sockets and 5-Tuple Demultiplexing
A web server is bound to wildcard IP address `0.0.0.0` on port `80`. The server currently has three active client connections.

The active sockets are identified by the following parameters:
* **Server Listening Socket:** File Descriptor `3` (Bound to `*:80`).
* **Connection Sockets:**
  * Socket A: File Descriptor `4`
  * Socket B: File Descriptor `5`
  * Socket C: File Descriptor `6`

The server's network card receives three TCP segments with the following headers:
1. `Segment 1: [Src: 198.51.100.12:45001, Dst: 128.32.10.5:80, Protocol: TCP]`
2. `Segment 2: [Src: 198.51.100.12:45002, Dst: 128.32.10.5:80, Protocol: TCP]`
3. `Segment 3: [Src: 203.0.113.84:53211, Dst: 128.32.10.5:80, Protocol: TCP]`

The kernel's socket table maps connections to file descriptors as follows:
```
{198.51.100.12:45001, 128.32.10.5:80, TCP}  ──> File Descriptor 4
{198.51.100.12:45002, 128.32.10.5:80, TCP}  ──> File Descriptor 5
{203.0.113.84:53211,  128.32.10.5:80, TCP}  ──> File Descriptor 6
```

1. Route each of the incoming segments to its corresponding socket descriptor.
2. If a new client sends a connection request (`SYN` packet) with headers `[Src: 192.0.2.55:36524, Dst: 128.32.10.5:80, Protocol: TCP]`, which file descriptor receives this request? What system call must the server execute to retrieve the new connection socket?

#### Solution:
1. **Routing Inbound Segments:**
   The kernel maps incoming segments to the appropriate file descriptors using their 5-Tuples:
   * **Segment 1** matches the 5-tuple for Socket A: routes to **File Descriptor `4`**.
   * **Segment 2** matches the 5-tuple for Socket B: routes to **File Descriptor `5`**.
   * **Segment 3** matches the 5-tuple for Socket C: routes to **File Descriptor `6`**.

2. **Handling New Connection Requests:**
   * **Target File Descriptor:** The `SYN` connection request is routed to the server's listening socket, **File Descriptor `3`**.
   * **System Call:** To accept the connection, retrieve the client details, and generate a new connection socket, the server process must execute the **`accept()`** system call.
