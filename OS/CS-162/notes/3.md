## Section 2: The Thread Abstraction

### 2.1 Programmer’s View: Threads as Virtual Cores
To a programmer, a thread is an abstraction of a **virtual CPU core (vCPU)**. 


If you have a single physical processor, the operating system can run multiple threads by **multiplexing them in time**. The OS runs one thread for a short period, saves its state, loads another thread's state, and resumes execution. This switch happens so quickly (on the scale of microseconds) that it creates the illusion that multiple processors are executing concurrently.

### 2.2 What Comprises a Thread?
A thread is the smallest schedulable unit of execution in an operating system. A thread is considered to be "executing" on a physical processor when its execution context is loaded into the processor's physical registers. 

Each virtual core (thread) maintains its own private execution state, which includes:
* **Program Counter (PC):** Points to the memory address of the next instruction the thread will execute.
* **Stack Pointer (SP):** Points to the top of the thread's private activation stack, which manages local variables and function calls.
* **Registers:** Holds active data values, including general-purpose, floating-point, and status flags.

### 2.3 Where Does a Thread Reside?
A thread's execution state exists in one of two physical locations:
1. **On the Physical Core:** The thread's state is loaded directly into the CPU's registers, program counter, and stack pointer.
2. **In Memory (Thread Control Block - TCB):** When a thread is not currently running on a physical processor, the operating system preserves its state in a dedicated kernel memory structure called the **Thread Control Block (TCB)**.

### 2.4 Motivation for Threads (Compute/I/O Overlap & Latency Masking)
While threads allow for concurrent execution of tasks, one of the most practical motivations for using them is to **mask I/O latency** and overlap compute with I/O operations (as highlighted by Jeff Dean's "Numbers Everyone Should Know").

Consider a simple, sequential program:
```c
main() {
    ComputePI("pi.txt");
    PrintClassList("classlist.txt");
}
```
In this scenario, the program would never print the class list because `ComputePI` would run indefinitely.

By utilizing threads, we can perform long-running or blocking operations in the background:
```c
main() {
    create_thread(ReadLargeFile, "pi.txt");
    create_thread(RenderUserInterface);
}
```
This multi-threaded version can still respond to user input and render the user interface while reading a large file in the background.

> **Key Insight:** A thread is typically in one of three states:
> 1. **RUNNING:** Currently executing on the processor.
> 2. **READY:** Eligible to run, but not currently running.
> 3. **BLOCKED:** Ineligible to run (e.g., waiting for an I/O operation to complete).
> 
> If a thread performs a blocking I/O operation, the OS marks it as BLOCKED. The scheduler then switches to another READY thread, keeping the CPU utilized. Once the I/O finally finishes, the OS marks the original thread as READY again.

## Section 3: The Process Abstraction

### 3.1 What is a Process?
A process is an **instance of a running program**. It represents a protected execution environment with **restricted rights** managed by the operating system kernel. 


A process contains:
* **A Protected Address Space:** A private range of memory addresses that the process can read or write.
* **One or More Threads:** The active units of execution running within that address space.
* **Associated OS Resources:** Private system handles allocated to the process, such as open file descriptors, network socket references, and user identity credentials.

### 3.2 Multi-Threaded Process Structure
In a multi-threaded process, the threads **share** most of the process's resources but retain some **private** state:

```
┌────────────────────────────────────────────────────────┐
│                      PROCESS A                         │
│                                                        │
│  Shared Resources:                                     │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │     Code     │  │  Global Data │  │ Open Files   │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
│                                                        │
│  Per-Thread State:                                     │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │   Thread 1   │  │   Thread 2   │  │   Thread 3   │  │
│  │ ┌──────────┐ │  │ ┌──────────┐ │  │ ┌──────────┐ │  │
│  │ │Registers │ │  │ │Registers │ │  │ │Registers │ │  │
│  │ └──────────┘ │  │ └──────────┘ │  │ └──────────┘ │  │
│  │ ┌──────────┐ │  │ ┌──────────┐ │  │ ┌──────────┐ │  │
│  │ │  Stack   │ │  │ │  Stack   │ │  │ │  Stack   │ │  │
│  │ └──────────┘ │  │ └──────────┘ │  │ └──────────┘ │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
└────────────────────────────────────────────────────────┘
```

* **Shared State:** Code segments, static global data, the heap, and system assets (like file descriptors and network connections) are shared by all threads within the process.
* **Per-Thread Private State:** Each thread has its own private set of register values (including the PC and SP) and its own private execution stack to manage function local frames.

### 3.3 Why Processes?
The operating system uses the process abstraction to enforce security and reliability through **fault containment**:
* **Isolation:** A bug or crash (like a segmentation fault) in one process cannot corrupt the memory or state of another process.
* **Kernel Protection:** Applications run outside the kernel in restricted environments. This ensures that if an application crashes, the operating system itself remains stable.

## Section 4: Dual-Mode Operation & Transition Mechanics

### 4.1 Dual-Mode Operation
To protect the system from application errors or malicious software, hardware architectures enforce a security boundary called **Dual-Mode Operation**. 


The hardware CPU provides at least two execution modes, tracked by a hardware **Mode Bit** or **Sysmode** register:
1. **User Mode (Mode Bit = 1):** The processor executes instructions with restricted rights. Applications running in user mode cannot access physical hardware devices directly, modify memory translation structures (like page tables), or disable hardware interrupts.
2. **Kernel Mode / Supervisor Mode (Mode Bit = 0):** The CPU has unrestricted access to all hardware resources, physical memory maps, and privileged CPU instructions.

### 4.2 System Architecture Layering (The UNIX Case Study)
The UNIX operating system organizes these boundaries into distinct architectural layers:

| Layer | Privilege Level | Components |
| :--- | :--- | :--- |
| **User Mode** | Restricted | Applications, Shells, Compilers, Interpreters, Standard System Libraries (e.g., `libc`) |
| **Kernel Mode** | Full Privileges | System-Call Interface, File Systems, Virtual Memory, CPU Schedulers, Page Replacement, Character/Block I/O, Device Drivers |
| **Hardware Layer** | Raw Physical Components | Terminal/Device Controllers, Hard Disks, Memory Controllers, Physical RAM, CPUs |

### 4.3 Transition Vectors between User and Kernel Mode
Applications transition from user mode to kernel mode through three controlled entry points:
* **System Calls (Traps):** Synchronous, programmer-initiated requests to the kernel (e.g., writing a file or spawning a thread).
* **Hardware Interrupts:** Asynchronous external events (e.g., a timer interrupt or a network packet arrival) that signal the kernel to handle hardware I/O.
* **Exceptions:** Synchronous events caused by application errors (e.g., divide-by-zero or page faults).

## Section 5: Multi-Process Multiplexing & Base and Bound Protection

### 5.1 The Process Control Block (PCB)
The operating system represents each running process with a kernel data structure called the **Process Control Block (PCB)**.


The PCB serves as the central record for a process's state and contains:
* **Process State:** Current status (e.g., Running, Ready, Blocked).
* **Process Identifier (PID):** A unique integer identifying the process.
* **Register Save State:** Space to store the CPU's registers (including the PC and SP) when the process is context-switched out of execution.
* **Memory Management:** Page table pointers or Base and Bound registers.
* **I/O & File State:** A list of open file descriptors and active network socket configurations.

### 5.2 Process Context Switching Timeline
When the operating system switches execution from Process $P_0$ to Process $P_1$, it performs a **Context Switch**:


1. **Trap/Interrupt:** An interrupt or system call triggers a switch from user mode to kernel mode.
2. **Save State ($P_0$):** The OS saves $P_0$'s execution registers into its corresponding Process Control Block ($PCB_0$).
3. **Execute Kernel Loop:** The scheduler selects the next process ($P_1$) to run.
   
   > **[Code]: Scheduler Loop Pattern**
   > ```c
   > if ( readyProcesses(PCBs) ) {
   >     nextPCB = selectProcess(PCBs);
   >     run( nextPCB );
   > } else {
   >     run_idle_process();
   > }
   > ```
   > *Scheduling is the mechanism for deciding which processes/threads receive hardware CPU time, when, and for how long. Policies can provide fairness, real-time guarantees, or latency optimization. Arguably, this is the main OS loop!*
4. **Restore State ($P_1$):** The OS loads $P_1$'s execution registers from its Process Control Block ($PCB_1$).
5. **Return to User Mode (RTU):** The OS switches the CPU mode bit back to user mode and resumes executing $P_1$.

## Section 6: Memory Virtualization: Base and Bound (B&B)

To run multiple processes safely, the operating system must isolate each process's memory. One of the simplest historical mechanisms to achieve this is the **Base and Bound (B&B)** relocation register model.

### 6.1 Hardware Translation Mechanics
Under the Base and Bound model, every memory address generated by an application (the virtual/program address) is translated on the fly by hardware before it reaches physical RAM:


1. **Hardware Check:** The hardware checks if the virtual address is within the process's allocated limits:
   $$\text{Virtual Address} < \text{Bound}$$
2. **Relocation:** If the address is within bounds, the hardware translates it to a physical address:
   $$\text{Physical Address} = \text{Virtual Address} + \text{Base}$$
3. **Fault Trigger:** If the virtual address is greater than or equal to the Bound register, the hardware triggers a memory fault trap into the kernel, terminating the application with a segmentation fault.

### 6.2 Step-by-Step Base and Bound Execution Lifecycle
Let's trace how the operating system loads, executes, and switches processes using the Base and Bound model:

#### Step 1: OS Loads the Process State
Initially, the CPU runs in kernel mode ($\text{sysmode} = 1$). The Base and Bound registers are ignored when running in kernel mode, giving the OS full access to physical memory.

```
[Hardware State]
sysmode: 1 (Kernel)
PC     : Points to OS Bootloader Code
SP     : Points to Kernel Stack
```

The OS writes the executable's code, static variables, and initial stack space into physical memory (for example, starting at address `0x1000` with a size of `0x0100` bytes).

#### Step 2: Preparing the Hardware Registers
The OS prepares the CPU registers for user-mode execution. The OS sets the Base and Bound registers to define the process's address space:

```
[Hardware Registers Configured by OS]
sysmode: 1 (Kernel)
Base   : 0x1000 (Physical address where process memory begins)
Bound  : 0x0100 (Maximum offset the process can access)
uPC    : 0x0001 (The process's entry point, e.g., main)
SP     : 0x00FF (The top of the process's private virtual stack)
```

#### Step 3: Return to User Mode (RTU)
The kernel executes an **RTU (Return to User Mode)** instruction. This instruction is a privileged operation that performs three actions atomically:
1. Flips the mode bit from 0 (Kernel) to 1 (User).
2. Loads the user Program Counter (`PC`) from the prepared value (`uPC`).
3. Activates Base and Bound address translation.

```
[Hardware State running User Code]
sysmode: 0 (User Mode)
PC     : 0x0001 (Translated to physical address 0x1001)
SP     : 0x00FF (Translated to physical address 0x10FF)
```

The application now runs in its isolated address space. Any attempt to access an address outside `0x0000` - `0x0100` is blocked by the CPU's memory management unit.

#### Step 4: Reclaiming Control via Interrupt
The user application runs until an external event (like a hardware timer interrupt or an I/O request) occurs.

When the timer interrupt fires, the CPU hardware performs the following actions:
1. Flips the mode bit back to 0 (Kernel Mode).
2. Saves the current program counter value into a temporary hardware register (`uPC`).
3. Loads the Program Counter with the address of the kernel's interrupt handler from the interrupt vector table (`PC = IntrpVector[i]`).

```
[Hardware State running Kernel Interrupt Handler]
sysmode: 1 (Kernel Mode)
PC     : Points to Kernel Scheduler Loop
uPC    : Holds suspended process's last PC
```

#### Step 5: Process Context Switch
The kernel scheduler decides to switch execution to another process. It saves the suspended process's state into its Thread Control Block ($TCB_2$, which may reside inside its PCB) and loads the state of the next process ($TCB_1$):

```
1. Save registers (PC, SP, general-purpose registers) of current thread to TCB_2.
2. Load registers of next thread from TCB_1.
3. Update Base and Bound registers to point to the new process's memory region.
```

#### Step 6: Resuming the New Process
The kernel executes an RTU instruction, resetting the mode bit to 0 (User Mode) and jumping to the new process's program counter.

```
[Hardware State running Process B]
sysmode: 0 (User Mode)
Base   : 0x3000 (Process B's Base)
Bound  : 0x0080 (Process B's Bound)
PC     : 0x0002 (Translated to physical address 0x3002)
```

### 6.3 Limitations of Base and Bound Relocation
While simple and fast, the Base and Bound relocation model has several major limitations for general-purpose computing:
* **Inflexible Allocation & Waste:** A process must be allocated contiguous memory for its peak potential stack and heap sizes, which wastes space in between.
* **Physical Memory Fragmentation:** Over time, loading and terminating processes of different sizes creates holes of unallocated memory (fragmentation), making it difficult to fit new processes into contiguous blocks.
* **No Memory Sharing:** Because processes are strictly confined to their own contiguous base and bounds window, they cannot directly share memory pages or read-only library code with other processes.

### 6.4 The Alternative: Translation via Page Tables
To solve these limitations, modern operating systems use **Page Tables** instead of Base and Bound registers.


Instead of mapping a process to a single contiguous block of physical memory, page tables divide virtual and physical memory into fixed-size chunks called **pages** (typically 4 KB). A process's virtual pages can be mapped to non-contiguous physical pages anywhere in memory, eliminating physical fragmentation and allowing processes to easily share memory pages.

## Section 7: Hardware Realities & Threading Models

Modern operating systems are designed around the physical parallel architecture of the underlying hardware.

### 7.1 Case Study: Intel Sapphire Rapids Architecture (2023)
The modern Sapphire Rapids microprocessor demonstrates this shift toward parallel, highly integrated architectures:
* **Compute Capacity:** Scales up to **60 physical cores and 120 simultaneous threads** per physical socket.
* **Chiplet Design:** Integrates up to **4 distinct chiplets** on a single physical package, connected via high-speed on-chip interconnects.
* **Memory Subsystem Layout:** Supports up to **4 TiB of DDR5 memory**.
* **On-Chip Networking:** Uses an on-chip **Mesh Interconnect Network** to handle cache-coherence traffic across cores, scaling up to 480 cores in a single shared-memory domain across 8 sockets.

### 7.2 Simultaneous Multi-Threading (SMT) vs. Hyper-Threading
To maximize hardware utilization, modern processors implement hardware-level scheduling known as **Simultaneous Multi-Threading (SMT)** or **Hyper-Threading**:

```
  Planar Instruction Cycles Timeline:

  a) Classical Single-Core Superscalar:
  [Cycle 1: T0] [Cycle 2: Idle] [Cycle 3: T0] [Cycle 4: Idle] (Stall on memory read)

  b) Hyper-Threading Core:
  [Cycle 1: T0] [Cycle 2: T1  ] [Cycle 3: T0] [Cycle 4: T1  ] (T1 executes during T0 stall)
```

* **How it works:** A single physical CPU core contains multiple execution units (adders, multipliers, etc.) but often stalls waiting for memory accesses. SMT duplicates the core's register state (including the program counter), exposing two "logical threads" to the operating system.
* **Operating System View:** The OS schedules threads onto each logical CPU as if they were independent, physical cores.
* **Performance:** SMT provides a **sub-linear speedup**. While it does not double processing power (since execution units are still shared), it improves performance by keeping execution units busy when one thread stalls.

## Section 8: POSIX Threads (pthreads) Programming API

In modern operating systems, anything that runs outside of the kernel runs inside a process. When a process starts, it initially executes with a single thread. To run tasks concurrently, the application makes system calls to create additional threads that share the process's address space. 

The industry standard API for thread management is **POSIX Threads (pthreads)**.

### 8.1 Core API Functions
```c
int pthread_create(pthread_t *thread, const pthread_attr_t *attr,
                   void *(*start_routine) (void*), void *arg);
```
* Spawns a new thread executing `start_routine` with `arg` as its sole argument.
* Returns `0` on success, or an error code.

```c
void pthread_exit(void *value_ptr);
```
* Terminates the calling thread and makes `value_ptr` available to any thread that joins it.

```c
int pthread_join(pthread_t thread, void **value_ptr);
```
* Suspends execution of the calling thread until the target `thread` terminates.
* If `value_ptr` is non-NULL, it receives the exit status passed to `pthread_exit()`.

### 8.1.1 Peeking Ahead: System Call Example (`pthread_create`)
When `pthread_create(...)` is called, the OS library (like `libc`) issues a system call to the kernel. Here is the step-by-step breakdown of what happens under the hood:

**In the Library (User Mode):**
1. Do some normal function setup work.
2. Put system call arguments into registers (e.g., `%ebx`, etc.).
3. Load the specific syscall number into a designated register (e.g., `%eax`).
4. Execute a special **trap instruction** to switch context to the kernel.

**In the Kernel (Kernel Mode):**
5. Get arguments from the registers.
6. Dispatch to the appropriate system function.
7. Do the actual work to spawn the new thread (allocate TCB, set up stack, etc.).
8. Store the return value in the designated register (`%eax`).
9. Return to user mode.

**Back in the Library (User Mode):**
10. Get the return values from the registers.
11. Do some more normal function work before returning to the caller.

### 8.2 Fork-Join Concurrency Pattern
The Fork-Join pattern is a common multi-threading paradigm where a main thread spawns (forks) a set of child threads to perform tasks in parallel, and then waits (joins) for them to complete before continuing:

```
               [Main Thread Starts]
                        │
                        ▼ (Fork / Create)
             ┌──────────┼──────────┐
             ▼          ▼          ▼
         [Thread 1] [Thread 2] [Thread 3]
             │          │          │
             ▼          ▼          ▼ (Work)
             └──────────┼──────────┘
                        ▼ (Join / Exit)
               [Main Thread Resumes]
```

### 8.3 Complete Code Example: The Fork-Join Pattern (With Data Race)
This example compiles and runs a fork-join application. However, it contains a critical **data race** because multiple threads attempt to modify a shared global variable without synchronization.

```c
#include <stdio.h>
#include <stdlib.h>
#include <pthread.h>

// Shared global variable
int common = 162; //

void *threadfun(void *threadid) {
    long tid = (long)threadid; //
    
    // CRITICAL SECTION WITHOUT LOCKING (Data Race!)
    // Multiple threads read, increment, and write to 'common' concurrently.
    int local_val = common;
    common = local_val + 1; // Equivalent to common++
    
    printf("Thread #%lx stack: %lx common: %d\n", 
           tid, (unsigned long)&tid, local_val); //
           
    pthread_exit(NULL); //
}

int main(int argc, char *argv[]) {
    int nthreads = 4; // Default to 4 threads
    if (argc > 1) {
        nthreads = atoi(argv[1]); //
    }
    
    pthread_t *threads = malloc(nthreads * sizeof(pthread_t)); //
    
    printf("Main stack: %lx, common: %d\n", 
           (unsigned long)&nthreads, common); //
           
    // Fork Phase: Create threads
    for (long t = 0; t < nthreads; t++) {
        int rc = pthread_create(&threads[t], NULL, threadfun, (void *)t); //
        if (rc) {
            printf("ERROR: return code from pthread_create() is %d\n", rc); //
            exit(-1); //
        }
    }
    
    // Join Phase: Wait for all threads to complete
    for (int t = 0; t < nthreads; t++) {
        pthread_join(threads[t], NULL); //
    }
    
    free(threads);
    printf("Final common value: %d\n", common);
    pthread_exit(NULL); //
}
```

### 8.4 Analyzing the Data Race
In this example, the output value of the shared variable `common` is **non-deterministic** because the threads execute concurrently. 

If four threads execute, the final value of `common` might be `163` or `164` instead of the expected `166`. This occurs because multiple threads can read the initial value of `common` (e.g., `162`), increment it in their local registers, and write back the same value (`163`), overwriting each other's updates.

## Section 9: Detailed Execution Stack Trace Walkthrough

To understand how threads manage separate execution paths, we must look at how function calls manipulate the **Execution Stack**.

### 9.1 Stack Activation Record Mechanics
An execution stack holds **activation records** (stack frames) for active function calls. Each frame contains:
1. **Function Parameters:** Input values passed to the function.
2. **Local Variables:** Temporary variables declared inside the function.
3. **Return Address (PC):** The memory address to jump back to when the function returns.

### 9.2 Tracing a Recursive Stack Lifecycle
Let's trace the stack frames for the following recursive C code:

```c
void A(int tmp) {
    if (tmp < 2) 
        B();
    printf("%d ", tmp);
}

void B() {
    C();
}

void C() {
    A(2);
}

int main() {
    A(1);
}
```

#### Step 1: Execution Begins (`main`)
The program starts by calling `A(1)` from `main` (return address: `exit`).

```
[Stack Layout] (Stack Pointer SP points to Top)
┌─────────────────────────┐
│ A: tmp = 1              │ <--- SP
│    Return PC: exit      │
└─────────────────────────┘
```

#### Step 2: `A(1)` Calls `B()`
Since `tmp = 1 < 2`, `A` calls `B()` (return address: instruction $A+2$, which is `printf`).

```
[Stack Layout]
┌─────────────────────────┐
│ A: tmp = 1              │
│    Return PC: exit      │
├─────────────────────────┤
│ B:                      │ <--- SP
│    Return PC: A+2       │
└─────────────────────────┘
```

#### Step 3: `B()` Calls `C()`
Function `B` immediately calls `C()` (return address: instruction $B+1$, the return point of `B`).

```
[Stack Layout]
┌─────────────────────────┐
│ A: tmp = 1              │
│    Return PC: exit      │
├─────────────────────────┤
│ B:                      │
│    Return PC: A+2       │
├─────────────────────────┤
│ C:                      │ <--- SP
│    Return PC: B+1       │
└─────────────────────────┘
```

#### Step 4: `C()` Calls `A(2)`
Function `C` calls `A` recursively, passing `2` as the argument (return address: instruction $C+1$, the return point of `C`).

```
[Stack Layout]
┌─────────────────────────┐
│ A: tmp = 1              │
│    Return PC: exit      │
├─────────────────────────┤
│ B:                      │
│    Return PC: A+2       │
├─────────────────────────┤
│ C:                      │
│    Return PC: B+1       │
├─────────────────────────┤
│ A: tmp = 2              │ <--- SP
│    Return PC: C+1       │
└─────────────────────────┘
```

#### Step 5: Evaluating `A(2)`
For this call frame, `tmp = 2`, which is not less than 2. The `if` condition is skipped, and the program executes `printf(2)`, outputting `2`.

#### Step 6: `A(2)` Returns
The function returns to `C+1`. Its stack frame is popped from the stack.

```
[Stack Layout]
┌─────────────────────────┐
│ A: tmp = 1              │
│    Return PC: exit      │
├─────────────────────────┤
│ B:                      │
│    Return PC: A+2       │
├─────────────────────────┤
│ C:                      │ <--- SP
│    Return PC: B+1       │
└─────────────────────────┘
```

#### Step 7: Unrolling the Stack
`C()` completes and returns to `B+1`. `B()` completes and returns to $A+2$ (the print statement in the first call of `A`).

```
[Stack Layout]
┌─────────────────────────┐
│ A: tmp = 1              │ <--- SP
│    Return PC: exit      │
└─────────────────────────┘
```

#### Step 8: Execution Completes
The program executes the print statement for `tmp = 1` in the first call of `A`, outputting `1`. The total printed output is `2 1`. `A(1)` returns to `exit`, and the program terminates.

### 9.3 Memory Layout with Multiple Stacks
When a process runs multiple threads, each thread must have its own private execution stack to manage function calls independently:


This layout introduces several architectural challenges:
* **Stack Positioning:** The operating system must arrange multiple stacks within the process's single virtual address space.
* **Choosing Maximum Size:** The OS must allocate a maximum size for each stack. If a stack is allocated too much space, virtual memory is wasted. If it is allocated too little, the stack can easily overflow.
* **Stack Overflow Risks:** If a thread's stack grows too large, it can silently overwrite the heap or another thread's stack, corrupting the process's data.

### 9.4 Catching Stack Violations (Guard Pages)
To prevent threads from overwriting neighboring memory, modern operating systems use **Guard Pages**:

```
┌──────────────────────────────────────┐
│             Thread 1 Stack           │
├──────────────────────────────────────┤
│  Guard Page (No Read/Write Access)   │ <--- Triggers SegFault if reached
├──────────────────────────────────────┤
│             Thread 2 Stack           │
└──────────────────────────────────────┘
```

* **How it works:** The operating system inserts an unmapped virtual memory page (a guard page with no read or write permissions) directly between each thread's stack space.
* **Violation Detection:** If a thread's stack grows beyond its limit and attempts to access the guard page, the CPU's memory management unit intercepts the access and triggers a page fault, terminating the process immediately with a segmentation fault instead of allowing silent data corruption.

## Section 10: Interleaving, Non-Determinism, & Race Conditions

Writing concurrent programs is challenging because the operating system's scheduler can run threads in any order and switch between them at any time.

### 10.1 Concurrency vs. Parallelism & Multiprogramming
* **Multiprocessing:** Multiple CPUs (cores) executing multiple threads/processes.
* **Multiprogramming:** Multiple jobs/processes running concurrently.
* **Multithreading:** Multiple threads running concurrently within processes.

* **Concurrency:** Managing multiple tasks at once (handling multiple things at once). On a single-core system, threads execute concurrently by interleaving their instructions over time. The scheduler is free to run threads in any order, allowing threads to run to completion or time-slice in big or small chunks.
* **Parallelism:** Executing multiple tasks simultaneously on separate physical processors.

```
  Single-Core System (Concurrency):
  [ Thread 1 ] [ Thread 2 ] [ Thread 1 ] [ Thread 2 ] ──> Time

  Multi-Core System (Parallelism):
  Core 1: [ Thread 1 ] [ Thread 1 ] [ Thread 1 ]
  Core 2: [ Thread 2 ] [ Thread 2 ] [ Thread 2 ]      ──> Time
```

### 10.2 Tracing a Race Condition
A **Race Condition** occurs when multiple threads read and write shared data concurrently, and the final outcome depends on the order in which their instructions interleave.

Let's analyze this example where two threads modify shared variables:

```
Initial State: x = 0, y = 0 

Thread A                      Thread B
x = y + 1;                    y = 2; 
                              y = y * 2; 
```

Depending on how the scheduler schedules these instructions, $x$ can result in one of three possible values:

#### Interleaving 1 (Thread A Runs First)
```
1. Thread A: x = y + 1  ==> x = 0 + 1 = 1
2. Thread B: y = 2
3. Thread B: y = y * 2  ==> y = 4

Result: x = 1
```

#### Interleaving 2 (Thread B Runs First)
```
1. Thread B: y = 2
2. Thread B: y = y * 2  ==> y = 4
3. Thread A: x = y + 1  ==> x = 4 + 1 = 5

Result: x = 5
```

#### Interleaving 3 (Thread B Suspended Mid-Execution)
```
1. Thread B: y = 2
2. Thread A: x = y + 1  ==> x = 2 + 1 = 3
3. Thread B: y = y * 2  ==> y = 3 * 2 = 6

Result: x = 3
```

This variation demonstrates that concurrent code can produce different results on different runs, making bugs difficult to reproduce and debug.

## Section 11: Synchronization and Locks

To prevent race conditions, cooperating threads must synchronize their access to shared resources.

### 11.1 Key Terminology
* **Synchronization:** Coordinating the execution of multiple threads to ensure correct behavior when accessing shared data.
* **Mutual Exclusion:** Ensuring that only one thread can access a shared resource or execute a critical section of code at any given time.
* **Critical Section:** A block of code that accesses shared resources and must not be executed by more than one thread concurrently.
* **Lock:** An object used to enforce mutual exclusion.

### 11.2 The Lock Abstraction
A lock provides two primary atomic operations:
1. **Acquire:** A thread attempts to acquire the lock. If the lock is free, the thread marks it as busy and continues execution. If the lock is held by another thread, the calling thread blocks until the lock is released.
2. **Release:** The thread holding the lock releases it, marking it as free and allowing blocked threads to acquire it.

### 11.3 Resolving Race Conditions with pthreads Mutexes
The pthreads library provides mutual exclusion objects called **mutexes** to synchronize thread access to shared data.

```c
#include <stdio.h>
#include <stdlib.h>
#include <pthread.h>

int common = 162; // Shared resource

// Initialize the mutex lock
pthread_mutex_t common_lock = PTHREAD_MUTEX_INITIALIZER; //

void *threadfun(void *threadid) {
    long tid = (long)threadid; //
    
    // Acquire the lock to enter the critical section safely
    pthread_mutex_lock(&common_lock); //
    
    // Critical Section: Only one thread can execute this block at a time
    int my_common = common++; //
    
    // Release the lock to allow other threads to enter
    pthread_mutex_unlock(&common_lock); //
    
    printf("Thread #%lx stack: %lx common: (%d)\n", 
           tid, (unsigned long)&tid, my_common); //
           
    pthread_exit(NULL); //
}

int main(int argc, char *argv[]) {
    int nthreads = 4;
    if (argc > 1) {
        nthreads = atoi(argv[1]);
    }
    
    pthread_t *threads = malloc(nthreads * sizeof(pthread_t));
    
    for (long t = 0; t < nthreads; t++) {
        pthread_create(&threads[t], NULL, threadfun, (void *)t);
    }
    
    for (int t = 0; t < nthreads; t++) {
        pthread_join(threads[t], NULL);
    }
    
    free(threads);
    printf("Safe final common value: %d\n", common);
    return 0;
}
```

By wrapping the modification of `common` inside a `pthread_mutex_lock` and `pthread_mutex_unlock` block, we guarantee that only one thread can increment the variable at a time, ensuring a deterministic and correct final outcome.

### Question 1: Base and Bound Relocation Address Mapping
A CPU runs a process under a Base and Bound relocation register system. The hardware registers are currently set to:
* **Base:** `0x4000`
* **Bound:** `0x1200`

For each of the following virtual memory addresses issued by the application, determine whether the access is valid. If it is valid, calculate the corresponding physical memory address.
1. Virtual Address `0x0050`
2. Virtual Address `0x1200`
3. Virtual Address `0x1500`

#### Solution:
The hardware enforces the check: $\text{Virtual Address} < \text{Bound}$.
If valid, the physical address is: $\text{Physical Address} = \text{Virtual Address} + \text{Base}$.

1. **Virtual Address `0x0050`:**
   * Check: `0x0050` < `0x1200` (Valid).
   * Translation: `0x0050` + `0x4000` = `0x4050`.
   * **Answer:** Valid; maps to physical address `0x4050`.
2. **Virtual Address `0x1200`:**
   * Check: `0x1200` < `0x1200` (Invalid; the virtual address must be strictly less than the Bound).
   * **Answer:** Invalid; triggers a memory protection fault (Segmentation Fault).
3. **Virtual Address `0x1500`:**
   * Check: `0x1500` < `0x1200` (Invalid; address is out of bounds).
   * **Answer:** Invalid; triggers a memory protection fault (Segmentation Fault).

### Question 2: Stack Execution Frames Trace
Given the following C program:

```c
void X(int val) {
    if (val > 1) {
        X(val - 1);
    }
    printf("%d", val);
}

int main() {
    X(3);
}
```

1. Sketch the layout of the execution stack when it reaches its maximum depth during execution.
2. What is the printed output of this program?

#### Solution:
1. **Stack trace at maximum depth:**
   * `main()` calls `X(3)` (Return PC: exit).
   * `X(3)` calls `X(2)` recursively (Return PC: print statement in `X`).
   * `X(2)` calls `X(1)` recursively (Return PC: print statement in `X`).
   * For `X(1)`, the condition `val > 1` is false, so it does not make any further recursive calls. This is the maximum depth of the stack:

```
┌─────────────────────────┐
│ X: val = 3              │
│    Return PC: exit      │
├─────────────────────────┤
│ X: val = 2              │
│    Return PC: X print   │
├─────────────────────────┤
│ X: val = 1              │ <--- SP (Max Stack Depth)
│    Return PC: X print   │
└─────────────────────────┘
```

2. **Program Output:**
   * At maximum depth, `X(1)` executes `printf` and outputs `1`. It then returns to `X(2)`.
   * `X(2)` resumes execution at its print statement, outputs `2`, and returns to `X(3)`.
   * `X(3)` resumes execution at its print statement, outputs `3`, and returns to `main`.
   * **Answer:** The program outputs `123`.
